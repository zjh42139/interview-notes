# Writing Rules

> 本知识库所有文档遵循以下规范，保证内容质量和一致性。

## 一、评分体系

每篇文章完成后自评，5 个维度各 20 分，满分 100。**85 分以上方可视为完成。**

| 维度（各 20 分）| 满分标准 | 扣分信号 |
|------|----------|----------|
| **结构完整度** | 该 type 的必写章节全部写完 | 缺章节、缺代码示例、缺 Mermaid 图 |
| **面试导向力** | 面试信号表覆盖 ≥3 个追问方向，每章都能直接引原话回答 | 教科书式罗列概念、没有面试信号表、纯定义无话术 |
| **深度分层** | 初级概念一句话带过，重点在中高级机制；有"为什么这样设计"的思考 | 初级内容（API 罗列）超过 50%、没有源码级解释 |
| **项目落地感** | 有真实场景的代码（不是伪代码），有踩坑经历和修复方案 | 例子太通用、没有易错点、和实际项目无关联 |
| **知识网络度** | 相关阅读 ≥3 个有效链接，被 ≥1 篇已有文章反向引用，出现在模块 mindmap 中 | 孤立文章、只有单向引用、mindmap 遗漏 |

**使用方式**：写完初稿 → 自评打分 → 对照扣分项逐条修 → 修完再评，≥85 分 → `status: reviewed`。

## 二、文章类型

每篇文章在 frontmatter 中声明 `type`，选一种。不同 type 有不同的必写/可选章节。

### mechanism（机制原理型）
解析某个技术的内部工作原理。代表：V8 引擎、原型链、Event Loop、渲染流程、Diff/Patch。

```
必写：一句话总结 + 核心机制 + 深度拓展 + 面试信号表
可选：项目实战、易错点（如果篇幅有限可精简）
```

### comparison（对比选型型）
两个或多个方案的本质差异和选择标准。代表：Canvas vs SVG、hash vs history、CSR vs SSR。

```
必写：一句话总结 + 对比表 + 决策逻辑 + 深度拓展
可选：核心机制（概念本身简单的话，合并入对比表）
```

### api-reference（API/操作型）
某个 API/工具/命令的用法和易错点。代表：HTML5 表单、BOM 全景、Observer API、Git stash。

```
必写：一句话总结 + 核心机制（含 API 速查） + 易错点 + 面试信号表
可选：深度拓展（API 本身不深的话可精简）、项目实战
```

### security（攻防安全型）
攻击原理和防御方案。代表：XSS/CSRF、浏览器安全机制。

```
必写：攻击原理（建议含 Mermaid 时序图） + 防御方案 + 易错点 + 面试信号表
可选：项目实战
```

### project（项目实战型）
端到端的实现指南，含方案选型和完整代码。代表：大文件上传、ECharts 实战、国际化。

```
必写：需求场景 + 方案选型 + 核心代码 + 边界处理
可选：深度原理（不属于本文件的原理链到对应文章）
```

### algorithm（算法/数据结构型）
算法思想和模板代码。代表：排序、动态规划、DFS/BFS。

```
必写：算法思想 + 模板代码 + 复杂度分析 + 变种题
可选：源码级优化（前端岗位不需要）
```

### exercise（手写题/代码实现型）
从零实现某个功能的完整过程。代表：手写 Promise、深拷贝、bind/call/apply。

```
必写：题目描述 + 实现思路 + 基础版代码 + 边界处理/升级版 + 易错点
可选：原理深挖（链到 mechanism 文章）
```

### interview（面试策略型）
非技术知识的面试应答框架。代表：自我介绍、HR 面、模拟面试、面试题库。

```
必写：场景/题目 + 回答框架/话术 + 追问预判 + 常见误区
可选：核心机制（面试应答不需要教技术原理）
```

### 新增类型规则

写文章时找不到匹配的 type → 回头看已有文章，至少找到 **2 篇**同类可归入 → 新建 type。过渡期（存量文章尚未全部打 type 前）不强制 2 篇门槛。最终每种 type 至少覆盖 5+ 篇文章，防止类型爆炸。

## 三、模板（按 type 选，不强制统一 9 段）

| 级别 | 板块 | 说明 |
|------|------|------|
| ✅ 必写 | Frontmatter、一句话总结 | 所有 type 都必须有 |
| ✅ 必写 | 对应 type 的必写章节 | 见上文每种 type 定义 |
| ⭐ 推荐 | 对应 type 的可选章节 | 该写但篇幅不够可精简 |
| ⭕ 按需 | Mermaid / 图片 / 额外代码示例 | 有则留，无则删 |

**硬规则**：该 type 的必写章节一个都不能少。可选章节空着比硬凑好——不要为了模板完整性填入伪代码或凑数段落。

## 四、Frontmatter 规范

```yaml
---
title: string           # 必填
description: string     # 必填
category: string        # 必填（所属模块：JavaScript / Vue3 / 浏览器 / ...）
type: string            # 必填（mechanism | comparison | api-reference | security | project | algorithm | exercise | interview）
difficulty: string      # 必填（初级 | 中级 | 高级）
frequency: string       # 必填（⭐ 1-5）
status: string          # 必填（draft | reviewed | published）
score: number           # 必填（0-100，自评分，完成后更新）
created: string         # 必填（YYYY-MM-DD）
updated: string         # 必填（修改文章时必须同步更新此日期）
reviewed: string|null   # 可选
tags:                   # 推荐
  - tag1
  - tag2
---
```

**status 三态**：`draft`（初稿/未达标）→ `reviewed`（自评 85+）→ `published`（设此状态表示可对外发布）。

**score 字段**：新文章 score 初始为 0，写完后自评填入。修改文章后重新评分。

## 五、写作原则

以下 6 条继承自旧版规范，保持不变：

**1. 不写教材式内容**

❌ Promise 是 ECMAScript 2015 引入的一种异步编程解决方案……
✅ 面试中可以这样回答：Promise 本质上是一个状态机，解决了回调嵌套问题……

**2. 每篇只讲一个核心知识点**

一篇文档只回答一个问题。发现写第二个知识点，拆成新文件。

**3. 优先解释"为什么"，而不是"是什么"**

面试官不考定义。考的是：为什么需要这个东西？它解决了什么问题？

**4. 项目实战按需写**

不再强制必须绑定 Vue3 后台管理系统。什么项目合适用什么——可以是内部管理系统、C 端活动页、监控平台、组件库，也可以是真实踩坑经历。原则：**真实场景 > 虚构场景，具体代码 > 伪代码。**

**5. 面试回答优先于概念定义**

开篇第一句话就是"面试中怎么回答"，而不是"什么是 XXX"。

**6. 不重复，通过链接引用（DRY）**

同一个知识点只在一处深入写。其他地方通过「相关阅读」链接：
- Promise 原理 → 只在 `JavaScript/promise.md` 中写
- 手写 Promise → 链到 `手写题/promise.md`
- Promise 真题 → 链到 `面试题库/JavaScript.md`

**7. 能画图就少写文字**

优先用 Mermaid 图表，减少纯文字描述。一张好的流程图胜过 500 字。

## 六、交叉引用规范

新增文件时，必须完成双向链接：

1. **正向引用**（新文件 → 旧文件）：新文件「相关阅读」至少 3 个链接
2. **反向引用**（旧文件 → 新文件）：扫同模块文章 + grep 关键词匹配的跨模块文章，判断是否需要加回链

**反向引用是知识库最重要的维护动作**——它让孤立文章变成知识网络的节点。新增子模块流程详见内部规范。
