# CI/CD 概述

> ⭐⭐⭐⭐｜难度：初级｜项目：★★

**CI/CD 不是你"要不要做"的选择题，而是现代团队协作的默认配置。面试官不关心你用过哪个 CI 工具，关心的是你能否说清楚：代码从 `git push` 到用户看到新页面，中间经过了哪些自动化环节。**

## 一句话总结

**CI（持续集成）让团队频繁合并代码并自动验证（构建 + 测试），CD（持续交付/持续部署）让验证通过的代码自动或一键发布到目标环境。核心价值是：把人从重复的手动操作中解放出来，用自动化保证每次交付的质量和一致性。**

---

## 核心机制

### CI：持续集成（Continuous Integration）

CI 解决的是**"在我机器上能跑啊"**问题。

```
开发者 push 代码 → CI 触发 → 安装依赖 → 代码检查(Lint) → 构建 → 单元测试 → 报告结果
```

- **高频合并**：开发者每天多次将代码合并到主分支，避免"合并地狱"
- **自动验证**：每次合并都自动触发构建 + 测试，问题在几分钟内暴露
- **反馈闭环**：构建失败立刻通知提交者，而不是等到提测/上线才发现

### CD：持续交付 vs 持续部署

这两个概念常被混淆，但它们有本质区别：

| 维度 | 持续交付（Continuous Delivery） | 持续部署（Continuous Deployment） |
|------|-------------------------------|----------------------------------|
| **最后一步** | 手动触发部署（需要人点按钮） | 自动部署到生产环境（无人干预） |
| **生产发布** | 需要人工审批 | 全自动 |
| **风险控制** | 人在发布前最后把关 | 依赖自动化测试和灰度策略兜底 |
| **适用场景** | 金融、医疗等强监管行业 | 互联网 SaaS、内部工具 |
| **常见工具** | GitHub Actions + 审批流程 | ArgoCD、Vercel 自动部署 |

**实际面试中**：面试官说"CI/CD"时，默认指 CI + 持续交付；如果特意提到"你们能做到持续部署吗"，才需要区分这两个概念。

### 与传统发布流程的对比

```
传统流程（纯手动）：
写代码 → 手动测试 → 手动 npm run build → 手动压缩 dist → FTP 上传到服务器 → 重启 Nginx
     ↑                                        ↑
  耗时 10 分钟                              耗时 15 分钟（还容易出错）

CI/CD 流程（自动化）：
git push → GitHub Actions 自动触发 → lint → test → build → 自动部署 → 通知结果
     ↑                                                                      ↑
  耗时 5 秒（你只需要 push）                                         耗时 2-3 分钟（全自动）
```

**核心差异**：传统流程是人驱动流程，CI/CD 是事件驱动流程。

## 关键概念

| 概念 | 说明 | 类比 |
|------|------|------|
| **Pipeline（流水线）** | 整个 CI/CD 流程的总称 | 工厂的流水线 |
| **Stage（阶段）** | Pipeline 中的一个阶段（如 Build、Test） | 流水线上的一个工位 |
| **Job（作业）** | Stage 中的一个执行单元，在 Runner 上运行 | 一个工人完成的具体任务 |
| **Step（步骤）** | Job 中的最小执行单位（如 `npm install`） | 工人的每一个操作动作 |
| **Runner（运行器）** | 执行 Job 的机器（GitHub 托管或自建） | 工人的工作台 |
| **Artifact（产物）** | Job 产生的文件（如 `dist/` 目录、测试报告） | 工位产出的半成品 |
| **Trigger（触发器）** | 什么事件启动 Pipeline（push / PR / 定时） | 开工信号 |
| **Environment（环境）** | 部署目标（dev / staging / production） | 产品交付的车间 |

### 流水线数据流

```
Trigger (git push)
    │
    ▼
Stage: Install ──────► 产物: node_modules/
    │
    ▼
Stage: Lint ─────────► 产物: lint 报告
    │
    ▼
Stage: Build ────────► 产物: dist/
    │
    ▼
Stage: Test ─────────► 产物: 测试报告
    │
    ▼
Stage: Deploy ───────► 把 dist/ 推送到服务器
```

## 面试场景：画出 Vue 项目的 CI/CD 流水线

面试时如果被问到"请画一下你们项目的 CI/CD 流水线"，按以下结构回答即可：

```
┌─────────────────────────────────────────────────────────┐
│                     CI Pipeline                          │
│                                                         │
│  git push ──► [Install] ──► [Lint] ──► [Unit Test]     │
│                     │           │            │          │
│                     ▼           ▼            ▼          │
│               node_modules   ESLint     Vitest 报告     │
│                                                         │
│  ──► [Build] ──► [E2E Test] ──► [Upload Artifact]      │
│          │            │               │                 │
│          ▼            ▼               ▼                 │
│       dist/      Cypress 报告   存储 dist/ 供后续使用    │
│                                                         │
├─────────────────────────────────────────────────────────┤
│                     CD Pipeline                          │
│                                                         │
│  ┌─ 合并到 main 分支 ─────────────────────────────────┐ │
│  │                                                     │ │
│  │  [Download Artifact] ──► [Deploy to Staging]        │ │
│  │         │                        │                  │ │
│  │         ▼                        ▼                  │ │
│  │      dist/               staging.example.com        │ │
│  │                                                     │ │
│  │  手动审批 ──► [Deploy to Production]                │ │
│  │                        │                            │ │
│  │                        ▼                            │ │
│  │                 example.com                          │ │
│  └─────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

**口述要点**：

1. "代码 push 到任意分支触发 CI，做 lint + 单测 + 构建"
2. "构建产物 `dist/` 上传为 artifact，PR 被合并到 main 后触发 CD"
3. "CD 先部署到 staging 环境，自动化 E2E 测试通过后，手动审批上生产"
