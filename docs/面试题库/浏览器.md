## 页面渲染与加载（5 题）

### Q1: 浏览器渲染流程（从 URL 到页面）

> 🏷️ 概念题
> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：请详细描述从输入 URL 到页面展示的完整过程（包含网络和渲染）。

**考察点**：
- DNS 解析（递归查询 + 迭代查询）
- TCP 三次握手 + TLS 握手（HTTPS）
- HTTP 请求/响应
- HTML 解析 → DOM 树构建
- CSS 解析 → CSSOM 树构建
- 布局（Layout）→ 绘制（Paint）→ 合成（Composite）
- JavaScript 阻塞解析（`defer` / `async` 的区别）

**30秒答**：DNS 解析→TCP 连接(HTTPS加TLS)→发 HTTP 请求→服务器返回 HTML→解析成 DOM 树→CSS 解析成 CSSOM→合并渲染树→布局计算位置→绘制像素→合成显示。script 阻塞 DOM 解析，CSS 阻塞渲染。
**追问预测**：
- "DOM 解析和 CSS 解析是什么关系" → 两者并行——但渲染树必须等 CSSOM 构建完成
- "script 标签会阻塞什么" → 普通 script 阻塞 DOM 解析；defer 等 DOM 完成后执行；async 下载完就执行——独立题详见 [HTML 题库 Q6](./HTML.md)、[../HTML/script-defer-async.md](../HTML/script-defer-async.md)
- "重排和重绘的区别" → 重排改布局（几何变化/代价大），重绘改外观（颜色/背景/不改变布局）

### Q2: 回流（Reflow）与重绘（Repaint）

> 🏷️ 概念题
> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：什么是回流和重绘？分别有哪些触发条件？如何减少回流和重绘？

**考察点**：
- 回流：布局改变，重新计算元素几何信息（尺寸、位置）
- 重绘：样式改变但布局不变（颜色、背景等）
- 回流一定触发重绘，重绘不一定触发回流
- 触发回流的操作：增删 DOM、修改尺寸/位置、`offsetTop` 等强制同步布局
- 优化：批量修改 DOM（`documentFragment`）、`class` 代替 `style`、`transform` 代替 `top/left`、`will-change` 创建合成层

**30秒答**：回流=布局改变重新计算几何位置（改宽高/display/读 offsetHeight）。重绘=外观改变不重布局（颜色/背景）。回流一定重绘，重绘不一定回流。优化：批量 DOM 操作、用 transform 代替 left/top、离线修改。
**追问预测**：
- "什么操作触发回流" → 修改宽高/位置/display、读写 offsetHeight、改变字体大小、resize 窗口
- "怎么减少回流" → 批量修改 DOM（documentFragment）、用 class 替代 style、离线 DOM（display:none 后修改）
- "transform 为什么不会回流" → transform 在合成层上操作——不触发布局重算，只触发合成

> 答案参考：[../浏览器/reflow-repaint.md](../浏览器/reflow-repaint.md)
> 🎤 回答稿：[30秒+2分钟版本](../面试回答/浏览器/reflow-repaint.md)

### Q3: Event Loop（事件循环）

> 🏷️ 概念题
> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：什么是事件循环？宏任务和微任务有什么区别？

**考察点**：
- 宏任务：`setTimeout`/`setInterval`/`I/O`/UI rendering/`MessageChannel`
- 微任务：`Promise.then`/`MutationObserver`/`queueMicrotask`
- 执行顺序：宏任务 → 所有微任务清空 → 渲染 → 下一个宏任务
- 经典题：`setTimeout` + `Promise` 混合输出顺序
- Node.js 的事件循环（6 个阶段，`process.nextTick`）

**30秒答**：浏览器 Event Loop——宏任务→清微→渲染→宏任务。微任务(Promise.then/MutationObserver)优先级高于宏任务(setTimeout/事件)。rAF 在渲染前执行，不在宏/微队列中。
**追问预测**：
- "微任务和宏任务的优先级" → 微任务在当前宏任务后、下一个宏任务前清空——微任务优先级高于宏任务
- "requestAnimationFrame 在哪个阶段" → 渲染之前——既不是宏也不是微任务
- "为什么 setTimeout(fn,0) 不精准" → 0 是最小延迟，嵌套 5 层强制 4ms，且到期后还要排队等待

> 答案参考：[../JavaScript/event-loop.md](../JavaScript/event-loop.md)

### Q4: 页面生命周期事件

> 🏷️ 概念题
> ⭐⭐⭐⭐ | 难度：中级

**题目**：DOMContentLoaded 和 load 有什么区别？beforeunload 和 unload 呢？

**考察点**：
- `DOMContentLoaded`：DOM 树构建完成 + defer 脚本执行完（图片/CSS 可能还在加载）
- `load`：所有资源（图片/字体/iframe）加载完毕
- `beforeunload`：用户即将离开，可弹确认框（不能做异步操作）
- `unload`：页面即将被销毁（已不推荐，会阻止 bfcache，用 `pagehide` 替代）
- `visibilitychange`：Tab 切后台/前台，用于暂停/恢复资源消耗
- bfcache（回退缓存）：`pageshow.persisted` 检测是否从缓存恢复

**30秒答**：DOMContentLoaded HTML 解析完 DOM 构建完成。load 所有资源加载完。beforeunload 可阻止离开。visibilitychange 页面可见性变化——切后台暂停动画/轮询。pageshow/pagehide 包含 bfcache 缓存恢复场景。
**追问预测**：
- "DOMContentLoaded 和 load 的区别" → DCL HTML 解析完 DOM 树构建完成；load 所有资源（图片/CSS/JS）加载完
- "beforeunload 和 unload 的区别" → beforeunload 可阻止页面关闭（弹确认框）；unload 页面正在卸载不能阻止
- "visibilitychange 有什么用" → 页面可见性变化——切后台暂停动画/视频/轮询，切回前台恢复

> 答案参考：[../浏览器/page-lifecycle.md](../浏览器/page-lifecycle.md)

### Q5: requestAnimationFrame

> 🏷️ 概念题
> ⭐⭐⭐ | 难度：中级

**题目**：requestAnimationFrame 和 setTimeout 做动画有什么区别？

**考察点**：
- rAF 在每帧渲染前执行（与屏幕刷新率同步 60fps）
- setTimeout 最小延迟 4ms（嵌套调用），可能在一帧之内或之外执行
- rAF 页面不可见时自动暂停（省电），setTimeout 继续执行
- requestIdleCallback 在帧末尾空闲时执行（低优先级任务）
- 具体时间线：每个帧内 rAF → Layout → Paint → requestIdleCallback

**30秒答**：rAF 跟随屏幕刷新率自动匹配 60fps(16.6ms)——页面不可见时自动暂停省 CPU。setTimeout(cb,16) 不受帧率同步可能丢帧。rAF 适合 JS 动画、滚动处理——天然节流。
**追问预测**：
- "rAF 和 setTimeout(cb,16) 的区别" → rAF 跟随屏幕刷新率自动匹配、页面不可见时自动暂停
- "rAF 适合什么场景" → JS 动画、滚动处理、批量 DOM 更新——天然节流 60fps
- "rAF 和 requestIdleCallback 的关系" → rAF 在帧开始前，rIC 在帧末尾空闲时——低优先级任务用 rIC

> 答案参考：[../浏览器/request-animation-frame.md](../浏览器/request-animation-frame.md)

### Q6: 同源策略

> 🏷️ 概念题
> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：什么是同源策略？跨域有哪些解决方案？

**考察点**：
- 同源 = 协议 + 域名 + 端口 三者完全相同
- 三处限制：DOM 访问、Cookie/Storage、AJAX 响应
- 跨域方案：CORS（简单请求 vs 预检请求）、JSONP（原理和局限）、postMessage、代理
- CORS 的 `Access-Control-Allow-Origin` 和 `credentials` 的配合
- 哪些标签不受同源策略限制（`<script>` `<img>` `<link>`）

**30秒答**：协议+域名+端口三者相同=同源。限制 DOM 访问、Cookie 读取、AJAX 请求。script/img/link 加载资源不受限制。跨域方案：CORS 标准、JSONP 古老只 GET、代理最实用、postMessage iframe 通信。
**追问预测**：
- "同源策略限制了什么" → DOM 访问、Cookie 读取、AJAX 请求——但不限制 script/img/link 加载
- "CORS 简单请求和预检请求区别" → 简单请求直接发；预检先发 OPTIONS 询问服务器是否允许
- "JSONP 为什么能跨域" → 利用 script 标签不受同源限制——返回 JS 代码而非 JSON

### Q7: XSS 和 CSRF

> 🏷️ 对比题
> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：什么是 XSS 和 CSRF？分别怎么防御？

**考察点**：
- XSS 三种类型（反射型/存储型/DOM 型）+ 攻击原理
- CSRF 攻击原理（利用浏览器自动携带 Cookie 的特性）
- XSS 防御：HTML 实体转义 + CSP + HttpOnly Cookie + DOMPurify
- CSRF 防御：SameSite Cookie + CSRF Token + Referer/Origin 校验
- XSS vs CSRF 的本质区别（XSS 偷数据，CSRF 伪造操作）
- XSS 能绕过 CSRF Token

**30秒答**：XSS 代码注入——反射型(请求中)/存储型(数据库中)/DOM 型(客户端)。防御：输出转义+HttpOnly Cookie+CSP。CSRF 身份盗用——利用已登录状态伪造请求。防御：SameSite Cookie+CSRF Token+Origin 校验。本质区别：XSS 攻客户端，CSRF 盗身份。
**追问预测**：
- "XSS 和 CSRF 的本质区别" → XSS 代码注入（在受害者页面执行脚本），CSRF 身份盗用（利用已登录状态伪造请求）
- "XSS 三种类型的区别" → 反射型在请求中、存储型在数据库中、DOM 型全在客户端不经过服务端
- "CSRF Token 为什么不能放 Cookie" → Cookie 会被浏览器自动带上——攻击者也带着 token

### Q8: 浏览器缓存策略

> 🏷️ 概念题
> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：强缓存和协商缓存有什么区别？Cache-Control 有哪些值？

**考察点**：
- 强缓存：`Cache-Control: max-age=3600` / `Expires`，状态码 200 (disk cache)
- 协商缓存：`ETag/If-None-Match` + `Last-Modified/If-Modified-Since`，状态码 304
- `Cache-Control` 指令：`max-age`/`no-cache`/`no-store`/`public`/`private`/`must-revalidate`
- 实际场景：`index.html` 用 `no-cache`（每次校验），静态资源用 `max-age=31536000`（文件名 hash）
- `ETag` 比 `Last-Modified` 更精确（文件内容 vs 修改时间）

**30秒答**：强缓存不发请求——Cache-Control:max-age 控制，命中 200 from cache。协商缓存发请求问服务器——ETag/Last-Modified 没变回 304，变了回 200。HTML 用 no-cache 每次验证，JS/CSS(hash)用 max-age 一年。
**追问预测**：
- "no-cache 和 no-store 的区别" → no-cache 缓存但每次验证；no-store 完全不缓存——真正的不存
- "ETag 和 Last-Modified 哪个优先" → ETag——内容级精确；Last-Modified 秒级精确，同秒内两次修改检测不到
- "为什么 HTML 要设 no-cache" → 发版后 JS/CSS hash 变了但旧 HTML 还在缓存→引用旧文件→白屏

> 答案参考：[../浏览器/cache.md](../浏览器/cache.md)
> 🎤 回答稿：[../面试回答/浏览器/cache.md](../面试回答/浏览器/cache.md)

### Q9: Cookie 的 SameSite 属性

> 🏷️ 概念题
> ⭐⭐⭐⭐ | 难度：中级

**题目**：SameSite 有哪些值？分别有什么安全影响？

**考察点**：
- `Strict`：任何跨站请求都不带 Cookie（最安全但体验差）
- `Lax`：浏览器默认值，跨站时只对"顶级导航 GET 请求"携带 Cookie
- `None`：允许跨站携带，必须配合 `Secure`（仅 HTTPS）
- HttpOnly（防 XSS）+ Secure（防 MITM）+ SameSite（防 CSRF）三层防御
- Cookie vs LocalStorage vs Token 的认证方案选型

**30秒答**：SameSite 防 CSRF——Strict 同站才带/Lax 默认/None 必须 Secure。HttpOnly 防 XSS 读 Cookie。Secure 只在 HTTPS 传。Cookie 4KB 自动发送；Storage 5MB 不自动发送。
**追问预测**：
- "SameSite 三个值各什么意思" → Strict 同站才带；Lax 默认/允许顶级导航 GET 带；None 必须配 Secure
- "HttpOnly Cookie 防什么" → 防 XSS 读取——JS 无法通过 document.cookie 访问
- "Cookie 和 Storage 的区别" → Cookie 4KB 自动发送；Storage 5MB 不自动发送

### Q10: 浏览器存储方案对比

> 🏷️ 对比题
> ⭐⭐⭐⭐ | 难度：初级

**题目**：Cookie、LocalStorage、SessionStorage、IndexedDB 有什么区别？

**考察点**：
- Cookie：4KB、自动携带到请求、可设 HttpOnly/过期时间
- LocalStorage：5MB、永久存储、同源共享
- SessionStorage：5MB、标签页关闭即删、标签页间不共享
- IndexedDB：无上限、异步、支持索引和事务、存储结构化数据
- 实际选择：Token 存内存（XSS 安全），Refresh Token 存 HttpOnly Cookie

**30秒答**：Cookie 4KB 自动发送/可设 HttpOnly+SameSite+Secure。LocalStorage 5MB 持久化/同源共享/不自动发送。SessionStorage 5MB 标签页关闭清除。IndexedDB 无上限/异步/支持索引/存结构化数据。
**追问预测**：
- "Token 存 LocalStorage 还是 Cookie" → Cookie（HttpOnly+SameSite+Secure）防 XSS 读；LocalStorage 可被 JS 读——XSS 直接偷
- "IndexedDB 和 LocalStorage 区别" → IndexedDB 异步/无上限/支持索引/存结构化数据；LS 同步/5MB/只存字符串
- "SessionStorage 和 LocalStorage 区别" → SS 标签页关闭即清除、不跨标签共享；LS 持久化、同源标签共享

### Q11: CSP（Content Security Policy）

> 🏷️ 概念题
> ⭐⭐⭐ | 难度：高级

**题目**：什么是 CSP？如何用它防御 XSS？

**考察点**：
- `script-src` / `style-src` 等指令体系
- nonce vs hash 两种白名单方式
- `'strict-dynamic'` 的信任传递
- `report-uri` / `report-to` 违规上报
- `frame-ancestors` 替代 X-Frame-Options
- 灰度上线：Content-Security-Policy-Report-Only

**30秒答**：Content-Security-Policy——白名单控制资源加载来源。防 XSS——即使恶意脚本注入了也不在白名单里不执行。nonce 机制：每次请求生成随机 token，合法 script 带上才放行。不能防 CSRF。
**追问预测**：
- "CSP 能防什么不能防什么" → 防 XSS/data 注入/mixed content；不能防 CSRF 和业务逻辑漏洞
- "CSP nonce 机制" → 每次请求生成随机 nonce 值——合法 script 带上相同 nonce，CSP 只放行正确的
- "CSP 为什么用的人不多" → 配置严格后容易误伤第三方脚本，需要细致调试和维护

### Q12: 跨标签页通信

> 🏷️ 对比题
> ⭐⭐⭐ | 难度：中级

**题目**：至少 4 种跨标签页通信方式，各适用什么场景？

**考察点**：
- BroadcastChannel：同源广播、API 最简洁——一个标签页退出登录全部同步
- postMessage：唯一支持跨域，需持有目标 window 引用，接收方必须验证 `event.origin`
- localStorage + `storage` 事件：轻量，写入的标签页自身不触发
- SharedWorker：同源共享单一线程——多标签页共享一个 WebSocket 连接
- 选型维度：是否跨域、实时性、数据量、复杂度

**30秒答**：BroadcastChannel 同源广播最简洁、postMessage 支持跨域、localStorage+storage 事件轻量、SharedWorker 最强但复杂。
**追问预测**：
- "storage 事件的坑" → 写入的标签页自身不触发——只通知其他同源标签页，非实时不适合高频写入
- "postMessage 怎么保证安全" → 接收方校验 event.origin，发送方指定 targetOrigin 不写 *
- "多标签页怎么共享一个 WebSocket" → SharedWorker 中转——Worker 持有连接，各页面通过 port 收发

### Q13: 内存泄漏

> 🏷️ 排查题
> ⭐⭐⭐⭐⭐ | 难度：高级

**题目**：什么是内存泄漏？常见的有哪些？怎么排查？

**考察点**：
- 四种常见泄漏：全局变量、遗忘的定时器/监听器、闭包中的 DOM 引用、detached DOM
- Chrome DevTools 排查三步法：Performance 录屏 → Memory 堆快照对比 → Allocation Timeline 定位
- WeakMap/WeakRef 弱引用防御
- 框架中的泄漏陷阱（React useEffect 清理、Vue onUnmounted）
- `console.log` 的生产环境泄漏隐患

**30秒答**：常见泄漏——未清理定时器/事件监听、闭包持有大对象、detached DOM（移除但 JS 引用）、WebSocket 未关闭。排查：Performance 看 JS Heap 是否持续增长→Heap Snapshot 对比快照找增长对象。
**追问预测**：
- "怎么用 DevTools 排查内存泄漏" → Performance 看 JS Heap 是否持续增长、Memory Heap Snapshot 对比两个快照找增长
- "常见内存泄漏场景" → 未清理定时器/事件监听、闭包持有大对象、detached DOM、WebSocket 未关闭
- "WeakMap 能防内存泄漏吗" → 能——key 是弱引用，key 被回收后 value 自动被 GC

### Q14: 垃圾回收（GC）

> 🏷️ 概念题
> ⭐⭐⭐⭐ | 难度：高级

**题目**：V8 的垃圾回收机制是怎样的？

**考察点**：
- 分代回收：新生代（Scavenge，空间换时间）+ 老生代（Mark-Sweep + Mark-Compact）
- 增量标记（Incremental Marking）：把一次 GC 拆成多段，减少主线程暂停时间
- 三色标记法：白（未访问）/灰（已访问，子节点未访问）/黑（已访问，子节点已访问）
- 引用计数 vs 标记清除
- WeakMap / WeakRef 弱引用在 GC 中的作用

**30秒答**：标记清除从根可达对象出发——不可达回收，解决循环引用。引用计数记录引用次数——简单但循环引用永远不回收。V8 分代：新生代 Scavenge 快但空间换时间，老生代 Mark-Sweep-Compact。
**追问预测**：
- "标记清除和引用计数区别" → 标记清除从根出发标记可达对象——解决循环引用；引用计数简单但无法处理循环引用
- "V8 的分代 GC" → 新生代 Scavenge 复制（快/空间换时间）、老生代 Mark-Sweep-Compact
- "怎么避免内存泄漏" → 及时清理定时器/事件监听、避免意外全局变量、WeakMap 弱引用

### Q15: 多进程架构

> 🏷️ 概念题
> ⭐⭐⭐ | 难度：中级

**题目**：Chrome 浏览器有哪些进程？多进程架构有什么好处？

**考察点**：
- Browser 进程（主控）、Renderer 进程（渲染）、GPU 进程、Network 进程、Plugin 进程
- 每个 Tab 一个 Renderer 进程（Site Isolation 后同 site 的 iframe 也独立进程）
- 好处：一个 Tab 崩溃不影响其他 Tab、sandbox 安全隔离
- 缺点：内存开销大（Chrome 的内存"大"主要来自多进程）
- 与操作系统进程/线程模型的关系

**30秒答**：Chrome 一个标签一个渲染进程——崩溃互不影响。主进程管 UI、渲染进程管页面、GPU 进程管合成、插件进程隔离。进程=资源分配单位，线程=CPU 调度单位。
**追问预测**：
- "Chrome 多进程架构的优势" → 一个标签崩溃不影响其他标签、安全隔离、资源管理更清晰
- "进程和线程的区别" → 进程是资源分配的最小单位，线程是 CPU 调度的最小单位——一个进程可有多个线程
- "为什么一个标签一个进程" → 隔离——恶意页面不能访问其他页面的内存和 DOM

> 答案参考：[../浏览器/browser-architecture.md](../浏览器/browser-architecture.md)

### Q16: V8 引擎

> 🏷️ 概念题
> ⭐⭐⭐ | 难度：高级

**题目**：V8 是如何执行 JS 代码的？什么是 JIT？

**考察点**：
- Ignition 解释器 → TurboFan 优化编译器 → Sparkplug 基线编译器
- 隐藏类（Hidden Classes）：对象属性顺序一致→共享隐藏类→快速访问
- 内联缓存（Inline Cache）：记住上一次调用参数类型→跳过类型检查
- 去优化（Deoptimization）：优化假设不成立时回退到解释执行
- 单态/多态/超态对性能的影响
- `delete` 破坏隐藏类、`try-catch` 阻止优化

**30秒答**：Ignition 解释器生成字节码快速执行；TurboFan 优化编译器将热点代码编译为机器码。Hidden Class 加速属性访问——同结构对象共享隐藏类。Deoptimization 发生在假设被打破时回到解释执行。
**追问预测**：
- "Ignition 和 TurboFan 各做什么" → Ignition 解释器生成字节码快速执行；TurboFan 优化编译器——热点代码编译为机器码
- "Hidden Class 是什么" → V8 优化——相同结构的对象共享同一个隐藏类，提高属性访问速度
- "Deoptimization 什么时候发生" → 假设被打破——如对象结构变了、类型变了，回到解释执行

### Q18: DOM 事件流（捕获 / 冒泡 / 委托）

> 🏷️ 概念题
> ⭐⭐⭐⭐ | 难度：中级

**题目**：DOM 事件流三阶段、事件委托原理、`e.target` vs `e.currentTarget` 区别。

**考察点**：
- 事件流三阶段：捕获（window → 目标）→ 目标 → 冒泡（目标 → window）
- `addEventListener` 第三参数：`capture` / `once` / `passive` / `signal`
- 事件委托原理：利用冒泡在父元素统一监听，`e.target.closest()` 定位实际来源
- `e.target`（实际触发元素）vs `e.currentTarget`（绑定监听的元素，即 this）
- `stopPropagation` / `stopImmediatePropagation` / `preventDefault` 的区别
- 不冒泡的事件无法委托：`focus`/`blur`、`mouseenter`/`mouseleave`

**30秒答**：捕获→目标→冒泡。事件委托利用冒泡——父元素统一监听，省内存、动态元素自动生效。e.target 是触发元素，e.currentTarget 是绑监听的元素。
**追问预测**：
- "10000 个 li 怎么绑定事件" → 委托到 ul 一个监听器——e.target.closest('li') 定位来源，动态增删自动生效
- "哪些事件不能委托" → focus/blur、mouseenter/mouseleave 不冒泡——用 focusin/focusout、mouseover/mouseout 替代
- "passive: true 有什么用" → 承诺不调 preventDefault——浏览器不等 JS 执行完直接合成滚动帧，滚动不卡顿

> 答案参考：[../浏览器/dom-event-delegation.md](../浏览器/dom-event-delegation.md)

### Q19: Service Worker 与 PWA

> 🏷️ 概念题
> ⭐⭐⭐ | 难度：高级

**题目**：Service Worker 是什么？它的生命周期是怎样的？

**考察点**：
- SW 是独立于页面的后台脚本，拦截网络请求，管理缓存
- 生命周期：register → install → waiting → activate → fetch
- 缓存策略：Cache First / Network First / Stale-While-Revalidate
- 离线可用、消息推送、后台同步
- 只能用于 HTTPS（或 localhost）
- 与 Web Worker 的区别（生命周期独立于页面、可离线运行）

**30秒答**：SW 是独立于页面的后台脚本——拦截网络请求/缓存资源/离线可用/推送通知。不能访问 DOM/localStorage。生命周期：install→waiting→activate→fetch。优先级高于 HTTP 缓存。
**追问预测**：
- "Service Worker 能做什么不能做什么" → 能拦截请求/缓存/推送/后台同步；不能访问 DOM/localStorage
- "Service Worker 的生命周期" → install→waiting→activate→fetch。更新时新 SW 在 waiting 直到旧 SW 控制的页面全部关闭
- "Service Worker 缓存和 HTTP 缓存哪个优先级高" → SW 拦截在 HTTP 缓存之前——SW 优先级最高

### Q20: BOM 和 DOM 的区别

> 🏷️ 对比题
> ⭐⭐ | 难度：初级

**题目**：什么是 BOM？BOM 和 DOM 有什么区别？

**考察点**：
- DOM：文档对象模型（HTML/XML 文档的编程接口）→ `document`
- BOM：浏览器对象模型（浏览器提供的 JS 全局能力）→ `navigator`/`screen`/`location`/`history`
- DOM 有 W3C 规范标准，BOM 没有统一规范（各浏览器自行实现）
- `location.href` vs `location.replace` 的区别
- `URLSearchParams` 解析 URL 参数
- `navigator.onLine` 的局限性

**30秒答**：BOM 是浏览器提供的全局对象——window/navigator/screen/location/history。DOM 是 document 对象——操作页面内容。BOM 没有统一标准，DOM 有 W3C 规范。
**追问预测**：
- "BOM 的核心对象有哪些" → window/navigator/screen/location/history——都是浏览器提供的全局对象
- "location.href 和 location.replace 区别" → href 赋值新增历史记录可后退；replace 替换当前记录不可后退
- "navigator 有哪些常用信息" → userAgent/onLine/language/hardwareConcurrency/cookieEnabled

> 答案参考：[../浏览器/bom.md](../浏览器/bom.md)
