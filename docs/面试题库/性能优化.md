### Q1: 首屏加载优化全链路
> ⭐⭐⭐⭐⭐ | 难度：高级 | 场景题

**题目**：首屏加载从 3 秒优化到 1 秒，你怎么做？从网络、资源、渲染三个层面说。

**回答框架**：
1. **网络层面**：CDN 加速、HTTP/2 多路复用、DNS 预解析、资源预加载（preload/prefetch/preconnect）
2. **资源层面**：路由懒加载、Tree Shaking、按需引入、代码分割（splitChunks）、Gzip/Brotli 压缩
3. **渲染层面**：骨架屏/SSR、关键 CSS 内联、字体 preload、图片懒加载、虚拟列表

**追问预测**：
- "你怎么衡量优化效果？" → LCP / TTI / 首屏 JS 大小
- "如果后端接口慢怎么办？" → 骨架屏提升感知、BFF 聚合、乐观更新

### Q2: Web Vitals 核心指标
> ⭐⭐⭐⭐⭐ | 难度：高级 | 概念题

**题目**：Web Vitals 的核心指标有哪些？LCP、INP、CLS 分别衡量什么？

**30 秒版**：LCP 衡量加载速度（首屏最大元素渲染时间），INP 衡量交互响应（用户交互到页面反馈的延迟），CLS 衡量视觉稳定性（页面布局偏移量）。

**2 分钟版**：LCP < 2.5s、INP < 200ms、CLS < 0.1。LCP 受首屏大图/字体/阻塞 JS 影响最大；INP 取代了 FID（首次输入延迟），因为 FID 只测第一次交互；CLS 的核心是无预留空间的动态插入——图片/广告/字体加载后把内容挤开。

**追问预测**：
- "怎么测量这些指标？" → Lighthouse（实验室）/ RUM（真实用户）/ web-vitals 库
- "LCP 优化怎么下手？" → 减少阻塞资源、preload 关键资源、CDN 加速

> 📖 答案参考：[Web Vitals](../性能优化/web-vitals.md)
> 🎤 回答稿：[项目性能优化](../面试回答/项目/project-optimization.md)

### Q3: 虚拟列表原理
> ⭐⭐⭐⭐⭐ | 难度：高级 | 手写题

**题目**：大数据列表（10000+ 条）怎么渲染？虚拟列表的实现原理是什么？

**你需要的骨架**：
1. 只渲染可视区域 + 上下缓冲区的 DOM 节点（通常 20-30 个）
2. 用 `startIndex` / `endIndex` 控制渲染范围
3. 用 `transform: translateY(offset)` 撑开滚动高度
4. 监听 `scroll` 事件 → 更新 `startIndex` → 重新渲染

**别踩的坑**：
- 每项高度不固定 → 需要用预估高度 + 实测值缓存（动态高度虚拟列表）
- 滚动白屏 → 缓冲区设太小，留 3-5 项余量

**追问预测**：
- "动态高度的虚拟列表怎么做？" → 维护高度缓存 Map，偏移量累加计算
- "React/Vue 里怎么实现？" → 核心逻辑与框架无关，用 ref 拿到容器 DOM

> 📖 答案参考：[虚拟列表](../性能优化/virtual-list.md)
> ✍️ 手写参考：[大数据表格](../项目实战/业务场景/big-data-table.md)

### Q4: 缓存策略体系
> ⭐⭐⭐⭐⭐ | 难度：高级 | 场景题

**题目**：前端缓存有哪些层次？你的项目里 HTML/JS/CSS/图片/API 分别怎么缓存？

**回答框架**：
1. **四层缓存**：Service Worker（应用层）→ Memory Cache（内存）→ HTTP 缓存（磁盘）→ CDN 缓存（边缘）
2. **HTML**：`Cache-Control: no-cache` —— 每次验证，保证发版后用户立刻拿到新版本
3. **JS/CSS（带 hash）**：`max-age=31536000, immutable` —— hash 变了就是新文件
4. **图片/字体**：`max-age=86400` + ETag —— 短期强缓存 + 协商兜底
5. **API GET**：不缓存或 `max-age=10`（短缓存），POST/PUT/DELETE 不缓存

**追问预测**：
- "为什么 HTML 不设强缓存？" → 发版后 JS/CSS hash 变了但旧 HTML 还在缓存 → 白屏
- "Service Worker 和 HTTP 缓存什么关系？" → SW 拦截在 HTTP 缓存之前，优先级更高

> 📖 答案参考：[缓存策略体系](../性能优化/caching-strategy.md)
> 🎤 回答稿：[浏览器缓存](../面试回答/浏览器/cache.md)

### Q5: 资源提示（Resource Hints）
> ⭐⭐⭐⭐ | 难度：中级 | 概念题

**题目**：preload、prefetch、preconnect、dns-prefetch 分别是什么？什么时候用哪个？

**30 秒版**：preload 当前页面一定需要的资源（字体、首屏图）、prefetch 下个页面可能需要的资源（懒加载）、preconnect 提前建立第三方连接（CDN、API）、dns-prefetch 只做 DNS 解析（轻量版 preconnect）。

**2 分钟版**：preload 用 `as` 属性指定资源类型决定优先级。preload 后必须被消费（页面里用到同一 URL），否则浏览器会警告。preconnect 不要滥用——每个连接占用资源，只对确定会用的第三方域名做。

**追问预测**：
- "preload 和 prefetch 同时用会怎样？" → 加载两次，浏览器警告
- "HTTP/2 下还需要这些吗？" → 需要——多路复用不解决"早发现资源"的问题

> 📖 答案参考：[网络传输优化](../性能优化/network-optimization.md)

### Q6: 图片优化全方案
> ⭐⭐⭐⭐ | 难度：中级 | 场景题

**题目**：前端图片优化有哪些手段？从格式、加载、渲染三个角度说。

**回答框架**：
1. **格式**：WebP/AVIF 替代 PNG/JPG（体积小 30-50%）、SVG 替代简单图标、响应式图片 `srcset` + `sizes`
2. **加载**：`loading="lazy"` 原生懒加载、IntersectionObserver + 占位图、首屏图设 `fetchpriority="high"`
3. **渲染**：`width`/`height` 预留空间防 CLS、`object-fit`/`object-position`、渐进式 JPEG / LQIP 提升感知速度

**追问预测**：
- "WebP 兼容性怎么处理？" → `<picture>` 标签 + fallback、CDN 自动转换
- "图片 CDN 怎么做？" → 上传原图、CDN 按 URL 参数裁剪/压缩/转格式

> 📖 答案参考：[图片优化](../性能优化/image-optimization.md)

### Q7: 打包优化实践
> ⭐⭐⭐⭐ | 难度：高级 | 场景题

**题目**：你的项目首屏 bundle 从 2MB 优化到 400KB，怎么做到的？

**回答框架**：
1. **分析**：`rollup-plugin-visualizer` 或 `webpack-bundle-analyzer` 生成 treemap 找大包
2. **按需引入**：组件库（Element Plus on-demand）、图表库（ECharts 只引用到的图表类型）
3. **路由懒加载**：`() => import()` 拆页面 chunk
4. **依赖优化**：Moment.js → dayjs（200KB→2KB）、lodash → lodash-es + Tree Shaking
5. **splitChunks / manualChunks**：vendor（node_modules）+ common（公共组件）+ per-page

**追问预测**：
- "你怎么保证优化成果不被后续破坏？" → CI 中打包体积检查，超阈值报警
- "HTTP/2 下还要拆包吗？" → 要——不是减少请求数的问题，是缓存粒度的问题

> 📖 答案参考：[打包优化](../性能优化/bundle-optimization.md)
> 🎤 回答稿：[Vite / Webpack](../面试回答/工程化/vite-webpack.md)

### Q8: requestAnimationFrame 与动画性能
> ⭐⭐⭐⭐ | 难度：中级 | 概念题

**题目**：requestAnimationFrame 和 setTimeout(cb, 16ms) 做动画有什么区别？为什么 rAF 更流畅？

**30 秒版**：rAF 跟随浏览器刷新率（通常 60fps = 16.6ms 一帧），浏览器会在重绘前调用回调。setTimeout 不受帧率同步——可能在帧中间执行导致丢帧。

**2 分钟版**：rAF 的优势：1) 自动匹配屏幕刷新率（120Hz 屏幕 = 8.3ms 一帧）；2) 页面不可见时自动暂停（节省 CPU）；3) 在浏览器重绘之前执行（减少无效计算）。rAF 也有 `requestIdleCallback` 配合——在帧末尾的空闲时间做低优先级工作。

**追问预测**：
- "用 rAF 实现一个 JS 动画？" → 每帧更新位置 + 判断边界 + `cancelAnimationFrame` 停止
- "rAF 和 CSS animation 怎么选？" → CSS animation 优先（GPU 加速、主线程不阻塞）、复杂/交互式动画用 rAF

> 📖 答案参考：[requestAnimationFrame](../浏览器/request-animation-frame.md)

### Q9: 内存泄漏排查
> ⭐⭐⭐⭐ | 难度：高级 | 排查题

**题目**：页面越用越卡，怎么排查是不是内存泄漏？常见的泄漏场景有哪些？

**排查步骤**：
1. DevTools Performance 面板录制 → 看 JS Heap 是否持续增长不回落
2. Memory 面板 → Heap Snapshot（对比两个快照，看哪些对象在增长）→ 找到 detached DOM 或持续增长的数组/闭包
3. 常见泄漏：未清理的定时器/事件监听、闭包持有大对象、detached DOM（移除节点但 JS 还引用着）、WebSocket 未关闭

**追问预测**：
- "WeakMap 能防内存泄漏吗？" → 能——key 是弱引用，key 被回收后 value 自动被 GC
- "你怎么定位具体哪行代码导致的？" → Heap Snapshot 的 Retainers 视图追踪引用链

> 📖 答案参考：[内存泄漏](../浏览器/memory-leak.md)

### Q10: 骨架屏与感知性能
> ⭐⭐⭐ | 难度：中级 | 场景题

**题目**：骨架屏（Skeleton Screen）和 Loading 转圈有什么区别？什么时候用骨架屏？

**回答框架**：
1. **骨架屏提升的是感知性能**——用户看到"内容在加载中"比看到"转圈"更有耐心
2. 骨架屏适合内容型页面（列表、卡片、详情），不适用交互型（表单、图表）
3. 实现：手动写骨架组件、自动生成（如 `react-content-loader`）、SSR 预渲染
4. 和 Loading 的区别：骨架屏暗示"内容即将出现"（时间感），Loading 暗示"请等待"（无时间感）

**追问预测**：
- "骨架屏怎么做动画？" → CSS 渐变动画（shimmer effect），避免复杂动画占主线程
- "如何避免骨架屏和真实内容的布局抖动？" → 骨架屏的占位尺寸和真实内容一致

> 📖 答案参考：[首屏优化](../性能优化/first-screen.md)

### Q11: 长任务拆分与 Total Blocking Time 优化

> ⭐⭐⭐ | 难度：中高级

**题目**：什么是长任务（Long Task）？如何拆分长任务降低 TBT？

**30秒答**：超过 50ms 的 JS 任务阻塞主线程——影响 INP 指标。拆分方案：scheduler.yield() 主动让出主线程、requestIdleCallback 利用空闲时间、Web Worker 移走计算内核。本质——把一锅炖拆成小碗分批吃。

> 📖 答案参考：[首屏优化](../性能优化/first-screen.md)
