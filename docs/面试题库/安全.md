### Q1: XSS 攻击的三类与防御
> ⭐⭐⭐⭐⭐ | 难度：中级 | 概念题

**题目**：XSS 有哪三种类型？你在项目里怎么防御 XSS？

**30 秒版**：反射型（URL 参数直接拼进 HTML）、存储型（恶意代码存在数据库里）、DOM 型（JS 取了不可信数据写进 DOM）。防御分四层：输出转义、HttpOnly Cookie、CSP 白名单、输入校验。

**2 分钟版**：反射型攻击在请求中（URL 参数），存储型攻击持久化（评论、用户资料），DOM 型攻击纯粹发生在客户端（JS 从 URL hash 取值插入页面）。关键是"数据出路的转义"——HTML 正文、属性值、script 块、URL 参数各需要不同的转义函数。CSP 是最后一道防线——即使某处 XSS 漏了，`Content-Security-Policy` 白名单阻止恶意脚本的执行。

**追问预测**：
- "innerHTML 和 textContent 哪个安全？" → textContent 始终当纯文本
- "CSP 的 nonce 怎么用？" → 每次请求生成随机 nonce，合法 script 标签带上相同 nonce

> 📖 答案参考：[XSS / CSRF](../浏览器/安全/xss.md)
> 🎤 回答稿：[XSS / CSRF](../面试回答/浏览器/xss-csrf.md)

### Q2: CSRF 攻击与防御
> ⭐⭐⭐⭐⭐ | 难度：中级 | 概念题

**题目**：CSRF 是什么？和 XSS 有什么本质区别？怎么防御？

**30 秒版**：XSS 是在受害者的页面里执行恶意脚本；CSRF 是利用受害者已登录的身份发送恶意请求。本质区别——XSS 攻客户端（代码注入），CSRF 盗身份（请求伪造）。防御：SameSite Cookie + CSRF Token + Origin/Referer 校验。

**2 分钟版**：CSRF 的攻击条件是：用户已登录网站 A、网站 B 诱导用户发请求到 A、浏览器自动带 Cookie。防御的三道防线：1) SameSite=Lax（默认）——跨站请求不带 Cookie；2) CSRF Token——每次请求验证一个随机 token（token 不能放 Cookie 里，放请求头或 body）；3) 校验 Origin/Referer 头——看请求来源是不是合法的。

**追问预测**：
- "CSRF Token 为什么不能放 Cookie？" → Cookie 会被浏览器自动带上——攻击者也带着 token
- "SameSite 的三个值各什么意思？" → Strict（同站 GET 才带）、Lax（默认/允许顶级导航 GET）、None 必须配 Secure

> 📖 答案参考：[XSS / CSRF](../浏览器/安全/csrf.md)
> 🎤 回答稿：[XSS / CSRF](../面试回答/浏览器/xss-csrf.md)

### Q3: CSP 内容安全策略
> ⭐⭐⭐⭐ | 难度：中级 | 概念题

**题目**：CSP 是什么？怎么配置？它能防什么、不能防什么？

**回答框架**：
1. CSP = Content-Security-Policy，HTTP 头或 `<meta>` 标签声明资源加载白名单
2. 核心指令：`script-src`、`style-src`、`img-src`、`connect-src`、`default-src`
3. 能防：XSS（阻止 inline script）、数据注入、mixed content
4. 不能防：业务逻辑漏洞、CSRF（另需防范）
5. `nonce` / `hash` 让合法的 inline script 通过
6. `report-uri` / `report-to` 收集违规报告，监控攻击尝试

**追问预测**：
- "CSP 和 XSS 防御的关系？" → CSP 是 XSS 的第二道防线——它假设 XSS 可能发生，然后限制攻击者能做什么
- "为什么 CSP 用的人不多？" → 配置严格后可能误伤第三方脚本（如监控 SDK），需要细致调试

> 📖 答案参考：[CSP](../浏览器/安全/csp.md)
> 🎤 回答稿：[CSP](../面试回答/浏览器/csp.md)

### Q4: Token 存储安全
> ⭐⭐⭐⭐⭐ | 难度：中级 | 对比题

**题目**：Token 应该存在 Cookie 里还是 LocalStorage 里？HttpOnly、SameSite、Secure 各有什么用？

**维度对比**：

| 方案 | 防 XSS | 防 CSRF | 复杂度 |
|------|--------|---------|--------|
| LocalStorage | ❌ JS 可直接读取 | ✅ 不会自动发送 | 低 |
| Cookie (HttpOnly + Secure + SameSite) | ✅ JS 不可读 | ✅ SameSite 控制 | 中 |
| 内存 + Refresh Token (BFF 方案) | ✅✅ 最佳 | ✅✅ 最佳 | 高 |

**一句话本质**：HttpOnly Cookie 防 XSS 读 Token、SameSite 防 CSRF 带 Token、Secure 防中间人窃听。三者组合是生产环境的标准配置。

**追问预测**：
- "Access Token 和 Refresh Token 分别存哪？" → Access Token 存内存（JS 变量）、Refresh Token 存 HttpOnly Cookie
- "为什么不把所有 Token 都放 HttpOnly Cookie？" → Access Token 需要被前端 JS 读取以附加到请求头

> 📖 答案参考：[Token 存储安全](../浏览器/安全/token-storage.md)
> 🎤 回答稿：[Token 存储安全](../面试回答/浏览器/token-storage.md) · [登录鉴权](../面试回答/项目/login-auth.md)

### Q5: 点击劫持（Clickjacking）
> ⭐⭐⭐ | 难度：中级 | 概念题

**题目**：点击劫持是什么？怎么防御？

**30 秒版**：攻击者把目标网站放在透明的 iframe 里覆盖在诱饵按钮上——用户以为在点诱饵，实际在点隐藏的目标网站按钮（如转账）。防御：`X-Frame-Options: DENY` 或 `Content-Security-Policy: frame-ancestors 'none'`。

**2 分钟版**：点击劫持利用了 iframe 的透明叠加。防御：1) `X-Frame-Options`（老，DENY/SAMEORIGIN）；2) CSP `frame-ancestors`（新，更灵活，可以用 CSP report 监控攻击）。还可以用 JS 判断 `window.top !== window.self` 但容易被绕。

**追问预测**：
- "X-Frame-Options 和 frame-ancestors 有什么区别？" → frame-ancestors 更灵活（能指定允许哪些域名嵌入），且支持 CSP 报告
- "iframe 的 sandbox 能做防护吗？" → sandbox 是限制 iframe 里页面的能力，不是防点击劫持的

> 📖 答案参考：[点击劫持与 iframe 安全](../浏览器/安全/clickjacking.md)

### Q6: 中间人攻击（MITM）与 HTTPS
> ⭐⭐⭐⭐ | 难度：中级 | 概念题

**题目**：中间人攻击是什么？HTTPS 如何防御中间人攻击？

**回答框架**：
1. MITM 的核心：攻击者在通信双方之间窃听/篡改数据
2. HTTPS 防御 MITM 的三步：1) 非对称加密（RSA/ECDH）安全交换对称密钥；2) 证书链验证（根 CA→中间 CA→网站证书）防止伪造身份；3) 对称加密保护数据传输
3. 前提：用户没有安装不可信根证书、没有忽略证书警告

**追问预测**：
- "Charles/Fiddler 怎么能抓 HTTPS 包？" → 用户安装了 Charles 的根证书——等于信任了 Charles 这个"中间人"
- "TLS 1.3 相比 1.2 改进了什么？" → 简化握手（1-RTT）、去掉不安全加密算法

> 📖 答案参考：[HTTPS 与传输安全](../浏览器/安全/https-security.md) · [HTTP / HTTPS](../网络/http-https.md)
> 🎤 回答稿：[HTTP / HTTPS](../面试回答/网络/http-https.md)

### Q7: 前端依赖安全（Supply Chain Security）
> ⭐⭐⭐ | 难度：中级 | 概念题

**题目**：npm 依赖包可能存在哪些安全风险？如何防范？

**回答框架**：
1. **风险类型**：恶意代码（如 event-stream 事件）、原型污染（lodash 漏洞）、被篡改的包（拼写欺骗如 `cross-env` vs `crossenv`）
2. **防范**：`npm audit` / `yarn audit` 扫描已知漏洞、锁版本（`package-lock.json` / `pnpm-lock.yaml`）、审查依赖许可证、CI 中集成安全检查
3. **SRI**（Subresource Integrity）：`<script integrity="sha384-...">` 确保 CDN 资源未被篡改

**追问预测**：
- "lock 文件的作用？" → 锁定依赖树的精确版本，防止 CI/生产环境安装到不同版本
- "怎么发现原型污染？" → `Object.freeze(Object.prototype)` 或使用 `hasOwnProperty` 检查

> 📖 答案参考：[前端依赖安全](../浏览器/安全/supply-chain-security.md) · [npm 深入](../工程化/npm-deep.md)

### Q8: 同源策略与跨域安全
> ⭐⭐⭐⭐⭐ | 难度：中级 | 概念题

**题目**：同源策略限制了什么？跨域有哪些方案？CORS 的简单请求和预检请求有什么区别？

**回答框架**：
1. 同源策略限制：DOM 访问、Cookie 读取、AJAX 请求（三个维度）
2. 跨域方案：CORS（标准）、JSONP（只支持 GET/古老）、postMessage（iframe 通信）、代理（Nginx/devServer proxy）、WebSocket（不受同源限制）
3. CORS 预检：非简单请求（`Content-Type: application/json`、自定义头、非 GET/POST/HEAD 方法）→ 浏览器发 OPTIONS 预检 → 服务器返回 `Access-Control-Allow-*` → 浏览器再发正式请求

**追问预测**：
- "JSONP 为什么能跨域？" → 利用了 `<script>` 标签不受同源限制——本质上请求一个 JS 文件返回
- "简单请求的条件？" → GET/POST/HEAD、Content-Type 仅限于 text/plain 等三种、无自定义头

> 📖 答案参考：[同源策略](../浏览器/same-origin-policy.md)
> 🎤 回答稿：[跨域 / CORS](../面试回答/网络/cors.md)
