# 前端架构 高频面试题

> 收录前端面试中的高频前端架构真题，共 9 题。
> 微前端 / Monorepo 已是 3 年经验中级岗二面的常规题，考法是：背景痛点 → 方案选型 → 收益 → 缺点。架构题考察的是"你能否从零设计一个项目的技术架构"而非具体 API。

### Q1: 你怎么设计一个大型前端项目的架构？

> ⭐⭐⭐⭐ | 难度：中高级

**题目**：从零开始一个大型后台管理系统，你如何做技术选型和架构设计？

**30秒答**：分层——项目骨架（路由/权限/布局/请求封装/错误处理）先搭好。目录按功能分而非按技术分。组件分通用组件（UI 库二次封装）和业务组件（耦合 Pinia/API）。状态管理：全局状态 Pinia、局部状态 composable、表单状态 reactive。

> 答案参考：[../前端架构/project-structure.md](../前端架构/project-structure.md)

### Q2: 你怎么设计一个可复用的组件？

> ⭐⭐⭐⭐ | 难度：中级

**题目**：如何设计一个通用组件？如何平衡灵活性和复杂度？

**30秒答**：三原则——props 进 emits 出（纯函数语义）、插槽留扩展点（不写死内部结构）、类型导出（TS 约束调用方）。参考 Element Plus——props 控制样式/数据/行为、slots 控制内容、events 通知父组件。

> 答案参考：[../前端架构/component-design.md](../前端架构/component-design.md)

### Q3: 微前端是什么？什么场景适合用？

> ⭐⭐⭐⭐ | 难度：中高级

**题目**：微前端解决了什么问题？qiankun 和 Module Federation 有什么区别？

**考察点**：
- 背景痛点：巨石应用构建慢、多团队发布互相阻塞、技术栈锁定、协作冲突
- 五方案对比与选型：qiankun / micro-app / wujie / Module Federation / iframe
- 应用级拆分（qiankun）vs 模块级共享（Module Federation）的本质差异

**30秒答**：微前端把巨石应用拆成独立开发、独立部署、技术栈无关的子应用，解决多团队构建慢、发布互相阻塞的问题。方案选型我按五个说：qiankun 生态最成熟——路由分发+沙箱+样式隔离+通信开箱即用；micro-app 组件式接入——像写一个自定义标签一样挂子应用，改造成本最低；wujie 用 iframe 执行 JS 加 Proxy 通信，隔离和体验双保险；Module Federation 是构建级模块共享——运行时拉取远程模块，适合模块粒度复用而非应用粒度拆分；iframe 隔离最强但体验最差，只适合嵌完全独立的第三方页面。

**追问预测**：
- "为什么不用 iframe 做微前端" → 四个硬伤：弹窗只能在 iframe 区域内居中、刷新丢状态、URL 不同步（前进后退失效）、通信只能 postMessage 很繁琐——所以 iframe 只适合嵌第三方页面
- "Module Federation 的 shared 依赖怎么处理" → shared 里声明 vue/lodash 等公共依赖，关键配置 `singleton: true`——保证运行时全局只有一个实例，避免两份 Vue 各自维护响应式导致诡异 bug

> 答案参考：[../前端架构/微前端/overview.md](../前端架构/微前端/overview.md)

### Q4: Monorepo 是什么？为什么用？怎么选？

> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：什么是 Monorepo？和 Multirepo 相比有什么优劣？什么样的项目适合引入？

**考察点**：
- 与 Multirepo 对比：代码共享、原子提交（一次 PR 联动多包）、统一工具链 vs 权限粒度、仓库体积
- 适用判断：多个包互相依赖才值得——不相干的项目硬放一个仓库是自找麻烦
- 必须会讲缺点：CI 复杂度、全量 clone 体积、权限边界——只会讲好处等于没真用过
- 配套工具链：pnpm workspace + Turborepo/Nx 增量构建 + Changesets 版本管理

**30秒答**：Monorepo 是把多个互相依赖的包放在一个仓库统一管理——比如组件库、shared 工具包、主应用放一起。三个核心收益：代码共享不用先发 npm 包、改 shared 一次提交原子性地联动所有下游、ESLint/TS 配置一份约束所有包。但它有真实成本：CI 必须上增量构建（Turborepo/Nx）否则每次全量跑、仓库体积越滚越大 clone 慢、权限是仓库粒度没法按包隔离。所以我的判断标准是：包之间有真实依赖关系（改一个要联动另一个）才用，几个不相干的项目塞一个仓库只会背上所有缺点。

**追问预测**：
- "Monorepo 有什么缺点" → CI 复杂度（要配增量构建和缓存）、全量 clone 体积大、权限没法按包隔离——面试官用这题过滤"只看过文章没实际用过"的候选人
- "多大规模适合 Monorepo" → 5-15 个互相依赖的包最合适；50+ 独立应用/微服务还是 Multirepo
- "多包的版本号怎么管" → Changesets 独立版本——改哪个包发哪个包，不搞全仓库统一版本号

> 答案参考：[../前端架构/monorepo.md](../前端架构/monorepo.md)

### Q5: pnpm workspace 是怎么管理 Monorepo 依赖的？

> ⭐⭐⭐⭐ | 难度：中高级

**题目**：pnpm workspace 的 `workspace:*` 协议做了什么？为什么 Monorepo 首选 pnpm 而不是 npm/yarn？

**考察点**：
- `workspace:*` 协议：本地包互相引用，publish 时自动替换为真实版本号
- 软链接机制：node_modules 里是指向本地源码的 symlink——改 shared 立即生效，无需重新 install
- 公共依赖提根：devDependencies（TypeScript/ESLint/turbo）放根 package.json，子包只保留各自的 dependencies
- pnpm 的两大优势：硬链接全局 store 省磁盘 + 严格（非扁平）node_modules 杜绝幽灵依赖

**30秒答**：pnpm-workspace.yaml 声明哪些目录是包，包之间用 `workspace:*` 协议互相引用——pnpm 在 node_modules 里创建指向本地源码的软链接，改 shared 的代码主应用立即生效，不用重新 install；publish 时协议自动替换成真实版本号。公共的 devDependencies 提到根目录一份管理，子包只留自己的运行时依赖。Monorepo 首选 pnpm 有两个原因：一是硬链接指向全局 store，同一个依赖整台机器只存一份，几十个包也不爆磁盘；二是 node_modules 是严格的非扁平结构，没在 package.json 里声明的包根本引不到，从机制上杜绝幽灵依赖——幽灵依赖细节点到为止，详见工程化模块。

**追问预测**：
- "幽灵依赖在 Monorepo 里为什么危害更大" → 子包偷偷用了 shared 的间接依赖，shared 升级换掉这个依赖后子包运行时悄悄崩——包越多这种隐式耦合越难排查
- "workspace:\* 发布到 npm 后消费方怎么装" → pnpm publish 时自动替换为实际版本（如 `^1.2.0`），消费方看到的就是普通依赖
- "软链接有什么坑" → 个别工具对 symlink 的路径解析有问题（如老版本 jest/webpack），需要注意 resolve 相关配置

> 答案参考：[../前端架构/monorepo.md](../前端架构/monorepo.md)
> 延伸：[../工程化/pnpm.md](../工程化/pnpm.md)

### Q6: qiankun 的 JS 沙箱是怎么实现的？

> ⭐⭐⭐⭐⭐ | 难度：中高级

**题目**：为什么微前端需要 JS 沙箱？qiankun 的三种沙箱（SnapshotSandbox / LegacySandbox / ProxySandbox）分别是什么原理？

**考察点**：
- 为什么需要沙箱：子应用直接读写 window（全局变量/定时器/事件），切换后污染残留影响其他应用
- SnapshotSandbox：激活时给 window 拍快照，失活时 diff 记录修改并还原——兼容 IE，但要遍历 window，仅单实例
- LegacySandbox：用 Proxy 记录子应用对 window 的新增/修改，失活时还原——省去遍历，但仍操作真实 window，仅单实例
- ProxySandbox：fakeWindow + Proxy——写操作进 fakeWindow 不碰真实 window，支持多实例，主流方案
- 子应用代码被包进 `with(proxy)` 的函数作用域执行，让代码里的 window 指向代理对象

**30秒答**：子应用会往 window 上挂变量、定时器、事件，不隔离的话切换应用后污染残留——所以需要沙箱。qiankun 有三种：SnapshotSandbox 激活时给 window 拍快照、失活时 diff 出子应用的修改记录下来并还原 window，不依赖 Proxy 所以兼容 IE，缺点是每次都要遍历 window 且只能单实例；LegacySandbox 用 Proxy 拦截 set、记录子应用改了哪些属性，失活时还原——省掉遍历，但写的还是真实 window，同样单实例；ProxySandbox 是主流——给每个子应用造一个 fakeWindow，Proxy 拦截后写操作全进 fakeWindow，真实 window 完全不被污染，所以多个子应用能同时运行。子应用的 JS 会被包在 `with(proxy)` 的函数里执行，代码里访问 window 实际拿到的是代理对象。

**追问预测**：
- "ProxySandbox 里读 window.location 这类属性会怎样" → fakeWindow 上没有的属性兜底从真实 window 读——读共享、写隔离
- "为什么前两种沙箱只能单实例" → 它们的修改/还原都作用在唯一的真实 window 上，两个子应用同时激活会互相覆盖
- "沙箱能拦住全部副作用吗" → 不能——setInterval、addEventListener、动态 `<style>`/`<script>` 需要额外劫持，qiankun 对这些也做了 patch，unmount 时统一清理

> 答案参考：[../前端架构/微前端/qiankun.md](../前端架构/微前端/qiankun.md)

### Q7: 微前端的样式隔离有哪些方案？

> ⭐⭐⭐⭐ | 难度：中级

**题目**：子应用之间样式互相污染怎么办？qiankun 的 strictStyleIsolation 和 experimentalStyleIsolation 有什么区别？

**考察点**：
- Shadow DOM：浏览器原生隔离最彻底，但挂到 document.body 的弹窗跑到 Shadow 树外——样式丢失
- 工程约定：命名空间前缀（postcss 插件统一加 `.app-xxx` 前缀）/ CSS Modules / scoped
- strictStyleIsolation（Shadow DOM 包裹）vs experimentalStyleIsolation（选择器加属性前缀）的原理与坑

**30秒答**：三类方案。一是 Shadow DOM——qiankun 的 strictStyleIsolation 就是把子应用包进 shadowRoot，隔离最彻底，但 element-plus 这类弹窗默认挂 document.body、跑到 Shadow 树外直接丢样式；二是工程约定——所有选择器加应用前缀，postcss 插件自动处理，或者 CSS Modules/scoped，成本最低也最稳；三是 qiankun 的 experimentalStyleIsolation——运行时给子应用每条样式规则加 `div[data-qiankun="应用名"]` 属性前缀，类似 Vue scoped 的思路，弹窗不受影响，但它是实验性功能、对动态插入的 `<style>` 标签处理不完善。实践中一般用约定前缀打底，qiankun 的隔离做兜底。

**追问预测**：
- "strictStyleIsolation 的弹窗问题怎么解决" → 给弹窗指定挂载点到子应用容器内（如 el-dialog 的 append-to），或退回 experimentalStyleIsolation / 命名空间约定
- "experimentalStyleIsolation 的原理" → 遍历子应用样式表，给每条规则加属性选择器前缀——和 Vue scoped 的 data-v 属性同思路
- "子应用改了 body 的全局样式怎么办" → 属性前缀拦不住 body/html 级别的样式——只能靠约定（子应用不写全局样式）+ unmount 时移除样式表

> 答案参考：[../前端架构/微前端/qiankun.md](../前端架构/微前端/qiankun.md)

### Q8: 微前端应用之间怎么通信？

> ⭐⭐⭐⭐ | 难度：中级

**题目**：主应用和子应用、子应用和子应用之间如何传递数据？各方案怎么选？

**考察点**：
- props 下发：注册子应用时传 token/用户信息——父传子最简单
- initGlobalState：qiankun 官方全局状态池——观察者模式，setGlobalState / onGlobalStateChange
- CustomEvent 自定义事件（不依赖框架、完全解耦）vs postMessage（iframe 跨文档场景）
- 共享 store 的风险：主子应用共用 Pinia/Redux 实例 = 深度耦合，破坏独立性
- 选型原则：通信越少越好——频繁通信说明拆分边界画错了

**30秒答**：四种方式。一是 props 下发——注册子应用时把 token、用户信息传进去，父传子的静态数据够用；二是 qiankun 官方的 initGlobalState——本质是观察者模式的全局状态池，各应用 setGlobalState 修改、onGlobalStateChange 监听，适合登录态、主题这类少量低频数据；三是事件——CustomEvent 完全解耦不依赖框架，iframe 场景用 postMessage；四是共享 store，但我不推荐——主子应用共用一个 store 实例是深度耦合，子应用没法独立运行，微前端白拆了。我的原则是通信越少越好，两个子应用频繁互相通信，说明拆分边界本身画错了，应该合并或重新划分。

**追问预测**：
- "initGlobalState 的实现原理" → 观察者模式——内部维护一份全局 state 和各应用注册的回调，setGlobalState 时逐个通知变更前后的值
- "兄弟子应用之间怎么通信" → 经基座中转（globalState 或事件），不要子应用直连——保持星型拓扑，方便追踪数据流向
- "为什么不共享一个 Pinia store" → 主子应用的依赖版本被锁死、子应用脱离基座跑不起来、状态变更来源不可追踪——违背独立开发部署的初衷

### Q9: 前端常用的设计模式有哪些？项目里怎么落地？

> ⭐⭐⭐ | 难度：中级

**题目**：说几个你在前端项目中真实用过的设计模式。观察者模式和发布订阅模式有什么区别？

**考察点**：
- 发布订阅：EventBus/EventEmitter——跨组件通信、埋点分发
- 单例：全局 loading/弹窗/消息管理——防止重复创建实例
- 策略：表单校验规则表、权限映射表——替代 if-else 长链
- 观察者 vs 发布订阅：有无中介——Subject 直接持有 Observer vs 经事件中心转发、双方互不感知
- 关键信号：每个模式必须绑定真实场景，只背 GoF 定义是减分项

**30秒答**：我说三个真实用过的。发布订阅——手写 EventEmitter（on/off/emit）做跨组件通信和埋点分发，Vue3 响应式的 track/trigger 本质就是加强版发布订阅；单例——全局 Loading 服务只保留一个实例，show 的时候已存在就复用，避免叠两层遮罩；策略模式——表单校验做成规则函数表，required/email/phone 每条规则一个函数，新增规则加一个函数就行，不动校验主逻辑。观察者和发布订阅的区别在有没有中介：观察者是 Subject 直接持有并通知 Observer，比如 Vue 的 dep 收集 effect；发布订阅中间隔了个事件中心，发布者和订阅者互不认识，解耦更彻底。

**追问预测**：
- "Vue3 源码里有哪些设计模式" → 观察者（track/trigger）、代理（reactive 的 Proxy）、单例（app 实例）、策略（diff 的快速路径分支）
- "EventBus 有什么坑" → 事件来源难追踪、忘记 off 造成内存泄漏——所以 Vue3 移除了 $on/$off，跨组件状态优先用 Pinia
- "什么时候不该上设计模式" → 3 个 if-else 就上策略模式是过度设计——先识别坏味道，重构自然长出模式

> 答案参考：[../前端架构/design-patterns.md](../前端架构/design-patterns.md)
