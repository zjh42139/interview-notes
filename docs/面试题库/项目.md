### Q1: 后台管理系统的权限设计

**30秒答**：RBAC 模型——用户绑定角色，角色绑定权限。前端三层：路由权限(addRoute 动态注册)、按钮权限(v-permission 指令)、数据权限(后端控制前端透传)。前端权限只是 UX，真正的安全在后端。

**追问预测**：
- "按钮权限怎么实现" → 自定义 v-permission 指令——mounted 时检查权限数组无权限则 removeChild
- "多角色权限怎么合并" → 取并集——用户所有角色的权限叠加
- "前端权限的局限性" → 只是 UX 优化——真正的安全校验必须在后端
> 🏷️ 场景题
> ⭐⭐⭐⭐⭐ | 难度：中高级

**题目**：请详细描述你做的后台管理系统的权限方案。包括菜单权限、按钮权限、接口权限、数据权限。前后端是如何配合的？

**考察点**：
- RBAC（基于角色的访问控制）：用户 -> 角色 -> 权限
- 路由权限：`addRoute` 动态注册路由，根据角色过滤可访问路由表
- 菜单权限：后端返回菜单树，前端递归渲染
- 按钮权限：`v-permission` 自定义指令 + 权限码字典
- 接口权限：后端中间件校验 + 前端请求拦截器根据返回码处理
- 实际细节：权限变更后的刷新策略、路由 404 兜底、默认跳转逻辑

### Q2: Token 无感刷新机制

**30秒答**：双 Token——Access Token 短期存内存，Refresh Token 长期存 HttpOnly Cookie。401→Refresh Token 换新 Access→重试原请求。并发 401 缓存刷新 Promise 只调一次刷新接口。

**追问预测**：
- "并发 401 怎么处理" → 缓存刷新中的 Promise——其他 401 请求 await 同一个 Promise
- "Refresh Token 也过期了怎么办" → 清除 Token 跳转登录——用户需重新输入密码
- "怎么判断 Token 即将过期" → 解析 JWT exp 字段——提前 5 分钟主动刷新
> 🏷️ 场景题
> ⭐⭐⭐⭐⭐ | 难度：中高级

**题目**：如何实现 Token 的无感刷新？当多个请求同时返回 401 时，如何处理并发刷新的问题？Refresh Token 过期了怎么办？

**考察点**：
- Access Token 短有效期（15-30min），Refresh Token 长有效期（7-30天）
- 请求拦截器判断 401 -> 用 Refresh Token 请求新 Access Token -> 重试原请求
- 并发 401 处理：缓存刷新中的 Promise，其他请求等待同一个刷新结果
- 请求队列：刷新期间的新请求排队，刷新成功后批量重试
- Refresh Token 也过期 -> 清除登录态 -> 跳转登录页
- 双 Token 存储策略：Access Token 放内存/变量，Refresh Token 放 HttpOnly Cookie

### Q3: Axios 封装与请求拦截

**30秒答**：请求拦截器加 Token/loading；响应拦截器处理 401/统一错误提示/数据解包。封装：baseURL+timeout+重试+取消重复请求。AbortController 取消请求——新请求前 abort 旧请求解决竞态。

**追问预测**：
- "拦截器怎么组织" → 请求拦截器加 Token/loading；响应拦截器处理 401/统一错误提示/数据解包
- "怎么处理重复请求" → Axios CancelToken 或 AbortController——同一接口未完成时取消重复
- "怎么处理请求竞态" → 后发请求的结果覆盖先发的——用请求序号或 AbortController 取消旧请求
> 🏷️ 场景题
> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：你是如何封装 Axios 的？请求拦截器和响应拦截器分别做了哪些处理？如何设计可复用、可扩展的请求模块？

**考察点**：
- 实例创建：`baseURL`、`timeout`、`withCredentials` 默认配置
- 请求拦截：Token 注入、loading 显示、重复请求取消（AbortController）
- 响应拦截：统一错误码处理、错误提示、登录态失效处理
- TypeScript 类型封装：泛型响应包装、API 方法签名
- 取消重复请求：Map 存储 pending 请求、`CancelToken` / `AbortController`
- 文件下载、上传的进度回调封装

### Q4: 虚拟列表（Virtual List）实现

**30秒答**：只渲染可视区 20-30 个 DOM 节点+transform 撑开滚动高度。startIndex/endIndex 控制范围+上下缓冲区防白屏。动态高度用预估+实测缓存。10000 条数据只渲染 20 个——性能提升 500 倍。

**追问预测**：
- "动态高度怎么办" → 预估高度+实测值缓存——渲染后 measure 更新高度 Map
- "滚动白屏怎么解决" → 缓冲区设大些——可视区上下各多渲染 3-5 项
- "和 react-window 的区别" → 原理完全一样——自己写更灵活；库封装好边界情况
> 🏷️ 手写题
> ⭐⭐⭐⭐ | 难度：中高级

**题目**：如果要展示一个 10 万条数据的表格，如何保证不卡顿？请描述虚拟列表的实现原理，包括固定高度和动态高度的处理方式。

**考察点**：
- 只渲染可视区域 + 缓冲区（前后预渲染 N 个）的 DOM 节点
- 固定高度：`startIndex = Math.floor(scrollTop / itemHeight)`、`visibleCount = containerHeight / itemHeight`
- 动态高度：维护位置缓存（positions 数组），二分查找 `startIndex`
- `transform: translateY()` 偏移可视内容，撑开总高度（占位元素）
- ResizeObserver 监听行高变化，动态更新缓存
- Element Plus Table 的虚拟化实现 vs vxe-table 对比

### Q5: 大文件上传

**30秒答**：Blob.slice 切片(5MB/片)→Web Worker 算文件 hash→后端查 hash 秒传/续传→并发上传(limit=3)切片→完成后通知合并。hash 相同=秒传；部分切片已有=断点续传。

**追问预测**：
- "秒传怎么实现" → 上传前算文件 hash→后端查 hash 是否已存在→存在直接返回已有 URL
- "hash 计算会卡主线程吗" → 大文件会——用 Web Worker 计算 hash 避免阻塞 UI
- "切片上传失败怎么重试" → 单切片失败自动重试 3 次；仍失败标记 pending 让用户选择
> 🏷️ 场景题
> ⭐⭐⭐⭐ | 难度：中高级

**题目**：如何实现一个大文件（如 2GB 视频）的上传功能？需要支持断点续传、秒传、上传进度展示、暂停/继续等功能。

**考察点**：
- 文件切片（Blob.slice）：将大文件按固定大小（如 5MB）切成多个 chunk
- SparkMD5 计算文件 hash，用于秒传和断点续传
- 秒传：服务端根据 hash 判断文件是否已存在
- 断点续传：服务端返回已上传的切片列表，前端跳过已上传的切片
- 并发上传控制（3-6 个并发请求）
- 切片合并：所有切片上传完成后服务端合并，校验完整 hash
- Web Worker 中计算 hash 避免阻塞主线程

### Q8: SSR vs CSR 对比

**30秒答**：CSR 空壳 HTML+JS 渲染——SEO 差首屏慢适合后台。SSR 服务端生成 HTML——SEO 好服务器压力大适合电商。SSG 构建时预渲染——性能极致适合博客。选型：toB 用 CSR，toC 用 SSR/SSG。

**追问预测**：
- "SSR 的代价是什么" → 服务器压力增大、部署复杂度提升、开发时需要关注同构问题
- "什么时候用 SSG 不用 SSR" → 内容不常变（博客/文档）——构建时预渲染，零服务器成本
- "Hydration 是什么" → 客户端给 SSR HTML 绑定事件和响应式——静态 HTML 变成可交互应用
> 🏷️ 对比题
> ⭐⭐⭐⭐ | 难度：中高级

**题目**：SSR（服务端渲染）和 CSR（客户端渲染）有什么区别？SSR 解决了什么问题、带来了什么挑战？如何在 Vue3 项目中实现 SSR？

**考察点**：
- CSR：空白 HTML + JS 渲染，SEO 差、首屏慢（需要 JS 下载+执行+渲染）
- SSR：服务端直接输出完整 HTML，SEO 友好、首屏快
- SSR 的代价：服务端压力大、开发复杂度高（区分服务端/客户端环境）、需要 Node.js 服务
- 同构（Isomorphic）：同一套代码在服务端和客户端都可运行
- Nuxt 3 的 SSR 模式、`useAsyncData`/`useFetch` 统一数据获取
- 脱水（Dehydration）和注水（Hydration）的过程

### Q9: 前端监控与错误上报

**30秒答**：JS 错误 window.onerror+unhandledrejection。性能 Web Vitals(LCP/INP/CLS)。上报 sendBeacon 保证页面关闭时发送。source map 还原错误位置不上传公开 CDN。采样减少数据量。

**追问预测**：
- "JS 错误怎么全局捕获" → window.onerror + unhandledrejection——原生错误和 Promise 异常
- "怎么定位生产环境错误" → Source Map 反解压缩代码→还原源码位置。Source Map 不要公开部署
- "性能监控的核心指标" → LCP/INP/CLS——Web Vitals 标准。用 web-vitals 库采集
> 🏷️ 场景题
> ⭐⭐⭐ | 难度：中高级

**题目**：前端如何做错误监控？JS 运行时错误、Promise 错误、资源加载错误、接口错误分别如何捕获？Sentry 的原理是怎样的？

**考察点**：
- `window.onerror`：捕获 JS 运行时错误（除 Promise）
- `window.addEventListener('unhandledrejection')`：捕获未处理的 Promise 异常
- `window.addEventListener('error')` 捕获阶段：资源加载失败（img/script 等）
- `console.error` 劫持 + 接口错误拦截
- Source Map 还原：上传 sourcemap 到监控平台，堆栈反向解析
- Sentry：错误分组（按堆栈指纹）、面包屑（Breadcrumbs）记录用户操作路径
- 自定义埋点：PV/UV、用户行为、性能指标上报

### Q10: 项目部署流程

**30秒答**：构建→上传 CDN→更新 HTML。静态资源 hash 文件名确保缓存更新。回滚 HTML 切换引用即可——保持旧文件在 CDN。灰度 Nginx 按 IP/Cookie 分流。Docker 容器化部署。

**追问预测**：
- "前端部署的关键步骤" → 构建→上传 CDN→更新 HTML。静态资源文件名 hash 确保缓存更新
- "怎么回滚" → CDN 保留旧版本文件——HTML 切换引用即可。一发一滚
- "灰度发布怎么做" → Nginx 按 IP/Cookie 分流——部分用户新版本部分旧版本
> 🏷️ 概念题
> ⭐⭐⭐ | 难度：中级

**题目**：你的项目是如何部署到线上的？Nginx 做了哪些配置？如何配置前端路由的 History 模式？静态资源如何处理缓存？

**考察点**：
- Nginx 配置：反向代理 API（解决跨域）、gzip 压缩、HTTPS 证书
- History 模式配置：`try_files $uri $uri/ /index.html`
- 静态资源缓存策略：JS/CSS 使用 content-hash + `Cache-Control: max-age=31536000`
- Docker 部署：Dockerfile 多阶段构建（build stage + nginx stage）
- CDN 加速：静态资源上传 CDN，`publicPath` 配置 CDN 域名
- 灰度发布：根据用户 ID/版本号将流量分发到不同版本

### Q11: Excel 导入导出

### Q12: 大数据表格渲染优化

**30秒答**：虚拟列表+列固定(sticky)+排序筛选+行选择。100 万条数据只渲染可视 20-30 行。后端分页+前端虚拟滚动组合。和虚拟列表区别——多了列固定/排序/选择等完整表格功能。

**追问预测**：
- "和虚拟列表有什么区别" → 大数据表格 = 虚拟列表 + 列固定 + 排序筛选 + 行选择——更完整的表格方案
- "怎么处理列固定" → sticky 定位——左固定列 position:sticky left:0
- "100 万条数据怎么渲染" → 虚拟滚动——只渲染可视区 20-30 行。后端分页+前端虚拟滚动组合
> 🏷️ 场景题
> ⭐⭐⭐ | 难度：中高级

**题目**：一个表格有 50 列 x 10000 行，如何保证流畅滚动？除了虚拟列表还有哪些优化手段？

**考察点**：
- 虚拟列表（行级虚拟化）+ 横向虚拟滚动（列级虚拟化）
- 列宽固定：避免自动计算列宽的 Auto Layout 开销
- `table-layout: fixed`：强制固定表格布局算法
- 数据扁平化：减少 computed 的深度遍历
- 懒计算：只计算可视单元格的内容
- Object.freeze() 冻结数据对象，避免 Vue 响应式开销（大量数据不需要响应式）
- 分片渲染（Time Slicing）：用 `requestIdleCallback` 分批次渲染

### Q15: 重复请求处理与请求竞态

**30秒答**：维护 pendingMap——URL+method+params 相同判定重复。新请求前 abort 旧请求——用 AbortController。并发控制不同——重复防同请求多次发，并发控制同时请求数量。

**追问预测**：
- "怎么判断是重复请求" → 请求 URL + method + params 相同——维护 pendingMap
- "重复请求和并发控制的区别" → 重复是防同请求多次；并发是控制同时请求数量
- "AbortController 怎么用于竞态" → 新请求发之前 abort 旧请求——保证只处理最新的结果
> 🏷️ 场景题
> ⭐⭐⭐ | 难度：中级

**题目**：如何解决前端的请求竞态（Race Condition）问题？比如搜索框连续输入，如何保证展示的结果匹配最后一次输入？重复请求如何取消？

**考察点**：
- 竞态问题：后发的请求先返回，展示的是旧数据
- 解决方案 1：请求取消（AbortController / axios CancelToken），发新请求时取消旧请求
- 解决方案 2：闭包判断（自增序号 + 返回时比对）
- 解决方案 3：useRequest（ahooks）的 `cancelOnUnmount`
- 防抖（debounce）减少请求频率，但不解决竞态
- 重复请求去重：请求拦截器中根据 url + params 判断，pending 中的相同请求返回缓存 Promise
