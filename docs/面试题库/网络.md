### Q1: HTTPS | 概念题 握手过程详解

> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：请详细描述 HTTPS 的 TLS 握手过程（ECDHE 密钥交换算法）。为什么 HTTPS 是安全的？中间人攻击如何被防止？

**考察点**：
- TLS 1.2 ECDHE 握手：Client Hello -> Server Hello + 证书 + Server Key Exchange -> Client Key Exchange -> Change Cipher Spec -> 加密通信
- 非对称加密（RSA/ECDHE）交换密钥，对称加密（AES）传输数据
- 数字证书和 CA 证书链验证防中间人
- TLS 1.3 将握手缩短为 1-RTT（合并了多个步骤）
- 前向安全性（ECDHE 的临时私钥每次会话不同）

**30秒答**：HTTPS=HTTP+TLS——非对称加密交换对称密钥，之后对称加密通信。过程：证书验证→密钥交换→对称加密。非对称慢 100-1000 倍所以只用来交换密钥。TLS1.3 握手 2-RTT→1-RTT。
**追问预测**：
- "TLS 1.3 相比 1.2 改进什么" → 握手 2-RTT→1-RTT、去掉不安全算法、默认前向安全性
- "为什么不全用非对称加密" → 非对称比对称慢 100-1000 倍——用非对称交换对称密钥是最好的平衡
- "中间人攻击怎么防御" → HTTPS 证书链验证——攻击者没有 CA 私钥无法伪造合法证书

### Q2: HTTP/2 | 概念题 多路复用

> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：HTTP/2 做了哪些改进？多路复用的原理是什么？解决了 HTTP/1.1 的什么问题？

**考察点**：
- 二进制分帧层（Binary Framing Layer）：将 HTTP 报文拆分为独立的帧和流
- 多路复用：一个 TCP 连接上并发传输多个 Stream（不需要排队）
- 解决 HTTP/1.1 的队头阻塞（Head-of-Line Blocking）
- 头部压缩（HPACK）：静态表 + 动态表 + Huffman 编码
- 服务器推送（Server Push）：主动推送相关资源
- HTTP/2 的队头阻塞：TCP 层面的丢包重传仍会影响所有流（HTTP/3 用 QUIC 解决）

**30秒答**：HTTP2 多路复用——一个 TCP 连接并发多个请求，解决 HTTP1.1 队头阻塞。头部压缩 HPACK、服务器推送、二进制分帧。HTTP2 下不用合并文件——多路复用让小文件并发加载，合并反降缓存效率。
**追问预测**：
- "HTTP2 多路复用解决什么问题" → HTTP1.1 队头阻塞——同连接上一个请求卡住后面都等
- "HTTP2 下还需要合并文件吗" → 不需要——多路复用让多个小文件并发加载；合并反而降低缓存效率
- "HTTP2 服务器推送的问题" → 可能推送浏览器已有缓存的资源——浪费带宽。Chrome 已移除支持

### Q3: TCP | 概念题 三次握手 + 四次挥手

> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：请描述 TCP 的三次握手和四次挥手过程。为什么握手是三次、挥手是四次？TIME_WAIT 状态为什么要等 2MSL？

**考察点**：
- 三次握手：SYN -> SYN+ACK -> ACK，确认双方收发能力
- 为什么不是两次：防止已失效的连接请求报文到达服务端
- 四次挥手：FIN -> ACK -> FIN -> ACK（被动方可能还有数据要发送，ACK 和 FIN 分开）
- TIME_WAIT 等 2MSL：确保最后的 ACK 能重传到对端，让旧连接的所有报文消失
- CLOSE_WAIT 过多的问题定位

**30秒答**：三次握手 SYN→SYN-ACK→ACK——建立双向可靠连接。四次挥手 FIN→ACK→FIN→ACK——TCP 全双工双方都要关闭。三次握手不是两次——防止旧连接请求被误建。
**追问预测**：
- "为什么是三次握手不是两次" → 防止旧连接请求到达——两次握手服务器直接建连但客户端不要了
- "四次挥手为什么多一次" → TCP 全双工——双方都要关闭。被动方收到 FIN 后可能还有数据要发
- "TIME_WAIT 状态是什么意思" → 主动关闭方等待 2MSL——确保最后的 ACK 到达、让旧连接包在网络中消失

### Q4: CORS | 概念题 跨域与预检请求

> ⭐⭐⭐⭐ | 难度：中级

**题目**：什么是 CORS 预检请求（Preflight）？什么情况下会触发预检？如何优化预检请求带来的性能问题？

**考察点**：
- 简单请求（不触发预检）：GET/HEAD/POST + 特定 Content-Type
- 非简单请求触发 OPTIONS 预检：`Content-Type: application/json`、自定义头、DELETE/PUT
- 预检响应头：`Access-Control-Allow-Methods/Headers/Origin` + `Access-Control-Max-Age`
- 优化：设置 `Access-Control-Max-Age` 缓存预检结果
- 避免触发预检：使用 `application/x-www-form-urlencoded` 或 multipart/form-data

**30秒答**：CORS 简单请求直接发——服务器返回 Access-Control-Allow-Origin，浏览器检查通过 JS 读到响应。非简单请求先发 OPTIONS 预检——服务器确认允许后才发正式请求。需带 Cookie 要 credentials:include+不允许 Origin:*。
**追问预测**：
- "简单请求的三个条件" → GET/POST/HEAD + 三种 Content-Type + 无自定义头。一个不满足就预检
- "CORS 请求能带 Cookie 吗" → 需 credentials: include + 后端 Access-Control-Allow-Credentials: true + Origin 不能是 *
- "预检请求返回 404 怎么排查" → 后端没处理 OPTIONS 方法——很多后端只注册了 GET/POST

### Q5: CDN | 概念题 原理与优化

> ⭐⭐⭐⭐ | 难度：中级

**题目**：CDN 的工作原理是什么？如何判断用户访问到了哪个 CDN 节点？CDN 的回源机制是怎样的？

**考察点**：
- CDN 本质：通过智能 DNS 解析将用户请求导向最近的边缘节点
- DNS 调度（CNAME 指向 CDN 域名）实现就近访问
- 缓存策略：边缘节点 -> 区域节点 -> 源站的分级缓存
- 回源：缓存未命中时向源站拉取
- 缓存刷新/预热：主动更新 CDN 缓存
- GSLB（全局负载均衡）选择最优节点

**30秒答**：CDN 分布式边缘缓存——用户请求先到最近的 CDN 节点，命中直接返回，未命中回源。文件 hash 控制更新——新 hash 新 URL 自动回源。域名和主站分开——避免带 Cookie。
**追问预测**：
- "CDN 缓存怎么更新" → 文件名 hash——新文件新 URL；紧急情况 CDN purge 刷新缓存
- "CDN 回源率是什么" → 多少请求 CDN 没命中回到源服务器——目标 95%+ 命中率
- "为什么 CDN 域名要和主站不同" → 避免带上不必要的 Cookie——减小请求头体积

> 答案参考：[../网络/dns-cdn.md](../网络/dns-cdn.md)

### Q6: HTTP 状态码 | 概念题

> ⭐⭐⭐⭐ | 难度：初级-中级

**题目**：请列举常见的 HTTP 状态码及含义。301 和 302 有什么区别？401 和 403 有什么区别？

**考察点**：
- 1xx：信息型（101 WebSocket 协议切换）
- 2xx：成功（200 OK、201 Created、204 No Content、206 范围请求）
- 3xx：重定向（301 永久、302 临时、304 协商缓存、307/308 不改变请求方法）
- 4xx：客户端错误（400 Bad Request、401 未认证、403 无权限、404 Not Found、405 Method Not Allowed）
- 5xx：服务端错误（500 内部错误、502 网关错误、503 服务不可用、504 网关超时）

**30秒答**：1xx 信息 2xx 成功(200/201/204) 3xx 重定向(301永久/302临时/304缓存) 4xx 客户端(400/401未认证/403无权限/404/405) 5xx 服务端(500/502网关/504超时)。301 浏览器缓存直接跳，302 每次问。
**追问预测**：
- "301 和 302 的区别" → 301 永久重定向（浏览器缓存下次直接跳）；302 临时重定向（每次询问服务器）
- "401 和 403 的区别" → 401 未认证需要登录；403 已认证但没有权限访问
- "502 和 504 的区别" → 502 网关收到无效响应（上游挂了）；504 网关等待上游响应超时

### Q7: WebSocket | 对比题 vs SSE（Server-Sent Events）

> ⭐⭐⭐⭐ | 难度：中级

**题目**：WebSocket 和 SSE 有什么区别？各自的适用场景是什么？WebSocket 的连接过程是怎样的？

**考察点**：
- WebSocket：全双工通信，二进制或文本帧，HTTP 升级为 ws:// 协议
- 连接建立：HTTP Upgrade 头（101 状态码），切换到 WebSocket 协议
- SSE：服务端到客户端的单向推送，基于 HTTP，自动重连
- WebSocket 适用于聊天、协同编辑；SSE 适用于消息推送、股票行情
- WebSocket 的心跳/重连机制设计

**30秒答**：WebSocket 通过 HTTP Upgrade 握手后切换到全双工通信——适合聊天/实时数据。SSE 单向服务端推送——简单/自动重连。WS 需心跳维护连接；SSE 天然基于 HTTP。
**追问预测**：
- "WebSocket 和 HTTP 的关系" → WebSocket 通过 HTTP Upgrade 握手——升级后协议切换到 WS，走全双工
- "WebSocket 心跳怎么实现" → 客户端定时发 ping/pong 帧，超时未收到 pong 认为断开——重连
- "SSE 和 WebSocket 怎么选" → SSE 单向服务端推（简单/基于 HTTP/自动重连）；WS 双向通信（复杂/需心跳维护）

### Q8: DNS | 概念题 解析过程

> ⭐⭐⭐⭐ | 难度：中级

**题目**：请详细描述 DNS 的解析过程。从浏览器输入域名到获取 IP 地址，经过了哪些步骤？

**考察点**：
- 缓存查询路径：浏览器 DNS 缓存 -> OS hosts 文件 -> 路由器缓存 -> ISP DNS 缓存
- 递归查询：DNS 解析器向 ISP DNS 服务器的请求
- 迭代查询：ISP DNS 服务器逐级查询（根域名服务器 -> 顶级域名服务器 -> 权威域名服务器）
- DNS 记录类型：A（IPv4）、AAAA（IPv6）、CNAME（别名）、MX（邮件）、NS（域名服务器）
- DNS 预解析：`<link rel="dns-prefetch">` 提前解析

**30秒答**：DNS 递归查询——浏览器缓存→OS hosts→本地 DNS 服务器逐级查根→顶级→权威。有 CDN 时 DNS 根据用户 IP 返回最近节点 IP。dns-prefetch 提前 DNS 解析减少延迟。
**追问预测**：
- "DNS 解析过程" → 浏览器 DNS 缓存→OS hosts→本地 DNS 服务器→递归查询根/顶级/权威
- "DNS 预解析是什么" → dns-prefetch——提前 DNS 查询，减少后续请求的 DNS 延迟
- "CDN 和 DNS 的关系" → DNS 智能解析——根据用户 IP 返回最近的 CDN 节点 IP

### Q9: HTTP 缓存 | 概念题策略（强缓存 + 协商缓存）

> ⭐⭐⭐⭐ | 难度：中级

**题目**：请从网络层面解释 HTTP 缓存策略。`Cache-Control` 的各个指令（no-cache、no-store、must-revalidate）有什么区别？实际项目中如何配置缓存策略？

**考察点**：
- `Cache-Control` 指令详解：`max-age`/`s-maxage`/`public`/`private`/`immutable`
- `no-cache`（需验证）和 `no-store`（完全不缓存）的区别
- `must-revalidate`：过期后必须验证才能使用
- 实际策略：HTML 用协商缓存（`Cache-Control: no-cache`），静态资源用强缓存 + hash（`max-age=31536000`）
- Service Worker 缓存 vs HTTP 缓存的分层

**30秒答**：强缓存先判断——Cache-Control:max-age 有效则直接返回不请求。过期走协商——浏览器带 If-None-Match(ETag)和 If-Modified-Since(Last-Modified)，服务器判断 304 还是 200。ETag 优先级高于 Last-Modified。
**追问预测**：
- "强缓存和协商缓存哪个优先" → 强缓存优先——不发起请求。强缓存过期才进入协商缓存
- "Cache-Control 和 Expires 哪个优先" → Cache-Control（HTTP1.1/相对时间）优先于 Expires（HTTP1.0/绝对时间）
- "immutable 指令的作用" → 告诉浏览器过期前绝对不变——不用发条件请求验证

### Q10: CSRF | 概念题 攻击与防御

> ⭐⭐⭐⭐ | 难度：中级

**题目**：什么是 CSRF 攻击？其原理是什么？有哪些防御手段？

**考察点**：
- CSRF 利用用户已登录的身份在第三方网站发起恶意请求
- 防御：CSRF Token（服务端生成、表单提交验证）
- SameSite Cookie（Strict/Lax/None）
- Referer/Origin 请求头校验
- 双重 Cookie 验证
- 关键操作使用验证码或二次确认

**30秒答**：CSRF 利用已登录身份伪造请求。防御：SameSite Cookie(Lax 默认跨站不带)、CSRF Token(非 Cookie 传递——请求头或 body)、Origin/Referer 校验请求来源。和 XSS 本质区别——CSRF 盗身份，XSS 注入代码。
**追问预测**：
- "SameSite Cookie 怎么防 CSRF" → SameSite=Lax 默认——跨站请求不自动带 Cookie，攻击者无法利用已登录状态
- "CSRF Token 放哪里" → 请求头或请求体——不能放 Cookie（会被自动带上）
- "Referer 校验的局限性" → 用户可能关闭 Referer 发送、HTTPS→HTTP 不传 Referer

> 答案参考：[../浏览器/安全/csrf.md](../浏览器/安全/csrf.md)

### Q11: XSS | 概念题 攻击与防御

> ⭐⭐⭐⭐ | 难度：中级

**题目**：什么是 XSS 攻击？存储型、反射型、DOM 型 XSS 有什么区别？前端如何防御 XSS？

**考察点**：
- 三种 XSS 的区别（存储型最严重：恶意代码存储到数据库）
- 输入过滤 + 输出转义（HTML 实体编码）
- CSP（Content-Security-Policy）HTTP 头
- HttpOnly Cookie 防止 JS 读取敏感 Cookie
- `innerHTML` / `document.write` 的安全替代（`textContent`）
- Vue/React 的自动转义机制

**30秒答**：三种：反射型(URL 参数→页面)/存储型(数据库→所有用户)/DOM 型(纯客户端不经过服务端)。防御：输出多语境转义+HttpOnly Cookie+CSP+DOMPurify 过滤富文本。textContent 天然防 XSS——始终当纯文本。
**追问预测**：
- "XSS 三种类型的区别" → 反射型在请求中、存储型在数据库、DOM 型全在客户端不经过服务端
- "怎么防御 XSS" → 输出转义（多语境）、HttpOnly Cookie、CSP 白名单、输入校验
- "innerHTML 和 textContent 哪个安全" → textContent 始终当纯文本显示——天然防 XSS

### Q12: HTTP/3 | 概念题 与 QUIC 协议

> ⭐⭐⭐ | 难度：中高级

**题目**：HTTP/3 相比 HTTP/2 做了哪些改进？QUIC 协议基于 UDP，如何保证可靠性？

**考察点**：
- HTTP/3 基于 QUIC（UDP），解决 TCP 层面的队头阻塞
- 连接迁移：基于 Connection ID 而非 IP+端口，切换网络无需重连
- 0-RTT 握手：缓存服务端配置，首次连接即可发送数据
- QUIC 在应用层实现可靠传输：类似 TCP 的确认重传，但流之间互相独立
- 内置 TLS 1.3 加密

**30秒答**：HTTP3 基于 QUIC(UDP)——在 UDP 上自实现可靠性和多路复用，解决 TCP 队头阻塞（TCP 丢包阻塞所有流，QUIC 只阻塞丢包的流）。0-RTT 恢复连接但可能被重放攻击。
**追问预测**：
- "HTTP3 基于 QUIC 解决了什么" → TCP 队头阻塞——TCP 丢包阻塞所有流；QUIC 在 UDP 上实现多路复用，丢包只影响一个流
- "QUIC 为什么选 UDP" → TCP 在操作系统内核——升级困难。UDP 在用户态——灵活可迭代
- "0-RTT 的风险" → 重放攻击——攻击者捕获 0-RTT 数据重新发送。只用于幂等请求

> 答案参考：[../网络/http2-http3.md](../网络/http2-http3.md)

### Q13: Token | 概念题 的存储与传输安全

> ⭐⭐⭐ | 难度：中级

**题目**：JWT Token 应该存在哪里？localStorage 和 cookie 各自的安全风险是什么？如何安全地传输 Token？

**考察点**：
- localStorage 存 Token 的风险：XSS 攻击可读取，无法设置 HttpOnly
- cookie 存 Token：设置 `HttpOnly`（防 XSS 读取）+ `Secure`（仅 HTTPS）+ `SameSite=Strict`
- Authorization 头（Bearer Token） vs cookie 自动携带
- CSRF 与 Token 存储方案的关系
- Access Token + Refresh Token 的设计

**30秒答**：Access Token 存 JS 内存——请求头带 Authorization。Refresh Token 存 HttpOnly Cookie——防 XSS 读。过期后响应拦截器捕获 401→调刷新接口→拿到新 Access Token→重试。并发 401 缓存刷新 Promise 只调一次。
**追问预测**：
- "Access Token 和 Refresh Token 分别存哪" → Access Token 存 JS 内存；Refresh Token 存 HttpOnly Cookie
- "Token 过期了前端怎么处理" → 响应拦截器捕获 401→调刷新接口→拿到新 Token→重试失败的请求
- "并发 401 怎么处理" → 缓存刷新中的 Promise——其他 401 请求 await 同一个 Promise，只调一次刷新

### Q14: OSI 七层模型 | 概念题 TCP/IP 四层

> ⭐⭐⭐⭐ | 难度：中级

**题目**：请描述 OSI 七层模型每层的职责和关键协议。TCP/IP 四层模型和 OSI 七层有什么对应关系？TCP 和 UDP 的区别是什么？

**考察点**：
- 七层从下到上：物理层/数据链路层/网络层/传输层/会话层/表示层/应用层
- 每层的关键协议和设备
- TCP/IP 四层与 OSI 七层的对应关系
- TCP（面向连接可靠）vs UDP（无连接不可靠）的本质区别
- 数据封装：应用层数据逐层加头，对端逐层解头

**30秒答**：OSI 七层从下到上——物理层（比特）、数据链路层（帧/MAC）、网络层（包/IP/路由器）、传输层（段/TCP·UDP）、会话层、表示层、应用层（HTTP/DNS）。实际用的是 TCP/IP 四层——应用层对应 OSI 上三层+传输层+网络层+网络接口层。TCP 面向连接可靠慢，UDP 无连接快不可靠。
**追问预测**：
- "TCP 和 UDP 的区别" → TCP 面向连接可靠(三次握手/确认重传/拥塞控制)——适合 HTTP/文件传输；UDP 无连接不可靠但快——适合视频直播/DNS/游戏
- "路由器和交换机分别在哪层" → 路由器在网络层(IP 转发)；交换机在数据链路层(MAC 转发)
- "为什么实际不用 OSI 七层" → 理论先行设计过于复杂——TCP/IP 先发优势——表示层和会话层功能被 TLS 和应用层吸收

### Q15: RESTful | 概念题 API 设计

> ⭐⭐⭐ | 难度：中级

**题目**：RESTful API 的设计原则有哪些？如何设计出规范的后端接口？GraphQL 相比 REST 有什么优势？

**考察点**：
- 资源导向：URL 表示资源，HTTP 动词表示操作（GET/POST/PUT/PATCH/DELETE）
- 状态码规范使用
- 版本管理（URL 版本 `/v1/` 或请求头版本）
- 分页、过滤、排序的 Query 参数设计
- GraphQL：按需获取字段、单个请求获取多资源、强类型 Schema
- 前端封装 axios，统一处理错误/loading/重试

**30秒答**：RESTful 资源用 URL 标识、操作用 HTTP 方法(GET/POST/PUT/DELETE)、无状态、统一接口。GET 幂等可缓存/有长度限制；POST 不幂等不可缓存/放请求体。PUT 全量更新；PATCH 部分更新。
**追问预测**：
- "RESTful 的核心原则" → 资源用 URL 标识、操作用 HTTP 方法表示、无状态、统一接口
- "GET 和 POST 的区别" → GET 幂等可缓存放在 URL 有长度限制；POST 不幂等不可缓存放在请求体
- "PUT 和 PATCH 的区别" → PUT 全量更新替换整个资源；PATCH 部分更新只传改变的字段
