### Q1: HTTPS | 概念题 握手过程详解
> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：请详细描述 HTTPS 的 TLS 握手过程（ECDHE 密钥交换算法）。为什么 HTTPS 是安全的？中间人攻击如何被防止？

**考察点**：
- TLS 1.2 ECDHE 握手：Client Hello -> Server Hello + 证书 + Server Key Exchange -> Client Key Exchange -> Change Cipher Spec -> 加密通信
- 非对称加密（RSA/ECDHE）交换密钥，对称加密（AES）传输数据
- 数字证书和 CA 证书链验证防中间人
- TLS 1.3 将握手缩短为 1-RTT（合并了多个步骤）
- 前向安全性（ECDHE 的临时私钥每次会话不同）

### Q2: HTTP/2 | 概念题 多路复用
> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：HTTP/2 做了哪些改进？多路复用的原理是什么？解决了 HTTP/1.1 的什么问题？

**考察点**：
- 二进制分帧层（Binary Framing Layer）：将 HTTP 报文拆分为独立的帧和流
- 多路复用：一个 TCP 连接上并发传输多个 Stream（不需要排队）
- 解决 HTTP/1.1 的队头阻塞（Head-of-Line Blocking）
- 头部压缩（HPACK）：静态表 + 动态表 + Huffman 编码
- 服务器推送（Server Push）：主动推送相关资源
- HTTP/2 的队头阻塞：TCP 层面的丢包重传仍会影响所有流（HTTP/3 用 QUIC 解决）

### Q3: TCP | 概念题 三次握手 + 四次挥手
> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：请描述 TCP 的三次握手和四次挥手过程。为什么握手是三次、挥手是四次？TIME_WAIT 状态为什么要等 2MSL？

**考察点**：
- 三次握手：SYN -> SYN+ACK -> ACK，确认双方收发能力
- 为什么不是两次：防止已失效的连接请求报文到达服务端
- 四次挥手：FIN -> ACK -> FIN -> ACK（被动方可能还有数据要发送，ACK 和 FIN 分开）
- TIME_WAIT 等 2MSL：确保最后的 ACK 能重传到对端，让旧连接的所有报文消失
- CLOSE_WAIT 过多的问题定位

> 答案参考：[../网络/tcp.md](../网络/tcp.md)

### Q4: CORS | 概念题 跨域与预检请求
> ⭐⭐⭐⭐ | 难度：中级

**题目**：什么是 CORS 预检请求（Preflight）？什么情况下会触发预检？如何优化预检请求带来的性能问题？

**考察点**：
- 简单请求（不触发预检）：GET/HEAD/POST + 特定 Content-Type
- 非简单请求触发 OPTIONS 预检：`Content-Type: application/json`、自定义头、DELETE/PUT
- 预检响应头：`Access-Control-Allow-Methods/Headers/Origin` + `Access-Control-Max-Age`
- 优化：设置 `Access-Control-Max-Age` 缓存预检结果
- 避免触发预检：使用 `application/x-www-form-urlencoded` 或 multipart/form-data

### Q5: CDN | 概念题 原理与优化
> ⭐⭐⭐⭐ | 难度：中级

**题目**：CDN 的工作原理是什么？如何判断用户访问到了哪个 CDN 节点？CDN 的回源机制是怎样的？

**考察点**：
- CDN 本质：通过智能 DNS 解析将用户请求导向最近的边缘节点
- DNS 调度（CNAME 指向 CDN 域名）实现就近访问
- 缓存策略：边缘节点 -> 区域节点 -> 源站的分级缓存
- 回源：缓存未命中时向源站拉取
- 缓存刷新/预热：主动更新 CDN 缓存
- GSLB（全局负载均衡）选择最优节点

> 答案参考：[../网络/dns-cdn.md](../网络/dns-cdn.md)

### Q6: HTTP 状态码 | 概念题
> ⭐⭐⭐⭐ | 难度：初级-中级

**题目**：请列举常见的 HTTP 状态码及含义。301 和 302 有什么区别？401 和 403 有什么区别？

**考察点**：
- 1xx：信息型（101 WebSocket 协议切换）
- 2xx：成功（200 OK、201 Created、204 No Content、206 范围请求）
- 3xx：重定向（301 永久、302 临时、304 协商缓存、307/308 不改变请求方法）
- 4xx：客户端错误（400 Bad Request、401 未认证、403 无权限、404 Not Found、405 Method Not Allowed）
- 5xx：服务端错误（500 内部错误、502 网关错误、503 服务不可用、504 网关超时）

### Q7: WebSocket | 对比题 vs SSE（Server-Sent Events）
> ⭐⭐⭐⭐ | 难度：中级

**题目**：WebSocket 和 SSE 有什么区别？各自的适用场景是什么？WebSocket 的连接过程是怎样的？

**考察点**：
- WebSocket：全双工通信，二进制或文本帧，HTTP 升级为 ws:// 协议
- 连接建立：HTTP Upgrade 头（101 状态码），切换到 WebSocket 协议
- SSE：服务端到客户端的单向推送，基于 HTTP，自动重连
- WebSocket 适用于聊天、协同编辑；SSE 适用于消息推送、股票行情
- WebSocket 的心跳/重连机制设计

### Q8: DNS | 概念题 解析过程
> ⭐⭐⭐⭐ | 难度：中级

**题目**：请详细描述 DNS 的解析过程。从浏览器输入域名到获取 IP 地址，经过了哪些步骤？

**考察点**：
- 缓存查询路径：浏览器 DNS 缓存 -> OS hosts 文件 -> 路由器缓存 -> ISP DNS 缓存
- 递归查询：DNS 解析器向 ISP DNS 服务器的请求
- 迭代查询：ISP DNS 服务器逐级查询（根域名服务器 -> 顶级域名服务器 -> 权威域名服务器）
- DNS 记录类型：A（IPv4）、AAAA（IPv6）、CNAME（别名）、MX（邮件）、NS（域名服务器）
- DNS 预解析：`<link rel="dns-prefetch">` 提前解析

### Q9: HTTP 缓存 | 概念题策略（强缓存 + 协商缓存）
> ⭐⭐⭐⭐ | 难度：中级

**题目**：请从网络层面解释 HTTP 缓存策略。`Cache-Control` 的各个指令（no-cache、no-store、must-revalidate）有什么区别？实际项目中如何配置缓存策略？

**考察点**：
- `Cache-Control` 指令详解：`max-age`/`s-maxage`/`public`/`private`/`immutable`
- `no-cache`（需验证）和 `no-store`（完全不缓存）的区别
- `must-revalidate`：过期后必须验证才能使用
- 实际策略：HTML 用协商缓存（`Cache-Control: no-cache`），静态资源用强缓存 + hash（`max-age=31536000`）
- Service Worker 缓存 vs HTTP 缓存的分层

### Q10: CSRF | 概念题 攻击与防御
> ⭐⭐⭐⭐ | 难度：中级

**题目**：什么是 CSRF 攻击？其原理是什么？有哪些防御手段？

- 关键操作使用验证码或二次确认

### Q12: HTTP/3 | 概念题 与 QUIC 协议
> ⭐⭐⭐ | 难度：中高级

**题目**：HTTP/3 相比 HTTP/2 做了哪些改进？QUIC 协议基于 UDP，如何保证可靠性？

**考察点**：
- HTTP/3 基于 QUIC（UDP），解决 TCP 层面的队头阻塞
- 连接迁移：基于 Connection ID 而非 IP+端口，切换网络无需重连
- 0-RTT 握手：缓存服务端配置，首次连接即可发送数据
- QUIC 在应用层实现可靠传输：类似 TCP 的确认重传，但流之间互相独立
- 内置 TLS 1.3 加密

> 答案参考：[../网络/http2-http3.md](../网络/http2-http3.md)

### Q13: Token | 概念题 的存储与传输安全
> ⭐⭐⭐ | 难度：中级

**题目**：JWT Token 应该存在哪里？localStorage 和 cookie 各自的安全风险是什么？如何安全地传输 Token？

**考察点**：
- localStorage 存 Token 的风险：XSS 攻击可读取，无法设置 HttpOnly
- cookie 存 Token：设置 `HttpOnly`（防 XSS 读取）+ `Secure`（仅 HTTPS）+ `SameSite=Strict`
- Authorization 头（Bearer Token） vs cookie 自动携带
- CSRF 与 Token 存储方案的关系
- Access Token + Refresh Token 的设计
