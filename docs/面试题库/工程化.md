### Q1: Vite vs Webpack 全面对比

**30秒答**：Vite 开发用浏览器原生 ESM——不打包按需编译，冷启动秒级。Webpack 先打包全量依赖再启动，大项目几十秒。Vite 生产用 Rollup 打包。HMR Vite 毫秒级——只替换变更模块。

**追问预测**：
- "Vite 生产构建为什么不用 esbuild" → esbuild 打包功能不完整——代码分割和 CSS 处理有局限。用 Rollup
- "Vite 的预构建做了什么" → CJS→ESM 转换、多小文件合并减少请求数
- "大项目 Vite 也快吗" → 冷启动依然快——不打包。但首次加载可能发几百个请求
> 🏷️ 对比题
> ⭐⭐⭐⭐⭐ | 难度：中级

**题目**：请详细对比 Vite 和 Webpack 的区别。Vite 为什么比 Webpack 快？Vite 在开发环境和生产环境分别用的是什么构建工具？

**考察点**：
- Webpack 先打包再启动，Vite 利用浏览器原生 ESM 按需编译
- Vite 开发环境用 esbuild 预构建依赖（快 10-100 倍），生产环境用 Rollup 打包
- esbuild 用 Go 编写，多线程并行
- Vite 冷启动快：只编译当前页面需要的模块
- HMR 速度：Vite 只需精确更新变更模块，Webpack 需要重新打包变更模块链
- Webpack 生态更成熟（loader/plugin），Vite 生态正在快速追赶

### Q2: ESBuild vs Babel

**30秒答**：ESBuild Go 编写多线程比 Babel(JS 单线程)快 10-100 倍。ESBuild 不支持 AST 级自定义转换——不能完全替代 Babel。最佳实践：ESBuild 做基础转译，Babel 做复杂代码转换。

**追问预测**：
- "ESBuild 比 Babel 快多少" → 10-100 倍——Go 编写多线程 vs JS 单线程
- "ESBuild 能替代 Babel 吗" → 不能完全替代——ESBuild 不支持 AST 级自定义转换、部分 ES 新特性滞后
- "实际项目怎么用" → ESBuild 做基础转译——Babel 做复杂代码转换和自定义 plugin
> 🏷️ 对比题
> ⭐⭐⭐⭐ | 难度：中级

**题目**：ESBuild 和 Babel 有什么区别？为什么 ESBuild 这么快？ESBuild 可以作为 Babel 的替代品吗？

**考察点**：
- Babel 用 JS 编写，单线程；ESBuild 用 Go 编写，多线程并行
- ESBuild 比 Babel 快 10-100 倍
- ESBuild 的功能：转译、压缩、打包，但功能不如 Babel 全面
- Babel 生态更丰富（大量 plugin 和 preset）、支持自定义插件
- ESBuild 的限制：不支持 AST 级别的自定义转换、部分 ES 新特性支持滞后
- 最佳实践：ESBuild 做基础转译 + Rollup 打包，Babel 做复杂的代码转换

### Q3: Tree Shaking 原理与实践

**30秒答**：Tree Shaking 依赖 ESM 静态结构——编译时确定 import/export。标记未使用导出+Terser 删除死代码。sideEffects:false 告知打包工具可安全删除。不生效原因：CJS 混用、Babel 转译破坏了 ESM。

**追问预测**：
- "Tree Shaking 为什么必须 ESM" → ESM 编译时静态结构——import/export 在代码最顶层、不能条件动态
- "sideEffects 字段的作用" → 标记文件有副作用——打包工具不会删除。false 表示可安全删除未使用导出
- "Tree Shaking 为什么不生效" → CJS 混用、Babel 转译写死了导入、副作用判断有误
> 🏷️ 概念题
> ⭐⭐⭐⭐ | 难度：高级

**题目**：Tree Shaking 的实现原理是什么？为什么 Tree Shaking 必须基于 ESM？如何确保项目的 Tree Shaking 生效？

**考察点**：
- Tree Shaking 依赖 ESM 的静态结构（import/export 编译时确定）
- Webpack 方式：标记未使用的 export（usedExports）+ Terser 删除死代码
- sideEffects：`package.json` 中标记，告知打包工具哪些文件有副作用
- `/*#__PURE__*/` 注释标记函数调用无副作用
- 为何不生效：CommonJS 混用、Babel 把 ESM 转成了 CJS、副作用判断失误
- 检查方式：webpack-bundle-analyzer 分析打包结果

> 答案参考：[../工程化/tree-shaking.md](../工程化/tree-shaking.md)

### Q4: HMR（热模块替换）原理

**30秒答**：HMR 替换变更模块保留应用状态——Live Reload 全页面刷新丢状态。Webpack HMR WebSocket 推送变更→请求模块→替换。Vite HMR 原生 ESM 精确替换——不需要重新打包模块链。

**追问预测**：
- "HMR 和 Live Reload 的区别" → HMR 替换模块保留状态；Live Reload 全页面刷新丢失状态
- "Vite HMR 为什么比 Webpack 快" → 原生 ESM——精确替换变更模块，不需要重新打包模块链
- "module.hot.accept 的作用" → 注册模块更新的回调——自定义 HMR 行为
> 🏷️ 概念题
> ⭐⭐⭐⭐ | 难度：高级

**题目**：Webpack 和 Vite 的 HMR 原理分别是什么？HMR 相比 Live Reload 有什么优势？如何实现自定义模块的 HMR？

**考察点**：
- HMR 只替换变更模块，保留应用状态；Live Reload 全页面刷新
- Webpack HMR：WebSocket 推送变更模块 hash -> 客户端请求变更模块 -> 模块热替换
- Vite HMR：原生 ESM + WebSocket，直接精确替换变更模块
- `module.hot.accept()` 注册模块更新的回调
- 实现原理：模块工厂函数、module cache 的替换
- 为什么有时 HMR 会失败（刷新页面）：模块依赖链变更、不可热替换的操作

### Q5: pnpm vs npm/yarn

**30秒答**：pnpm 内容寻址存储——全局共享包节省磁盘。严格依赖访问——未声明包无法 import(解决幽灵依赖)。硬链接+符号链接机制。npm/yarn 扁平化可访问未声明依赖。

**追问预测**：
- "pnpm 的幽灵依赖是什么" → npm/yarn 扁平化后可以引用未声明的包——pnpm 直接禁止访问
- "pnpm 怎么节省磁盘" → 内容寻址存储——所有包全局共享一份，硬链接到项目
- "pnpm 的缺点" → 某些包未正确声明依赖可能在 pnpm 下报错——需补 declaration
> 🏷️ 对比题
> ⭐⭐⭐⭐ | 难度：中级

**题目**：pnpm 相比 npm 和 yarn 有什么优势？pnpm 的"幽灵依赖"问题是如何解决的？硬链接和符号链接是怎么工作的？

**考察点**：
- npm/yarn 的扁平化 node_modules 导致"幽灵依赖"（可引用未声明的包）
- pnpm 使用内容寻址存储（Content-Addressable Store），所有包全局共享一份
- pnpm 的 node_modules：`.pnpm` 目录 + 符号链接，只暴露 `package.json` 中声明的依赖
- 硬链接节省磁盘空间，依赖安装速度更快
- 严格的依赖访问控制：未声明的依赖无法 require/import

> 答案参考：[../工程化/pnpm.md](../工程化/pnpm.md)

### Q6: Webpack Plugin vs Loader

**30秒答**：Loader 文件转换器——单个文件转换。Plugin 构建流程控制——整个生命周期。Loader 从右到左链式执行。Plugin 基于 Tapable 钩子系统——通过 compiler.hooks 注册。

**追问预测**：
- "Loader 和 Plugin 怎么通信" → Loader 通过 this.emitFile 输出文件；Plugin 通过 compilation 对象读取
- "写一个 Plugin 需要什么" → class + apply(compiler) 方法 + 挂载 compiler.hooks
- "Tapable 是什么" → Webpack 插件系统——SyncHook/AsyncSeriesHook 多种钩子类型
> 🏷️ 对比题
> ⭐⭐⭐ | 难度：中级

**题目**：Webpack 中 Loader 和 Plugin 有什么区别？Loader 的执行顺序是怎样的？请写出一个自定义 Plugin 的基本结构。

**考察点**：
- Loader：文件转换（将非 JS 文件转为 JS 模块），作用于单个文件
- Plugin：扩展 Webpack 功能，作用于整个构建流程（通过钩子 hooks）
- Loader 执行顺序：从右到左、从下到上（`use: ['style-loader', 'css-loader']`）
- Plugin 基本结构：class 包含 `apply(compiler)` 方法，通过 `compiler.hooks` 挂载
- Tapable：Webpack 的插件系统，同步（SyncHook）和异步（AsyncSeriesHook）钩子

### Q7: Source Map 类型与选择

**30秒答**：开发 eval-cheap-module-source-map 快。生产 hidden-source-map 不暴露源码。cheap 忽略列减小体积。Source Map 不要部署到公开 CDN——上传 Sentry 用于错误还原。

**追问预测**：
- "开发/生产环境 Source Map 怎么选" → 开发 eval-cheap-module-source-map（快）；生产 hidden-source-map（不暴露源码）
- "Source Map 太大怎么办" → cheap 忽略列信息减小体积——生产用 hidden 不部署到 CDN
- "怎么防止 Source Map 泄露" → hidden-source-map 或上传到 Sentry 不在公开 CDN 上
> 🏷️ 概念题
> ⭐⭐⭐ | 难度：中级

**题目**：Source Map 有哪些类型（eval、cheap、module、nosources、inline）？开发环境和生产环境分别应该选择哪种？各自有什么区别？

**考察点**：
- `eval`：最快的构建速度，不生成 `.map` 文件，用 eval 包裹
- `source-map`：最全的信息（行列映射），最慢的构建速度
- `cheap`：只映射到行，忽略列，略快
- `module`：包含 loader 的 source map
- `inline`：source map 内联到文件（Data URL），文件体积大
- 开发推荐：`eval-cheap-module-source-map`
- 生产推荐：`hidden-source-map`（不暴露源码位置）或 `source-map`（监控需要）

### Q12: Webpack 构建速度优化

**30秒答**：measure 各阶段耗时→缩小 loader 范围(include/exclude)→Webpack5 持久化缓存(filesystem)→thread-loader 多线程。noParse 跳过无依赖库。resolve.extensions 精简减少文件查找。

**追问预测**：
- "怎么测量构建速度" → speed-measure-webpack-plugin——每个 loader/plugin 耗时一目了然
- "Webpack5 持久化缓存的配置" → cache:{type:filesystem}——二次构建快 60-80%
- "thread-loader 适用场景" → CPU 密集型 loader（babel-loader/ts-loader）——I/O 密集型不要用
> 🏷️ 排查题
> ⭐⭐⭐ | 难度：高级

**题目**：Webpack 构建速度慢的问题如何排查和优化？请列出具体的优化手段。

**考察点**：
- speed-measure-webpack-plugin 测量各阶段耗时
- 缩小 Loader 作用范围：`include` / `exclude`
- 缓存：Webpack 5 持久化缓存（`cache: { type: 'filesystem' }`）、babel-loader cacheDirectory
- 多线程：thread-loader / happyPack（Webpack 4）
- noParse 跳过无需解析的库
- resolve 规范：减少 resolve.extensions、使用 resolve.alias
- DllPlugin 预编译不常变化的依赖（Webpack 4，5 中不推荐）

### Q14: 模块化方案（ESM / CJS / AMD / UMD）

**30秒答**：CJS 运行时加载/值拷贝/同步——require。ESM 编译时加载/值引用/异步——import。Node .mjs 或 type:module 启用 ESM。ESM 可 import CJS；CJS 不能 require ESM(需动态 import)。

**追问预测**：
- "CJS 和 ESM 的本质区别" → CJS 运行时加载/值拷贝/同步；ESM 编译时加载/值引用/异步
- "Node.js 怎么用 ESM" → .mjs 扩展名或 package.json type:module
- "import 和 require 能混用吗" → ESM 可以 import CJS；CJS 不能 require ESM（需动态 import）
> 🏷️ 对比题
> ⭐⭐⭐ | 难度：中级

**题目**：请对比 CommonJS 和 ES Module 的区别。`require` 和 `import` 在加载和执行上有什么不同？如何在 Node.js 中使用 ESM？

**考察点**：
- CJS 运行时加载（`require` 可放在条件语句中），ESM 编译时加载（静态结构）
- CJS 输出值的拷贝，ESM 输出值的引用（动态绑定）
- CJS 同步加载，ESM 异步加载（浏览器环境）
- `__dirname`/`__filename` 在 ESM 中不可用，需用 `import.meta.url`
- Node.js 中 `.mjs` 或 `package.json` 中 `"type": "module"`
- `require` 加载 ESM 模块的限制

> 答案参考：[../工程化/Node/commonjs-esm.md](../工程化/Node/commonjs-esm.md)
