# UDP 协议

> ⭐⭐⭐⭐｜难度：中级｜传输层对比必考题

## 一句话总结

**UDP 无连接、不保证可靠、不保证顺序——但极低开销和低延迟。牺牲可靠性换速度——适合实时音视频/DNS/QUIC（HTTP3 底层）。TCP 和 UDP 的选择是传输层面试的核心——根据场景选协议。**

## 核心机制

### UDP 的特点

- **无连接**：发送前不建立连接，直接发——省了 TCP 的三次握手时间
- **不可靠**：不保证送达、不重传、不确认——包丢了就丢了
- **无序**：后发的包可能先到——不保证顺序
- **无拥塞控制**：不会因为网络拥塞而降低发送速率
- **低开销**：头部仅 8 字节（TCP 头部 20-60 字节）

### UDP 报文结构

```
UDP 头部（8 字节）
├── 源端口     2 字节
├── 目的端口   2 字节
├── 长度       2 字节  ← 头部+数据总长度
├── 校验和     2 字节  ← 可选，IPv4 中可关闭
└── 数据
```

对比 TCP 头部最少 20 字节——UDP 不到一半。没有序号、确认号、窗口大小、紧急指针——UDP 真就是"端口+数据"，什么都不管。

### TCP vs UDP 全面对比

| | TCP | UDP |
|---|------|-----|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 确认重传 + 校验和 | 只有校验和（可选） |
| 顺序 | 序号保证有序 | 无序 |
| 流量控制 | 滑动窗口 | 无 |
| 拥塞控制 | 慢启动/拥塞避免/快重传/快恢复 | 无——尽力发 |
| 头部大小 | 20-60 字节 | 8 字节 |
| 适用场景 | HTTP/HTTPS、邮件、文件传输 | DNS、视频/语音、在线游戏、QUIC |

**面试话术**："TCP 像挂号信——签收、按顺序、保证送到。UDP 像明信片——直接投、不管到不到、不保证顺序。挂号信可靠但慢，明信片不可靠但快。"

### 为什么 QUIC（HTTP3）选择 UDP？

TCP 的问题——队头阻塞：HTTP2 的多路复用在 TCP 层仍然串行——一个包的丢失阻塞后续所有流。TCP 实现在操作系统内核——无法快速迭代。

QUIC 在 UDP 之上自己实现了可靠传输（重传、拥塞控制）、多路复用（无队头阻塞）、0-RTT 连接——全部在用户态实现，浏览器更新就能升级协议。UDP 只做"数据报传输"这一件事——QUIC 接管了 TCP 的所有职责。

## 项目实战

### 前端中的 UDP 场景

前端几乎不直接操作 UDP——WebRTC 底层用 UDP 传输音视频（SRTP+DTLS）。WebTransport（QUIC 上层的浏览器 API）未来可能让前端更直接地使用 UDP 特性——低延迟的实时数据推送。

### UDP 在监控/日志中的使用

服务端打点/日志上报——用 UDP 发送统计数据：丢了不影响业务（丢一个埋点数据比卡住主流程好），发完就走不等待确认。前端埋点也类似——`navigator.sendBeacon()` 底层可能是 HTTP POST 但语义上是"尽力发送不等待响应"的思维。

## 易错点

❌ **UDP 完全不可靠** —— UDP 本身不可靠，但上层协议可以实现可靠。QUIC 在 UDP 上实现了不逊于 TCP 的可靠性——不要混淆 UDP 的特性和应用层对它的增强。

❌ **UDP 比 TCP 快所以 HTTP3 用 UDP** —— 不只是快。QUIC 选 UDP 的核心原因是"用户态实现+无队头阻塞+连接迁移"——TCP 在内核态无法快速迭代才是关键。

❌ **UDP 适合所有场景** —— 需要可靠传输的场景（文件传输、API 请求）UDP 不合适——用 TCP 或基于 QUIC 的上层协议。
