# 一面：网络 + 安全模拟面试（45 分钟）

## 面试整体说明

**面试定位**：一面中网络和安全的综合考察。大多数一面会在这两个领域各问 3-5 题——如果候选人 JS 基础不错但网络/安全答得差，面试官会认为"技术视野窄"。

**面试时长**：45 分钟，分为 4 个阶段。

**评分维度**：

| 维度 | 权重 | 说明 |
|------|------|------|
| HTTP 协议理解 | 35% | HTTP/HTTPS 原理、状态码、版本演进 |
| 网络安全认知 | 30% | XSS/CSRF/CSP/同源策略——不仅知道怎么攻击，更知道怎么防御 |
| 浏览器缓存 | 20% | 强缓存/协商缓存的区别和实际应用 |
| 知识串联度 | 15% | 能否把网络→安全→缓存串成一个完整链路来理解 |

**评级标准**：

| 评级 | 总分 | 典型表现 |
|------|------|----------|
| S | 90+ | 不仅回答正确，还能串联——如从 HTTPS 谈到证书链再谈到中间人攻击 |
| A | 80-89 | 各领域独立答对，但跨领域串联不够 |
| B | 65-79 | 核心概念正确，细节有遗漏（如只说出强缓存说不出协商缓存） |
| C | <65 | 多个核心问题无法回答，安全方案基本空白 |

---

## 第一阶段：HTTP 基础（0-10 分钟，3 题）

### 第 1 题：HTTP 状态码

**面试官话术**："先来一个基础的——HTTP 状态码你了解哪些？随便说几个，按类别讲就行。"

**内心 OS**：热身题。看候选人能不能按类（1xx-5xx）组织知识，而不是零散地报数字。这是一个区分"背过"和"理解"的题——背过的人说 200/301/302/404/500，理解的人说"1xx 信息，2xx 成功，3xx 重定向，4xx 客户端错误，5xx 服务端错误"。

**考点**：状态码分类 + 实际场景关联

**预期回答**：
- **初级**：说出 200/301/302/304/400/401/403/404/500/502/503 中的 6 个以上
- **中级**：按 5 类组织 + 说出 301 vs 302（永久 vs 临时重定向）、401 vs 403（未认证 vs 无权限）
- **高级**：补充 304 和缓存的配合、502 和 504 的区别（502 = 网关错误/上游无响应，504 = 网关超时）

**追问阶梯**：
1. "304 状态码什么时候返回？" → 协商缓存命中 → "Etag 和 Last-Modified 哪个优先级更高？"
2. "502 和 504 有什么区别？" → 502 是上游服务器挂了或返回了无效响应，504 是上游服务器响应超时
3. "遇到过 429 吗？" → Too Many Requests——频率限制，Nginx 和 API Gateway 常用

### 第 2 题：HTTPS 握手过程

**面试官话术**："用户访问一个 HTTPS 网站，从输入 URL 到看到页面，安全性是怎么保证的？说说 TLS 握手。"

**内心 OS**：这道题同时考 URL 到页面 + 安全。看候选人能不能把"TLS 握手"讲清楚——特别是证书校验和对称/非对称加密的配合。

**考点**：TLS 握手流程 + 非对称加密 + 证书机制

**预期回答**：
- **初级**：HTTPS = HTTP + SSL/TLS，数据加密传输
- **中级**：TLS 1.2 握手过程——Client Hello → Server Hello + 证书 → Client 验证证书 → 双方协商对称密钥 → 加密通信开始
- **高级**：TLS 1.3 简化握手（1-RTT），证书链验证（根证书→中间证书→网站证书），为什么用非对称加密交换对称密钥（非对称太慢不能用于全量数据）

**追问阶梯**：
1. "证书链是怎么验证的？" → 浏览器内置根证书 → 逐级验证签名 → 到网站证书
2. "为什么不全用非对称加密？" → 非对称加密比对称加密慢 100-1000 倍
3. "中间人攻击怎么防御？" → 证书 + CA + 浏览器证书链验证

### 第 3 题：HTTP1.1 vs HTTP2 vs HTTP3

**面试官话术**："HTTP 经历了几个大版本，说说 1.1、2、3 之间最重要的差异。"

**内心 OS**：考察候选人对协议的关注广度。HTTP2 的多路复用是必答点，HTTP3 基于 QUIC 是进阶点。

**考点**：HTTP 演进逻辑

**预期回答**：
- **初级**：HTTP1.1 持久连接，HTTP2 多路复用
- **中级**：1.1 队头阻塞/连接数限制 → 2 多路复用+头部压缩+服务器推送 → 3 基于 QUIC(UDP)+0-RTT
- **高级**：HTTP3 用 QUIC 解决了 TCP 层面的队头阻塞（TCP 丢包会阻塞所有流，QUIC 只阻塞丢包的流）

**追问阶梯**：
1. "HTTP2 的服务器推送有什么问题？" → 可能推送浏览器已有缓存的资源——浪费带宽 → Chrome 已移除支持
2. "HTTP2 下还需要合并文件吗？" → 不需要——多路复用让多个小文件可以并发加载，合并反而降低缓存效率

## 第二阶段：浏览器安全（10-22 分钟，3 题）

### 第 4 题：XSS 攻击和防御

**面试官话术**："说说 XSS 是什么，你在项目里怎么防的？"

**考点**：XSS 三类 + 四层防御

**预期回答**：
- **初级**：XSS = 跨站脚本攻击，注入恶意脚本到页面
- **中级**：反射型/存储型/DOM 型三种。防御：输出转义 + HttpOnly Cookie + CSP
- **高级**：多语境转义（HTML 实体/JS 字符串/URL/属性值各不相同）。CSP 作为第二道防线——即使某处 XSS 漏了，CSP 白名单阻止恶意脚本执行

**追问阶梯**：
1. "innerHTML 和 textContent 哪个安全？" → textContent 安全——自动转义。innerHTML 直接插 HTML，是 XSS 的常见入口
2. "CSP 怎么配？" → `Content-Security-Policy` HTTP 头。默认阻止所有 inline script，用 nonce 或 hash 放行合法脚本

### 第 5 题：CSRF 攻击和防御

**面试官话术**："XSS 说完了，那 CSRF 呢？和 XSS 有什么本质区别？"

**考点**：CSRF 原理 + 防御 + 与 XSS 的本质区别

**预期回答**：
- **初级**：CSRF = 跨站请求伪造，利用已登录状态发送恶意请求。防御：Token、SameSite Cookie
- **中级/高级**：本质区别——XSS 是在页面上执行恶意脚本，CSRF 是利用浏览器的自动带 Cookie 机制。防御：SameSite=Lax（默认）+ CSRF Token（每次请求验证）+ Origin/Referer 校验。关键：CSRF Token 不能用 Cookie 传递（Cookie 会自动带，token 就失去意义）

**追问阶梯**：
1. "SameSite 有三个值，各什么意思？" → Strict（同站请求才带 Cookie）、Lax（默认，允许顶级导航的 GET 带 Cookie）、None（所有请求都带，必须配合 Secure）
2. "CSRF Token 放哪里？" → 放在请求头（`X-CSRF-Token`）或请求体——不能放 Cookie

### 第 6 题：同源策略和跨域

**面试官话术**："说说同源策略是什么，跨域有哪些方案？"

**考点**：同源策略定义 + 跨域解决方案

**预期回答**：
- **初级**：协议+域名+端口三相同 = 同源。跨域用 CORS
- **中级**：CORS 简单请求 vs 预检请求。预检请求发 OPTIONS，服务器返回 Access-Control-Allow-* 头
- **高级**：跨域方案全景——CORS(最标准)、JSONP(只支持GET/古老)、postMessage(iframe通信)、代理(Nginx反向代理/devServer proxy)、WebSocket(不受同源限制)。选型：CORS 是首选

**追问阶梯**：
1. "预检请求什么时候触发？" → 非简单请求（如 Content-Type: application/json、自定义头）→ 浏览器自动发 OPTIONS
2. "JSONP 为什么能跨域？" → JSONP 利用 `<script>` 标签不受同源限制——本质上是在请求一个 JS 文件

## 第三阶段：浏览器缓存（22-32 分钟，2 题）

### 第 7 题：强缓存 vs 协商缓存

**面试官话术**："浏览器缓存是怎么工作的？强缓存和协商缓存有什么区别？"

**考点**：强缓存 + 协商缓存机制

**预期回答**：
- **初级**：强缓存直接不请求，协商缓存问服务器
- **中级**：强缓存——Cache-Control: max-age / Expires。协商缓存——ETag/If-None-Match、Last-Modified/If-Modified-Since。强缓存命中返回 200(from disk cache)，协商缓存命中返回 304
- **高级**：实际项目的缓存策略——HTML 设 no-cache 每次验证，JS/CSS 文件名 hash 设 max-age=31536000 immutable，API 不缓存或短缓存

**追问阶梯**：
1. "ETag 和 Last-Modified 哪个优先级高？" → ETag——精确到内容级别。Last-Modified 精确到秒，同一秒内多次修改检测不到
2. "Cache-Control: immutable 是干什么的？" → 告诉浏览器"过期前绝对不变，不用发条件请求验证"

### 第 8 题：实际缓存策略设计

**面试官话术**："如果让你设计一个项目的缓存策略，HTML/JS/CSS/图片/API 分别怎么缓存？"

**考点**：缓存策略设计能力

**预期回答**：
- **HTML**：no-cache —— 每次验证，保证发版后用户立刻拿到最新 HTML
- **JS/CSS（带 hash）**：max-age=31536000 immutable —— 文件名变了就是新资源
- **图片/字体**：max-age=86400 + ETag —— 短期强缓存 + 协商兜底
- **API GET**：不缓存或 max-age=10（短缓存）。POST/PUT/DELETE 不缓存

**追问**：
1. "为什么 HTML 不设强缓存？" → 发版后 JS/CSS 文件名变了但旧 HTML 还在缓存中 → 引用不存在的资源 → 白屏

## 第四阶段：综合追问（32-45 分钟，2 题）

### 第 9 题：一次完整的 HTTPS 请求中涉及了多少种缓存和校验？

**面试官话术**："把前面讲的串起来——用户访问一个 HTTPS 网站，从输入 URL 到页面渲染完成，过程中涉及了哪些缓存、哪些安全校验？"

**考点**：知识串联度

**预期回答**（高级）：
1. DNS 缓存（浏览器 DNS → OS → 路由器 → DNS 服务器）
2. TLS 握手 → 证书链校验（浏览器内置根证书→中间 CA→网站证书）
3. TCP 连接建立（三次握手）
4. HTTP 请求 → 浏览器缓存检查：强缓存？→ 协商缓存？
5. 如果 HTML 命中强缓存但服务器已更新 → 可能拿到旧的 JS/CSS 引用（这就是为什么 HTML 不能强缓存）
6. HTML 中加载的 JS/CSS/图片 → 各自检查缓存策略
7. 页面渲染中的安全：CSP 检查每个 script 来源、XSS 防御、Mixed Content 阻止

### 第 10 题：开放方案题——某个 API 响应太慢怎么排查？

**面试官话术**："用户反馈某个页面加载很慢，你怀疑是某个 API 接口慢，怎么排查和解决？"

**考点**：问题排查思路 + 网络优化知识

**预期回答**：
- **排查**：DevTools Network 面板 → 找到耗时长的请求 → 看 Timing 细分（DNS/TCP/TLS/Waiting/Download）→ Waiting 长 = 后端慢 / Download 长 = 数据量大
- **解决**：Waiting 长 → 推进后端优化；Download 长 → 考虑分页、字段裁剪、gzip；连接耗时 → preconnect；可缓存的数据 → 强缓存或 Service Worker
- **高级**：提及 APM 工具（如 Sentry Performance）+ RUM 从用户侧监控真实接口耗时
