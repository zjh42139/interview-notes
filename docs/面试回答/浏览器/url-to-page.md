# 从 URL 到页面展示 面试回答

> 对应题库：[面试题库/浏览器 Q1](../../面试题库/浏览器.md)

## 30 秒版

整个过程分两大段。网络段：DNS 把域名解析成 IP → TCP 三次握手建立连接 → TLS 握手（HTTPS）→ 发送 HTTP 请求 → 服务器返回 HTML。渲染段：浏览器解析 HTML 构建 DOM 树 → 解析 CSS 构建 CSSOM → 两者合成渲染树 → 布局计算位置 → 绘制像素 → 合成层显示到屏幕上。

---

## 2 分钟版

**第一段：网络。**

浏览器先检查缓存——强缓存命中直接拿（状态码 200 from disk cache），没命中往下走。DNS 解析——递归查浏览器缓存→操作系统 hosts→本地 DNS→根域名服务器→逐级往下，最终拿到 IP。然后是 TCP 的三次握手建立连接——HTTPS 的话再加 TLS 握手，1.5 个 RTT。连接建立后发 HTTP 请求——浏览器会做请求排队（同域名 HTTP/1.1 最多 6 个并发连接），请求发出去后等服务器响应。服务器处理完返回 HTML，状态码可能是 200（新鲜数据）、304（协商缓存命中，用本地缓存）、301/302（重定向）等等。

**第二段：渲染。**

浏览器拿到 HTML 后开始边下载边解析。解析过程构建 DOM 树——遇到 script 标签暂停解析（除非有 defer 或 async），遇到 CSS 解析构建 CSSOM。DOM 和 CSSOM 合成渲染树——display:none 的元素不会出现。然后进入布局阶段——计算每个节点的几何位置和尺寸。再是绘制阶段——把元素画成像素（颜色、边框、阴影、文字）。最后合成——把不同图层的像素合并显示到屏幕上。

**第三段：关键优化节点。**

CSS 放 head 顶部——尽早构建 CSSOM，避免 FOUC。JS 放 body 底部或用 defer——避免阻塞 HTML 解析。首屏关键资源用 preload，下一页可能需要的用 prefetch。DOMContentLoaded 触发时 DOM 解析完成但图片可能还在加载，load 触发时所有资源加载完毕。

## 追问预判

| 追问 | 回应要点 |
|------|----------|
| "TCP 为什么是三次握手" | 确认双方收发能力。第一次：客户端发 SYN，服务端知道客户端能发。第二次：服务端回 SYN+ACK，客户端知道服务端能收能发。第三次：客户端发 ACK，服务端知道自己能发。两次不够（服务端不知道自己的包能不能到客户端），四次浪费 |
| "渲染阻塞和解析阻塞有什么区别" | 解析阻塞——JS 下载和执行时 HTML 解析停下来。渲染阻塞——CSS 下载时页面不显示（防 FOUC），但 HTML 继续解析。所以 JS 是解析阻塞+渲染阻塞，CSS 只是渲染阻塞 |
| "什么是 FOUC" | Flash of Unstyled Content。CSS 加载太慢导致页面先展示了无样式的内容再突然变成有样式的。解决方案——CSS 放 head、关键 CSS 内联 |

## 别踩的坑

- "DNS 查询不是浏览器一个人做的"——面试官可能会追问递归查询和迭代查询的区别。简单说：递归——DNS 服务器替你查到底，迭代——DNS 服务器告诉你去问谁，你一个个问
- "把 TLS 和 TCP 混在一起说"——它们是两层。TCP 三次握手建立的是传输层连接，TLS 握手在此基础上建立的是加密通道
- "只说过程没说优化"——面试官问这道题不是考你记忆，是看你能不能识别瓶颈。主动说哪个步骤可以优化（DNS→prefetch、TCP→preconnect、HTML→缓存、渲染→减少回流）
