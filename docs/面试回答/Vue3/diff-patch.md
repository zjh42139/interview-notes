# Diff / Patch 面试回答

> 对应题库：[面试题库/Vue3](../../面试题库/Vue3.md)

## 30 秒版

Vue3 的 Diff 算法用了"双端比较 + 最长递增子序列"。先从新旧两组子节点两端向中间同时比较——头头、尾尾、头尾、尾头四种情况——处理完不匹配的再用 key 映射表查找可复用节点。最后用最长递增子序列算法找出不需要移动的节点，只移动剩下的。整体还是 O(n)，但比 Vue2 优化了很多常系数。

---

## 2 分钟版

**第一：为什么要 Diff。**

修改数据后，Vue 不知道哪些 DOM 需要更新——最暴力的方式是全部重建，但代价太大。Diff 就是在新旧两组虚拟 DOM 之间找出最小差异，然后 patch 到真实 DOM。因为跨层级移动 DOM 的情况极少，Diff 只比较同层级的节点——这个假设让复杂度从 O(n³) 降到 O(n)。

**第二：Vue3 的双端比较。**

四个指针——`oldStartIdx`、`oldEndIdx`、`newStartIdx`、`newEndIdx`。每轮比较四种组合：新头 vs 旧头（相同→同时后移）、新尾 vs 旧尾（相同→同时前移）、新尾 vs 旧头（相同→旧头移动到尾部）、新头 vs 旧尾（相同→旧尾移动到头部）。四种都不匹配时，拿新头的 key 去旧节点数组中找——找到了复用并移动，没找到就新建。这比 Vue2 的双端比较多了一个步骤——Vue2 在四种都不匹配时做了额外的遍历优化，Vue3 直接走 key 映射。

**第三：最长递增子序列——Vue3 的关键优化。**

当新旧子节点都是"大部分相同、少数移动"时，找出旧节点在新数组中的位置序列，对这个序列求最长递增子序列（LIS）。LIS 中的节点不需要移动——它们已经在新位置上了。只需要移动不在 LIS 中的节点。这个优化在"列表排序"场景下特别明显——之前 Vue2 移动了大量不需要动的节点。

**第四：PatchFlag——编译时优化。**

Vue3 的编译器给每个动态绑定的节点打上 PatchFlag——TEXT、CLASS、STYLE、PROPS 等。Diff 时只看 PatchFlag 标记的类型，比如标了 TEXT 就只比文本，不会去比 class 和 children。静态节点没有 PatchFlag，直接跳过。这个"靶向更新"让 Vue3 的 Diff 跳过大量无意义比较。

## 追问预判

| 追问 | 回应要点 |
|------|----------|
| "key 有什么用" | key 是节点复用的唯一标识——没有 key 就只能按位置匹配，有 key 可以精确找到可复用的节点。index 做 key 等于没做——数组重排后 key 全变了 |
| "最长递增子序列为什么能少移动" | 因为 LIS 中的节点已经在目标位置上了——只需要移动不在 LIS 中的节点，移的次数 = 总节点 - LIS 长度 |
| "Vue3 Diff 和 React 有什么区别" | React 从左到右逐个比，找到 key 匹配的复用。Vue3 从两端向中间比——头尾命中率高时更高效。React 可中断（fiber），Vue 不可中断（但通常很快） |

---

## 别踩的坑

- "Diff 比的是真实 DOM"——不是，比的是新旧虚拟 DOM 树。结果才 patch 到真实 DOM
- "v-for 不用 key 没问题"——是能运行，但更新时会用"就地复用"策略——只改内容不移动 DOM，输入框内容会在列表重排时错位
- "O(n) 不代表快"——O(n) 的常系数差一些就比不上 O(n log n)。Vue3 的双端比较 + LIS 主要优化的是常系数
