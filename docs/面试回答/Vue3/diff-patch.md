# Diff / Patch 面试回答

## Q1: Vue3 的 Diff 算法是怎么做的？

### 30 秒版本

"Vue3 的 Diff 是双端比较 + 最长递增子序列。先从头尾各放一个指针往中间走——头对头、尾对尾、头对尾、尾对头，逐个匹配相同节点。匹配不上了就找出中间未匹配部分的最长递增子序列——这些节点不用移动。剩下的该移动移动、该新增新增、该删除删除。总体 O(n)。"

### 2 分钟版本

"Diff 的本质是新旧两个虚拟 DOM 树对比，找出最小更新量。分三步：

**第一步：预处理——从两端往中间收**。头尾各放一个指针——oldHead/newHead、oldTail/newTail。四个对比方向：头对头（same node → patch 后双双后移）、尾对尾（same node → patch 后双双前移）、头对尾（oldHead 移到 oldTail 后）、尾对头（oldTail 移到 oldHead 前）。能匹配上的节点直接 patch 复用，指针继续内收。这一步把"无需移动"的节点全部处理掉。

**第二步：最长递增子序列（LIS）**。预处理完后中间有一段还没匹配的。遍历新节点中间段，记录每个节点在原节点段中的索引位置。对这些索引求 LIS——最长递增子序列表征"这些节点在原顺序中就是递增的，不需要移动"。LIS 之外的节点就要插入、移动或删除。

**第三步：从后往前 Patch**。新节点段从后往前遍历——在 LIS 中的跳过（不需要动），不在 LIS 中的插入到正确位置。最后删除不在新节点段中的原节点。

**为什么 Diff 是 O(n) 不是 O(n³)**：只做同层比较——不会把 A 层的一个节点拿去和 B 层的节点对比。加上两端预处理和 LIS 贪心，最终 O(n)。"

### 追问预判

| 面试官追问 | 你的回答 |
|-----------|---------|
| "key 的作用是什么" | key 是节点的身份标识。没有 key 时 Diff 只能靠索引匹配——节点只是"位置对了"就复用，内容完全不对。有 key 时精确匹配——同 key 同类型才复用 |
| "index 当 key 有什么问题" | 数组头部插入新元素——所有旧元素的 index 都变了，key 全错。Diff 会认为所有节点都是新的——全部重建。用唯一 ID 当 key 才能保证 Diff 正确 |
| "PatchFlag 是什么" | Vue3 编译时优化——模板编译时给动态节点打标记（如"这个节点只有 class 会变"）。Diff 时只检查标记对应的属性，跳过静态部分——靶向更新 |

## 别踩的坑

1. **"Diff 是 O(1)"** —— 面试中绝对不要说。Diff 是 O(n)，n 是节点数。常数优化让它很快但不是 O(1)
2. **只讲 LIS 不讲双端比较** —— 很多人一上来就说 LIS。面试官期待先讲双端预处理——这一步处理了大部分简单情况。LIS 是处理"最坏情况"的
3. **key 用 index** —— 最常见的实际项目错误。面试中能指出"索引当 key"的问题说明你真正做过 Diff 相关优化
