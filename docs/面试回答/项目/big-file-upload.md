# 大文件上传 面试回答

> 大文件上传是后台管理系统中最常见的"复杂场景"题。面试官不指望你写过完整方案，但指望你能说清楚核心三段：分片、秒传、续传。

## Q1: 大文件上传怎么实现的？

### 30 秒版本

"核心方案是用 Blob.slice 把大文件切成多个小块，每个小块独立上传，全部上传完后服务端合并。两个关键优化：一是秒传——上传前先算文件 hash 发给后端，如果后端已有这个文件就跳过上传；二是断点续传——上传中断后下次从断点继续，不用重头开始。"

### 2 分钟版本

"大文件上传的核心流程分四步：

**第一步：前端切片**。用 `File.slice(start, end)` 把文件切成固定大小的块（我们设的是 5MB 每块）。每个切片是一个 Blob，加上索引和总块数包装成 FormData 发送。

**第二步：计算文件 hash（秒传的关键）**。上传前用 SparkMD5（增量式 MD5）计算整个文件的 hash。增量计算的意思是边读文件边算 hash——不需要把文件全部加载到内存。hash 算好后先发一个"检查"请求给后端：'这个 hash 你有了吗？'。后端查一下数据库，三种情况：
- 已经有完整文件 → 返回 { status: 'done', url } —— 秒传，用户秒看到上传成功
- 已有部分切片 → 返回 { status: 'partial', uploaded: [0,1,2,...] } —— 断点续传，只传缺失的切片
- 完全没传过 → 返回 { status: 'new' } —— 传全部切片

**第三步：并发上传切片**。不是一片一片顺序发——太慢。用并发控制（我们设 limit=3），同时发 3 个切片，一个完成立刻补一个。每个切片发给后端时带上三个关键参数：fileHash（哪个文件）、chunkIndex（第几片）、totalChunks（共几片）。

**第四步：通知合并**。所有切片上传完成后，前端发一个 merge 请求给后端：'fileHash=xxx，我的 100 个切片全传完了，你合并吧'。后端把所有切片按索引拼接成完整文件，写入存储，返回最终 URL。

**补充：上传进度**。每个切片的 XMLHttpRequest 或 fetch 都可以监听进度。总进度 = (已上传切片数 × 切片大小 + 当前上传切片的已传字节) / 文件总大小。用这个公式可以算出精确到字节的上传进度。"

### 追问预判

| 面试官追问 | 你的回答 |
|-----------|---------|
| "为什么用 Blob.slice 不用别的" | `File` 继承自 `Blob`，`slice` 是 Blob 的原生方法——不会真正复制文件内容，只是在文件上标记一个视图。性能几乎零消耗。 |
| "切片大小怎么选" | 太小 → 请求数太多，HTTP 开销占比大。太大 → 并发优势弱，单次失败重传成本高。一般 2-10MB 是平衡点。我们选 5MB。 |
| "hash 计算会不会阻塞主线程" | SparkMD5 的增量计算是同步的，大文件（>1GB）会卡主线程。解决方案是用 Web Worker 计算 hash——Worker 里读文件、算 hash，算完 postMessage 回主线程。 |

## Q2: 断点续传和秒传的原理？

### 30 秒版本

"秒传和断点续传都是基于文件 hash 的。秒传是——后端发现 hash 已存在，直接返回已有文件的 URL，跳过上传。断点续传是——上传中断后，下次上传前先问后端'哪些切片你已经有了'，后端返回已上传的切片索引，前端只传缺失的切片。"

### 2 分钟版本

"秒传和断点续传共用同一套 hash 机制，但触发时机不同：

**秒传**：上传前检查。用户选了文件 → 前端先算 hash → 发 check 请求 → 后端查 hash 是否已存在。如果存在且文件完整 → 直接秒传。秒传的关键条件：hash 相同。同一份文件的不同副本 hash 相同——比如多个用户上传同一份产品说明书 PDF。

**断点续传**：中断后恢复。上传过程中网络断了、用户关了页面——已经传上去的那部分切片不能浪费。下次用户重新上传同一文件时，前端重新算 hash → 发 check 请求 → 后端返回"你已上传了第 0、1、2 片，还缺 3-99 片"→ 前端直接从第 3 片开始传。

**实现细节**：
- 后端用 Redis 记录每个 hash 的切片状态——`SET file:hash:abc123:chunks:3 = 'done'`，TTL 设 24 小时（一天后未完成的自动清理）
- merge 完成后把切片记录清理掉——标记这个 hash 为完整文件
- 并发上传时如果某片上传失败——自动重试（我们最多重试 3 次）。如果仍失败——标记该片为 pending，等全部重试完再让用户选择是否放弃

**和普通上传的本质区别**：普通上传是一锤子买卖——断了就从头来。分片上传把一个不可恢复的大任务变成了 N 个可恢复的小任务。这是面试官想听到的核心思想。"

### 追问预判

| 面试官追问 | 你的回答 |
|-----------|---------|
| "hash 冲突怎么办" | MD5 的碰撞概率极低——实践上可以忽略。如果不放心，可以加文件大小作为辅助校验——hash 相同 + size 相同才判定为同一文件。极端场景（金融、医疗）可以用 SHA-256。 |
| "并发上传 3 个切片，为什么不是 6 个" | Chrome 同域名最多 6 个 TCP 连接——留 3 个给页面上其他请求（API、图片、CSS）。全占满用户会感觉页面卡住。 |
| "上传过程中用户关了页面怎么办" | 如果前端有用 Service Worker 作离线上传——SW 可以在后台继续。但大多数场景下关页面 = 中断，只能靠下次打开后的断点续传来恢复。告诉面试官"这是一个 UX 取舍——我们选择让用户感知到上传状态，不强制后台继续"。 |

## 别踩的坑

1. **忘记说 hash 计算** —— 这是大文件上传方案里最有技术含量的部分。不提到"在 Web Worker 中计算 hash"等于没说清楚方案的精髓。

2. **"秒传就是跳过上传"太笼统** —— 说清楚秒传的机制：hash 相同 + 文件完整 = 秒传；hash 相同 + 切片不完整 = 断点续传。两个概念用"hash 检查接口的返回状态"区分开。

3. **不说合并后的清理** —— 切片上传完成、merge 成功后，服务器上残留的切片文件要清理——不然后端磁盘很快撑爆。面试时提一句"清理中间切片"体现了对生产环境的周全考虑。
