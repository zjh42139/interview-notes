## Q1: "介绍下你们项目的登录鉴权方案"

**预计时长：2分钟**

---

好的，我用 STAR 法则来介绍我们项目的登录鉴权方案。

**Situation 背景：**我参与的是一个 Vue3 + TypeScript + Element Plus 的后台管理系统，面向企业内部多个部门使用。不同角色的用户——管理员、普通编辑、只读访客——他们能看到的页面和能做的操作是完全不一样的。所以登录不只是验证身份，还要在这里完成权限初始化。

**Task 任务：**我需要设计一套完整的登录鉴权链路，包含三个核心目标：**安全的凭证管理**、**无感的 Token 续期**、以及**登录后自动完成权限系统的初始化**。

**Action 行动：**我设计了这样一套流程。

首先是**登录环节**——用户提交用户名和密码，前端用 Axios 发送到后端的 `/api/login` 接口。后端验证通过之后，返回两个 Token：一个 **accessToken**，有效期两小时，用于日常请求鉴权；一个 **refreshToken**，有效期七天，用于 accessToken 过期后的无感刷新。同时后端还会返回用户的基本信息和权限标识列表。

然后是**拦截器层面**——我封装了 Axios 的请求拦截器和响应拦截器。请求拦截器负责在每个请求的 header 里自动注入 `Authorization: Bearer <accessToken>`。响应拦截器负责统一处理两种异常：**401 表示 accessToken 过期**，触发刷新逻辑；**403 表示权限不足**，提示用户无权操作。

最关键的是**路由守卫 + 动态路由**这一层。登录成功后，路由守卫的 `beforeEach` 钩子会先检查 Pinia store 里有没有用户的权限列表。如果没有——说明是首次登录或者刷新了页面——就调用 `fetchPermissions` 获取权限标识，然后用 `filterRoutes` 过滤前端定义的全量路由表。过滤结果通过 `router.addRoute()` 逐一注册，再把能看到的菜单路由存到 Pinia 里，侧边栏就根据这个来渲染。**这里有一个细节**——addRoute 之后必须 `next({ ...to, replace: true })` 重新触发路由匹配，否则页面会一片空白。

**Result 成果：**这套方案上线后，登录耗时控制在 1.5 秒以内——包括调接口、拿权限、注册路由。Token 刷新对用户完全无感知，即使在表单页面停留超过两小时再提交，也不会丢数据。权限维度实现了**页面级 + 按钮级 + 接口级**三层控制，满足了安全审计的要求。

---

### 如果面试官追问"你们的Token存哪里"，你可以这样回答：

我们用的是**双 Token 分层存储**策略。accessToken 存在 **Pinia store 的内存里**，通过 pinia-plugin-persistedstate 持久化到 localStorage 做刷新恢复——虽然 localStorage 有 XSS 可读的风险，但 accessToken 只有两小时的有效期，即使泄露攻击窗口也有限。refreshToken 由后端直接通过 `Set-Cookie` 写入 **HttpOnly + Secure + SameSite=Strict 的 Cookie** 里，前端 JS 完全读不到，XSS 攻击者拿不走。这样长期凭证安全、短期凭证方便，各取其长。

补充一点——我们做过对比。如果把 accessToken 也放 HttpOnly Cookie，前端在请求拦截器里就拿不到它，必须靠后端从 Cookie 里读——这要求前后端同域部署，或者额外的代理配置。对我们这种前后端分离的架构来说，accessToken 放内存是最灵活的方案。

## Q2: "Token过期了怎么处理？"

**预计时长：2分钟**

---

Token 过期处理我们分了两种情况：**短期 accessToken 过期**和**长期 refreshToken 过期**，处理策略完全不同。

**先讲 accessToken 过期的场景。**这是每天都会发生的情况——accessToken 有效期两小时，用户在系统里操作一段时间后，某个请求突然返回 401。我们的处理是**无感刷新**，用户完全不知道 Token 过期了。具体实现是这样的：

我在 Axios 响应拦截器里，对所有返回 401 的请求做统一拦截。检测到 401 之后，首先判断是不是刷新接口 `/api/auth/refresh` 自己返回的 401——如果是，说明 refreshToken 也出问题了，不能继续刷，否则会死循环。

然后判断一个关键的标志位——**`isRefreshing`**。这是一个模块级的布尔变量，相当于一把锁。

如果 `isRefreshing` 是 false，说明这是第一个遇到 401 的请求，我来负责刷新。先把 `isRefreshing` 设为 true 加锁，然后调用刷新接口用 refreshToken 换一个新的 accessToken，拿到后更新 Pinia 里的 Token，释放锁，用新 Token 重试刚才失败的请求，最后返回结果给业务层。

**但是关键问题来了——如果在刷新期间，又来了新的请求怎么办？**这就是 `isRefreshing` 为 true 的情况。这些请求不能直接发出去，因为它们带的还是过期的 Token，发出去肯定也是 401。我们的方案是**请求队列**——把这些请求的 retry 逻辑封装成一个 Promise，放到 `pendingRequests` 数组里存着。等刷新完成后，用新 Token 逐个 resolve 这些 Promise，它们会用新 Token 重新发起请求。整个过程对调用方来说，只是一个稍慢一点的 await，完全不感知 Token 已经刷新过了。

关于这个队列，我说一下实现细节——它存的不是请求 config，而是一个个 `(newToken: string) => void` 的回调函数。每个回调内部用 newToken 更新 config 的 Authorization header，然后重新发送 `http(config)`。这样每个请求能保持自己的 config 上下文（比如 url、method、params），重放的时候不会互相干扰。这个设计本质上就是**发布-订阅模式**——401 是事件，刷新完成是广播，队列里的每个请求是订阅者。

**再讲 refreshToken 也过期的场景。**如果 refreshToken 过期了——比如用户七天没登录——刷新接口本身也会返回 401。这时候就是真正的"认证失败"了。我会先清理队列：调用 `rejectPendingRequests`，让所有等待中的请求都 reject，避免它们无限等待。然后清除所有 Token 存储，通过 `window.dispatchEvent(new CustomEvent('auth:unauthorized'))` 派发一个全局事件。在 App.vue 里监听这个事件，收到后跳转到登录页并记录当前路径，登录后能回跳回去。

还有一个小优化：我们会在请求发出前先用 JWT 的 `exp` 字段判断 `accessToken` 是否即将过期——比如还剩不到 5 分钟——**提前刷新**而不是等 401。这样能减少一次失败请求的损耗，对用户体验更友好。

---

### 如果面试官追问"刷新期间又来了新请求怎么办"，你可以这样回答：

我刚才提到的**请求队列**就是为了解决这个问题的。我打个比方：就像银行柜台，第一个人发现自己的凭证过期了，柜台说"你等一下，我去给你换一个新的"——把窗口暂时锁住。后面来的人发现窗口锁了，不重新排队，而是把自己的业务单子放在一个等待区。等新的凭证拿回来，窗口工作人员按照等待区里的单子一个一个办理。每一张单子都能拿到新凭证，业务不受影响，只是多等了一会儿而已。

这里有一个实现细节：刷新成功后的重放是**用 forEach 顺序执行**的，不是 Promise.all。因为队列里的请求各自独立，谁先谁后没关系，不需要互相等待。但如果刷新失败了，队列里所有的请求都要 reject，否则它们会永久 pending 下去，造成内存泄漏。

## Q3: "你们的权限系统怎么设计的？"

**预计时长：2分钟**

---

我们的权限系统采用 **RBAC 模型**——基于角色的访问控制，分为三个层面：**路由权限、按钮权限和数据权限**。

**首先是数据模型。**核心是五张表：用户表、角色表、权限表，以及两张中间关联表——用户-角色表和角色-权限表。一个用户可以有多个角色，一个角色可以有多个权限，经典的**多对多关系**。权限表里有一个关键字段叫 `type`，用来区分权限粒度——`page` 代表页面级、`button` 代表按钮级、`api` 代表接口级。

**然后是前端落地，分为三个层面。**

**第一个层面：路由权限。**这是我们项目里最核心的部分。前端先在 `dynamic-routes.ts` 里定义好全量路由表，每一条路由在 `meta` 里挂一个 `permission` 字段，比如 `dashboard:view`、`user:create`。登录成功后，后端返回当前用户的权限标识列表，前端用 `filterRoutes` 函数递归过滤全量路由——用户有对应权限的才保留，没有的直接删掉。过滤结果通过 `router.addRoute()` 注册到 Vue Router，侧边栏渲染的就是过滤后的菜单。**这里的关键是**：没有权限的页面不是前端"隐藏"了，而是路由表里根本不注册——你就算手动在地址栏里输入路径，`to.matched.length` 为零，直接 404，根本进不去。

**第二个层面：按钮权限。**路由控制的是能不能进页面，按钮控制的是进了页面之后能做什么操作。我们封装了一个 `v-permission` 自定义指令。使用方式是 `<el-button v-permission="{ permission: 'user:delete' }">删除</el-button>`。指令内部的逻辑是，在 `mounted` 钩子里调用 `hasPermission` 函数检查当前用户的权限列表里有没有 `user:delete` 这个标识。如果没有，直接 `removeChild` 把按钮从 DOM 里删掉。指令还支持数组和 `mode: 'and'` / `mode: 'or'` 模式——比如需要同时有编辑和审批两个权限的按钮，就可以写 `{ permission: ['user:edit', 'user:approve'], mode: 'and' }`。

我还特别加了一个**超级管理员**的特殊处理。权限列表里如果包含 `*:*:*` 标识，或者在角色列表里是 `super_admin`，函数直接返回 true，跳过具体权限的判断。这样超级管理员永远不用维护具体的权限列表。

**第三个层面：数据权限。**这个主要是后端的职责——后端根据用户所属部门对数据做过滤。前端只需要在请求参数里带上部门 ID 或用户 ID，后端在 SQL 层做 where 过滤。我们前端能做的是，在权限变了的时候及时同步——比如管理员改了某个角色的权限，我们用 WebSocket 接收推送通知，前端重新拉权限列表、重新生成路由，然后提示用户"权限已更新，请刷新页面"。

---

### 如果面试官追问"权限变更后怎么让用户立即生效"，你可以这样回答：

这个问题确实很关键。权限不是静态的——管理员随时可能修改一个角色的权限范围。我们的方案分了三个层次：

**主动推送**是第一层——通过 WebSocket，后端修改完权限后，主动向前端推送一个 `permission:changed` 事件，携带被修改的用户 ID。前端收到后判断是不是当前登录用户——如果是，重新调用 `fetchPermissions` 拉取最新权限列表，然后 `removeRoute` 删掉所有动态路由，再重新 `filterRoutes + addRoute`。这里要注意 `removeRoute` 要先清掉旧的，否则路由会重复注册。

**定时轮询**是第二层——如果 WebSocket 断了，每隔五分钟用 `authApi.getPermissions()` 拉一次权限列表，和 Pinia 里存的做个深比较，有变化就触发上面一样的更新流程。这个不需要高频，权限变更不是秒级需求。

**关键操作前兜底校验**是第三层——也是最保险的一层。比如删除用户这个操作，即使 v-permission 指令已经显示了按钮，在 `handleDelete` 函数执行之前，还是会再调一次 `hasPermission('user:delete')`。用后端的话说，这叫"前端的乐观渲染 + 后端的保守执行"。前两项是用户体验优化，这一项是安全底线——万一权限被撤销了但页面还没来得及同步，这次校验就是最后的防线。

**那用户正在编辑的表单数据怎么办？**我们会在触发权限更新前，先用 `watch` 或 Pinia 订阅检查是否有未保存的表单数据。如果有，先提示用户保存，再执行权限重置。避免权限更新导致用户正在填写的页面突然消失。

### 在我们的后台管理系统中，权限配置是这样的：

我补充一个实际的项目细节。我们在后端管理界面里提供了一个"角色管理"页面，管理员可以在这里看到所有角色，点进某个角色后有一个**权限树**——树形结构展示了所有可分配的权限，按模块分组。比如"用户管理"节点下面有"查看列表"、"新增用户"、"编辑用户"、"删除用户"四个子权限。管理员勾选完之后保存，后端更新角色-权限关联表，然后通过 WebSocket 推送给所有在线的该角色用户。

权限标识的命名规则是 `模块:操作`，全小写、英文、冒号分隔——比如 `user:create`、`dashboard:view`、`report:export`。这个规则前后端统一约定好，避免出现 `userCreate`、`create-user`、`USER_CREATE` 这种命名混用导致的维护灾难。
