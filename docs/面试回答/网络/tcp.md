# TCP 面试回答

## Q1: TCP 三次握手和四次挥手的过程是怎样的？

### 30 秒版本

"三次握手：客户端 SYN → 服务端 SYN+ACK → 客户端 ACK。三次而不是两次——防止旧连接请求导致服务器错误建连。四次挥手：双方各自 FIN+ACK——TCP 全双工，双方都要单独关闭。主动关闭方最后等 2MSL 确保最后的 ACK 到达。"

### 2 分钟版本

"先说三次握手——本质是确认双方的收发能力都没问题：

第一步客户端发 SYN（同步序列号），意思是'我要连接，我的起始序号是 x'。第二步服务端回 SYN+ACK，意思是'我收到了，我的序号是 y，我确认你发的 x——我知道你能发我能收'。第三步客户端回 ACK，意思是'我确认你的 y——我知道你能发我也能收'。三步走完，双方都确认了对方的发送和接收能力。如果只握手两次——服务端收到 SYN 就认为连接建立——那么网络中的旧 SYN 包延迟到达时，服务端会为一个不存在的请求建立连接，白白浪费资源。

**四次挥手**：因为 TCP 是全双工——双方都能收发数据，所以关闭连接时双方各自发 FIN、各自确认。

第一次：主动方发 FIN，说'我没有数据要发了'。第二次：被动方回 ACK，说'知道了，但我可能还有数据要发完'。此时主动方到被动方的方向已关闭，但被动方还能继续发数据。第三次：被动方发完剩余数据后发 FIN，说'我也没有数据要发了'。第四次：主动方回 ACK，然后进入 TIME_WAIT 等 2MSL——确保最后的 ACK 没丢、确保旧连接的所有报文都在网络中消失。"

### 追问预判

| 面试官追问 | 你的回答 |
|-----------|---------|
| "为什么握手三次挥手四次" | 握手时服务端 SYN+ACK 可以合并（三次）；挥手时被动方的 ACK 和 FIN 不能合并——ACK 是'收到你的 FIN'立刻发的，FIN 是'我发完剩余数据'后才发的，中间可能有数据要传 |
| "TIME_WAIT 为什么等 2MSL" | 两个原因——一是防止最后的 ACK 丢了服务端重发 FIN 客户端还能回 ACK，二是让旧连接的所有报文在网络中超时消失不干扰新连接 |
| "大量 CLOSE_WAIT 怎么排查" | 被动方没有调 close()——服务端代码里没关闭 socket。常见原因：忘了 close、回调没执行到、异步处理中异常了没进 finally |

## 别踩的坑

1. **"SYN 攻击"答成泛概念** —— SYN 泛洪攻击本质是只发 SYN 不回 ACK，服务端为每个半开连接分配资源，耗尽 SYN 队列。防御：SYN Cookie——不用给半开连接分配资源
2. **把 TIME_WAIT 说成 CLOSE_WAIT** —— TIME_WAIT 是主动关闭方等 2MSL；CLOSE_WAIT 是被动关闭方收到 FIN 后等应用程序调 close()。说反了直接扣分
3. **"TCP 可靠"不解释** —— 只说"TCP 可靠"太空洞。具体说：序号+确认+重传机制保证数据不丢，校验和保证数据不错，流量控制保证接收方不被淹没
