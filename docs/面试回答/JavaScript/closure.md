# 闭包 面试回答

> 对应题库：[面试题库/JavaScript](../../面试题库/JavaScript.md)

## 30 秒版

闭包就是一个函数能记住并访问它被创建时的作用域，即使这个作用域已经执行完了。最常见的场景是函数里面返回函数——返回的函数能访问外部函数的变量。闭包让 JS 拥有了私有变量、数据封装和函数工厂的能力。

---

## 2 分钟版

**第一：闭包的本质是作用域链。**

JS 在创建函数时会维护一个 `[[Environment]]` 内部属性，指向函数被定义时的词法环境。当外部函数执行完、执行上下文从调用栈弹出后，如果内部函数被返回并持有引用，外部函数的词法环境就不会被 GC 回收。这个内部函数就是闭包。关键理解：闭包记住的是"变量本身"，不是"变量的快照"——变量改了，闭包访问的值也跟着变。

**第二：经典面试题——for 循环 + setTimeout。**

```javascript
for (var i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 0)  // 5 5 5 5 5
}
```
`var` 声明的 `i` 是函数作用域的，所有 setTimeout 回调共享同一个 `i`。解决：用 `let`（块作用域，每次迭代创建新的绑定）或者 IIFE 包一层。用闭包原理解释——每个回调都闭包引用了同一个变量 `i`，当定时器回调执行时 `i` 已经是 5 了。

**第三：闭包的实战价值。**

数据私有化——模块模式、工厂函数返回闭包来隐藏内部状态。React 的 `useState`、Vue3 的 `ref` 本质上也是闭包——`state` 和 `setState` 闭包引用了 fiber 节点上的状态存储。防抖节流——timer 变量活在返回的闭包函数中，每次调用都能访问同一个 timer。

## 追问预判

| 追问 | 回应要点 |
|------|----------|
| "闭包会导致内存泄漏吗" | 会——如果闭包持有大对象引用且不释放。但现代应用中大多数闭包生命周期可控，不是泄漏。真正泄漏的场景：忘记清理的定时器中闭包引用了 DOM |
| "怎么释放闭包" | 置 null——把引用闭包的那个变量设为 null，没有引用指向它，GC 自然回收 |
| "闭包和 this 有什么关系" | 没有直接关系。闭包基于词法作用域（定义时决定），this 基于调用方式（调用时决定）。箭头函数让两者统一——箭头函数的 this 也是词法的，就是外层作用域的 this |

---

## 别踩的坑

- "闭包 = 函数里面返回函数"——这只是最常见的形式。事件回调、定时器回调、Promise.then 的回调，只要引用了外部变量，都构成闭包
- "闭包一定导致内存泄漏"——不是。合理使用闭包不会泄漏。只有长期持有、忘记释放、且引用大对象时才是问题
- "把闭包和作用域链说成两个东西"——它们是一体的。闭包是作用域链的产物——函数访问变量时沿着作用域链查找，闭包就是让这条链在外部函数执行完之后还能被访问到
