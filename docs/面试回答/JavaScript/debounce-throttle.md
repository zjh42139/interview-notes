# 防抖 / 节流 面试回答

> 对应题库：[面试题库/JavaScript](../../面试题库/JavaScript.md)

## 30 秒版

防抖和节流都是高频事件（滚动、输入、resize）的优化手段。防抖是"等你停下来再执行"——连续触发只执行最后一次，场景是搜索框输入。节流是"固定频率执行"——不管怎么触发，每 N 毫秒最多执行一次，场景是页面滚动。一个管"最后一次"，一个管"执行频率"。

---

## 2 分钟版

**第一：防抖 debounce。**

原理：每次事件触发时重置定时器——如果 N 毫秒内又触发了，取消上次的，重新计时。只有最后一次触发后 N 毫秒内没有新触发，才执行函数。两个经典变种：leading 版——第一次触发立刻执行，之后 N 毫秒内的触发忽略；trailing 版（默认）——等最后一次触发后 N 毫秒再执行。实际项目中搜索框常用 trailing + leading 组合——用户输入第一个字符立刻执行一次搜索（减少等待感），后续连续输入等停止后再搜。

**第二：节流 throttle。**

原理：固定一个时间窗口 N 毫秒，窗口内只执行一次。两种实现——时间戳版：记录上一次执行的时间，当前时间减去上次时间 ≥ N 就执行；定时器版：用定时器标记，执行完后 N 毫秒才开始下一个。区别在第一次和最后一次——时间戳版第一次立刻执行、最后一次不执行；定时器版第一次延迟、最后一次会执行。scroll 事件用节流最合适——用户需要持续的反馈。

**第三：选择指南。**

搜索框输入建议 → 防抖（trailing + leading），等用户停下再搜、第一次立刻给反馈。滚动加载更多 → 节流，保持固定的加载频率。窗口 resize → 防抖，等用户拖完窗口再重新计算布局。按钮点击防重复 → 节流或互斥锁，第一次点击后 N 秒内不能再点。

**第四：实际应用中节流还有一个 requestAnimationFrame 版本。**

如果是做动画（跟视觉渲染相关），应该用 rAF 做节流而不是 setTimeout。rAF 天然跟帧率对齐（~16ms），不需要自己管理时间窗口。

## 追问预判

| 追问 | 回应要点 |
|------|----------|
| "防抖和节流能互相替代吗" | 不能。防抖关注"停止后的状态"，节流关注"过程中的频率"。搜索用防抖、滚动用节流——强行互换体验会很怪 |
| "怎么取消一个已经触发的防抖" | 在防抖函数上挂一个 `cancel` 方法——清除定时器并重置状态。用户点了搜索框的 X 关闭按钮时，取消等待中的搜索请求 |
| "防抖能做成 Promise 版吗" | 可以——返回一个 Promise，等 trailing 执行后 resolve 结果。async/await 场景下更自然 |

---

## 别踩的坑

- "节流和防抖混用"——面试中最怕听到"它们差不多"。本质完全不同——防抖是"等你停下来"，节流是"不管你停不停"
- "忘了清理定时器"——组件销毁时定时器还在跑——典型的闭包内存泄漏。Vue 的 onUnmounted / React 的 useEffect cleanup 中必须清理
- "throttle 用 setTimeout 实现忘了时间戳"——纯 setTimeout 版节流，第一次触发要等 N 毫秒才执行，这在 UI 反馈上是致命的
