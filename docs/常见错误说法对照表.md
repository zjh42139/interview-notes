# 常见错误说法对照表

> 模拟面试训练材料——盖住右半边自测，或者让朋友拿「面试官可能怎么问」那栏随机抽查。每一条陷阱都来自 Phase 3 事实审计 + Phase 4 回答稿对齐审计的真实发现。

---

## 为什么要有这张表

Phase 3 事实审计审了 200+ 篇知识文件，Phase 4 回答稿对齐审计审了 60+ 篇回答稿，累计发现 **~450 处事实错误和表述偏差**。

这些错误有个共同特征：**不是"完全不懂"，而是"听起来差不多对"**。面试中脱口而出时，你说的人没意识到，听的人已经在心里扣分了。

这 50 条是从 ~450 处错误中精选出来的——标准就一个：**面试中被问到的概率最高 + 错误说法最容易脱口而出**。全是实战陷阱，不是学术细节。

---

## 使用方法

**自测模式**：用纸或窗口遮住「❌ 错误说法」和「✅ 正确说法」两栏，只看「面试官可能怎么问」，口头作答后对照检查——你的回答更靠近左边还是右边？

**互测模式**：让朋友按以下节奏逐栏展开——
1. 先抛「面试官可能怎么问」——等你完整作答
2. 再亮出「❌ 错误说法」——你的回答有没有踩中？
3. 最后展开「✅ 正确说法」——差距在哪，是细节遗漏还是方向偏差？

**三轮递进法**：
- 第一轮：遮住右边两栏，只看问题自己答。标记答不上来或卡壳的条目
- 第二轮：遮住右边一栏，看错误说法当场说出它错在哪
- 第三轮：全部暴露，只看问题是否能准确复述正确说法

**复习节奏**：50 条分 5 天，每天 2 个模块（10-12 条）。考前最后一天 30 分钟快速全扫一遍，重点盯标记过的卡壳项。

## JavaScript

> JS 基础题几乎每轮一面必问。最容易翻车的地方不是"不知道"，而是**记混了**——不同场景的 this 指向、不同队列的执行时机、不同 API 的边界条件。以下 6 条是"记混"重灾区。

| 题号 | 话题 | 面试官可能怎么问 | ❌ 错误说法（会扣分） | ✅ 正确说法 |
|:---:|------|------|------|------|
| 1 | 箭头函数 this | "箭头函数的 this 能用 call 改吗？" | 可以，箭头函数也是函数，call/apply/bind 对所有函数都生效 | 箭头函数的 this 在定义时词法绑定，call/apply/bind 对它无效——传参可以传，但 this 不变 |
| 2 | Event Loop | "requestAnimationFrame 是宏任务还是微任务？" | rAF 是宏任务 / rAF 是微任务 | rAF 既不是宏任务也不是微任务，它在渲染帧之前执行，属于独立的回调队列，与宏任务队列/微任务队列是并列关系 |
| 3 | 严格模式 this | "函数里的 this 默认指向 window 对吧？" | 对，普通函数 this 就是 window | 严格模式下普通函数 this 为 `undefined`。ESM 模块和 class 内部默认严格模式，这些场景下函数 this 不会指向 window |
| 4 | 闭包与内存泄漏 | "闭包一定会导致内存泄漏吗？" | 闭包就会内存泄漏，所以要少用 | 闭包只是保留对外部变量的引用，只有不需要的引用持续存在且未被释放时才导致泄漏。合理使用（及时置 null、解绑事件、销毁定时器）不会泄漏 |
| 5 | structuredClone | "structuredClone 能克隆哪些类型？能克隆 Error 吗？" | structuredClone 不能克隆 Error，也不能克隆 Blob | structuredClone 可以克隆 Error 对象（Chrome 98+/Firefox 94+），也能克隆 Blob、File、ImageData、ArrayBuffer 等。不能克隆的是 Function、DOM 节点、Symbol |
| 6 | Promise 链 | "`.then(success, fail)` 和 `.then(success).catch(fail)` 一样吗？" | 一样，只是写法不同 | 不一样。`.then(s, f)` 中 f 只捕获原 Promise 的 rejection，不捕获 s 中抛出的异常。`.then(s).catch(f)` 中 catch 同时捕获原 Promise rejection 和 s 中的异常 |

## Vue3

> Vue3 面试中"看起来懂"和"真的懂"的分界线往往是源码机制。面试官可能会用一句"能讲讲原理吗"来区分评级——你说出来的细节决定了评分档次。下面 6 条集中在机制层面最容易被简化的点。

| 题号 | 话题 | 面试官可能怎么问 | ❌ 错误说法（会扣分） | ✅ 正确说法 |
|:---:|------|------|------|------|
| 1 | computed 缓存 | "computed 每次访问都会重新计算吗？" | 每次访问都执行 getter，只是结果缓存下来返回 | computed 内部通过 `dirty` 标记控制——依赖不变直接返回缓存值（不执行 getter），只有依赖变化才标记 `dirty=true` 并在下次访问时重算 |
| 2 | defineModel | "Vue3 里怎么实现 v-model 双向绑定？" | 只能手动用 `defineProps` + `defineEmits` 发 `update:xxx` | Vue 3.4+ 引入 `defineModel()`，自动创建 modelValue prop 和 `update:modelValue` emit，一行代码替代传统 props + emit 模式。需注意 3.4 以下版本不支持 |
| 3 | nextTick | "nextTick 底层是怎么实现的？用 setTimeout 吗？" | 对，就是 setTimeout(fn, 0)，把回调放到宏任务 | nextTick 优先使用 Promise.then（微任务），降级到 MutationObserver（微任务），最后才降级到 setImmediate/setTimeout（宏任务）。Vue 利用微任务在当前宏任务结束后、浏览器渲染前批量执行 DOM 更新 |
| 4 | Diff 算法 | "Vue3 的 Diff 算法和 Vue2 一样吗？" | 差不多，都是双端对比 | Vue3 使用**最长递增子序列**算法优化 Diff——找出新旧 children 中可复用的最长稳定序列，减少不必要的 DOM 移动。Vue2 是双端对比，算法不同，Vue3 移动到接近 O(n) |
| 5 | ref 解包 | "ref 在什么地方会自动解包 `.value`？" | template、reactive、数组、Map 中都会自动解包 | ref 只在 **template 顶层绑定**和 **reactive 对象属性**中自动解包 `.value`。在数组、Map、Set 等集合类型中不解包——`reactive([ref(1)])` 访问元素时仍需要 `.value` |
| 6 | watchEffect | "watchEffect 和 watch 的主要区别是什么？" | watchEffect 就是不需要写 immediate 的 watch | watchEffect **自动追踪**回调内访问的响应式数据，不需手动指定监听源。watch 需显式声明目标且默认懒执行。最大差异：watchEffect 创建时立即执行以收集依赖，watch 需 `immediate: true` 才有同样效果 |

## CSS

> CSS 面试最常见的误区出在"凭直觉回答"——BFC 怎么触发、margin 何时合并、flex 简写背后是什么。这些不看规范答不对。下面 5 条是你可能已经用了 3 年但从来没翻过文档的点。

| 题号 | 话题 | 面试官可能怎么问 | ❌ 错误说法（会扣分） | ✅ 正确说法 |
|:---:|------|------|------|------|
| 1 | BFC 触发 | "overflow 的哪些值能触发 BFC？" | hidden、auto、scroll、visible 都能触发 | `overflow: visible`（默认值）**不触发** BFC；hidden/auto/scroll 触发但有裁剪副作用；`display: flow-root` 是专门创建 BFC 的现代方式——无裁剪、无浮动清除副作用 |
| 2 | margin 合并 | "相邻元素的 margin 一定会合并吗？" | 只要上下相邻就会合并 | margin 合并只在**垂直方向**、**普通文档流块级元素**之间发生。float、absolute、flex/grid 子元素、inline-block 元素的 margin 都不合并 |
| 3 | flex: 1 | "flex: 1 等价于什么？" | `flex: 1 1 auto`，flex-basis 默认就是 auto | `flex: 1` 是 `flex: 1 1 0%` 的简写——flex-basis 为 0% 而非 auto。这个差异在子元素内容宽度超过等分宽度时表现截然不同：0% 严格等分，auto 先按内容分配再按比例分剩余空间 |
| 4 | Grid auto-fill/auto-fit | "auto-fill 和 auto-fit 有区别吗？" | 没区别，效果一样，只是写法不同 | 有本质区别。`auto-fill` 在空间有剩余时保留空白列轨道（空列可见）；`auto-fit` 在空间有剩余时拉伸已有 item 填满整行。空 item 数量不同时表现完全不同 |
| 5 | 选择器优先级 | "11 个 class 能覆盖 1 个 id 的样式吗？" | 可以，class 10 分一个，11 个就是 110 分 | 优先级是四元组 (a, b, c, d)，不是十进制进位——**1000 个 class 也覆盖不了 1 个 id**。高组别无条件胜出：id > class > element > 通配符 |

## 浏览器

> 浏览器考题的陷阱来自两个方向：(1) 缓存策略的细节容易记反——no-cache vs no-store、优先级关系；(2) 安全机制"做不到什么"容易被忽略——HttpOnly 的盲区、事件委托的盲区。下面 5 条集中在这两类。

| 题号 | 话题 | 面试官可能怎么问 | ❌ 错误说法（会扣分） | ✅ 正确说法 |
|:---:|------|------|------|------|
| 1 | 缓存优先级 | "Cache-Control 和 Expires 哪个优先级高？" | Expires 后写的覆盖先写的 / 两者不冲突 | Cache-Control 优先级高于 Expires。Cache-Control: no-cache 不是"不缓存"，是"每次用之前问服务器（走协商缓存）"——真正不缓存的是 no-store |
| 2 | HttpOnly | "Cookie 设了 HttpOnly 就能防 XSS 吗？" | 对，XSS 脚本读不到 Cookie，攻击就无效了 | HttpOnly 只能阻止 JS **读取** Cookie（`document.cookie`），不能阻止浏览器在 JS 发请求时**自动携带**该 Cookie。XSS 仍可通过自动发送请求（fetch/XMLHttpRequest）利用该 Cookie |
| 3 | 事件委托 | "事件委托绑定在父元素上，所有子元素事件都能处理吗？" | 都能，无论什么事件类型 | focus、blur、load、unload、scroll 等事件不冒泡，事件委托对它们无效。需要捕获阶段处理（addEventListener 第三个参数 true），或用 focusin/focusout（可冒泡替代） |
| 4 | Storage 作用域 | "localStorage 和 sessionStorage 的作用域一样吗？" | 一样，都是同源页面共享 | localStorage 同源页面共享；sessionStorage 只在**同一标签页/窗口**内共享——新开标签页即使同源，sessionStorage 也独立 |
| 5 | DOMContentLoaded | "DOMContentLoaded 和 load 哪个先触发？" | load 先触发，因为 load 表示全部加载完了 | DOMContentLoaded 先触发——它在 HTML 解析完成（DOM 树构建完成）时就触发，不等样式表/图片/子框架。load 须等所有资源（图片、样式、脚本等）加载完才触发 |

## 网络

> 网络协议题极易出现"把某一层解决了当全部解决"的错误——HTTP/2 解决队头阻塞但 TCP 层仍在、TLS 1.3 快了但没快到零 RTT。回答这类题要明确**层级边界**。下面 5 条的核心错误模式都是"少了一个限定条件"。

| 题号 | 话题 | 面试官可能怎么问 | ❌ 错误说法（会扣分） | ✅ 正确说法 |
|:---:|------|------|------|------|
| 1 | HTTP/2 队头阻塞 | "HTTP/2 彻底解决了队头阻塞吗？" | 对，多路复用让所有请求并发，不存在阻塞 | HTTP/2 只在**应用层**（HTTP 层）通过多路复用解决了队头阻塞，但 **TCP 层**的队头阻塞依然存在——一个 TCP 包丢失，整个连接的所有流都被阻塞。HTTP/3 用 QUIC（基于 UDP）才从传输层解决 |
| 2 | CORS 简单请求 | "什么样的跨域请求不发预检？" | 只要方法是 GET/POST 就不发预检 | 简单请求必须同时满足三个条件：(1) 方法为 GET/HEAD/POST；(2) 请求头只含 Accept/Accept-Language/Content-Language/Content-Type；(3) Content-Type 值仅限于 `application/x-www-form-urlencoded`、`multipart/form-data`、`text/plain`。任一不满足即触发 OPTIONS 预检 |
| 3 | TLS RTT | "HTTPS 连接比 HTTP 多几次往返？" | TLS 1.3 完全不用额外 RTT / TLS 1.2 只用 1 个 RTT | 纯 TLS 握手：TLS 1.2 需 2-RTT，TLS 1.3 需 1-RTT。完整 HTTPS（含 TCP 三次握手）：TLS 1.2 共 3-RTT，TLS 1.3 共 2-RTT。TLS 1.3 支持 0-RTT 恢复（前提是曾经连接过，有 PSK） |
| 4 | 重定向状态码 | "301 和 302 的区别是什么？" | 301 是临时重定向，302 是永久重定向 | 301（永久）、308（永久）搜索引擎更新索引；302（临时）、307（临时）不更新。关键差异：301/302 可能将 POST 改为 GET，307/308 严格保持原 HTTP 方法不变 |
| 5 | HTTPS 加密 | "HTTPS 是用非对称加密传输数据的对吧？" | 对，HTTPS 全程用非对称加密保证安全 | HTTPS 用非对称加密（RSA/ECDHE 等）只在**握手阶段**交换对称密钥；数据传输阶段用对称加密（AES 等）——因为对称加密速度远快于非对称。混合加密 = 非对称交换密钥 + 对称传输数据 |

## TypeScript

> TS 的陷阱集中在"看起来对但编译器行为完全不同"的地方——satisfies vs 类型注解、条件类型什么时候分配、void 在回调中的特殊语义。这些不是八股文，是**真正写代码时会踩的坑**。下面 5 条全部可实操验证。

| 题号 | 话题 | 面试官可能怎么问 | ❌ 错误说法（会扣分） | ✅ 正确说法 |
|:---:|------|------|------|------|
| 1 | 分布式条件类型 | "条件类型 `T extends U ? X : Y` 在联合类型上会怎样？" | 任何条件类型遇到联合类型都会自动分配 | 只有**裸类型参数**才会触发分布式条件类型——`T extends U ? X : Y` 中 T 必须是裸露的类型变量。如果 T 被包裹（如 `[T]`、`{ k: T }`），就不会分配，而是把联合类型整体传入判断 |
| 2 | satisfies vs 类型注解 | "satisfies 和类型注解 `: Type` 有什么区别？" | 作用一样，satisfies 只是新语法糖 | 本质区别：satisfies 保留变量**最精确的字面量类型**（不拓宽），同时校验是否满足目标类型。`: Type` 注解会拓宽类型，丢失字面量信息——`const x = {a: '1'} satisfies Record<string, string>` 中 `x.a` 是 `'1'`；加 `: Record<string, string>` 则 `x.a` 是 `string` |
| 3 | void 返回值 | "函数声明返回 void 就不能 return 任何值吗？" | 对，void 和 undefined 一样，return 了就报错 | void 在函数类型中的语义是「**返回值被忽略，你可以返回任何值**」。因此 `(x: number) => void` 可以接受 `() => 1`——回调的返回值不会被使用。这与普通函数声明中的 void 不同 |
| 4 | any 在条件类型中 | "`any extends string ? 'a' : 'b'` 的结果是什么？" | 结果是 `'a'`，因为 any 可以赋值给 string | 结果是 `'a' \| 'b'`。any 在条件类型中会**分裂为两支**——同时走 true 和 false 分支，返回两分支结果的联合类型。这是 any 的特殊行为，unknown 则不会分裂 |
| 5 | as const 开销 | "as const 对一个大对象做深度只读，有运行时性能开销吗？" | 有，相当于递归调用 Object.freeze，对象越大越慢 | **零**运行时开销。as const 是纯类型层面的断言，编译产物中完全消失，只在类型系统中将字面量收窄为最精确的只读类型。不管对象多大，运行时没有任何额外操作 |

## 工程化

> 工程化面试最怕的是混淆工具的角色边界——esbuild 编译 TS 但不检查类型、Vite 预构建不只是跑一遍依赖、Tree-Shaking 不是用了 ESM 就自动生效。每个工具**能做什么**和**不能做什么**同样重要。下面 4 条是角色边界误区的典型。

| 题号 | 话题 | 面试官可能怎么问 | ❌ 错误说法（会扣分） | ✅ 正确说法 |
|:---:|------|------|------|------|
| 1 | Tree-Shaking | "写 ES6 模块就能自动 Tree-Shaking 吗？" | 对，只要用 import/export 就行 | Tree-Shaking 需要两个前提：(1) ESM 静态 import/export 结构（编译时可确定依赖关系）；(2) 模块在 package.json 中声明 `"sideEffects": false`，或构建工具能判定无副作用。CJS 的 `require()` 是运行时动态加载，无法做 Tree-Shaking |
| 2 | Vite 预构建 | "Vite 预构建到底做了什么？" | 就是把 node_modules 里的依赖跑一遍编译 | 预构建有三重作用：(1) 将 CJS/UMD 模块转为 ESM；(2) 将多文件内部模块合并为一个文件减少 HTTP 请求（如 lodash 数百个文件合并为几个）；(3) 缓存到 `node_modules/.vite`，二次启动直接读缓存 |
| 3 | Webpack loader 顺序 | "Webpack 配置里 loader 数组按什么顺序执行？" | 从左到右依次执行 | 从右到左（从下到上）执行。`use: ['a-loader', 'b-loader', 'c-loader']` 的执行顺序是 c → b → a——loader 是管道模型，数据先经过最后的 loader 处理 |
| 4 | esbuild 定位 | "Vite 用 esbuild 编译 TS，是不是不需要 tsc？" | 对，esbuild 能编译 TS，不用单独装 TypeScript | esbuild 只做**转译**（transpile），不做类型检查——它直接剥离类型注解，不验证类型正确性。仍需 `tsc --noEmit` 或 IDE 做类型检查。esbuild 快正因为省掉了类型检查这一步 |

## Git

> Git 面试侧重操作后果——merge 和 rebase 之后 git log 长什么样、reset 三种模式分别丢了什么、reflog 能救多少天前的提交。不是让你背命令参数，是让你在脑子里能**画出提交图**。下面 4 条都是"操作做了，后果说不清"的典型。

| 题号 | 话题 | 面试官可能怎么问 | ❌ 错误说法（会扣分） | ✅ 正确说法 |
|:---:|------|------|------|------|
| 1 | merge vs rebase | "merge 和 rebase 最终结果一样吗？" | 一样，最终代码都合进去了，只是操作方法不同 | 结果不一样。merge 产生一个合并提交（保留分叉历史，非线性图）；rebase 将当前分支的提交"搬移"到目标分支顶部（线性历史，原提交 hash 改变）。git log 看到的提交图完全不同——rebase 看起来像"一条直路到头" |
| 2 | reset 三模式 | "git reset --hard 和 --soft 有什么区别？" | hard 删文件，soft 不删文件 | `--soft`：只移动 HEAD（提交回到暂存区，工作区和暂存区不变）；`--mixed`（默认）：移动 HEAD + 重置暂存区（提交回到工作区）；`--hard`：移动 HEAD + 重置暂存区 + 覆盖工作区（改动全部丢弃，不可恢复） |
| 3 | reflog 窗口 | "reflog 记录永久保存吗？误删的提交随时可以找回？" | 对，reflog 记录永远不会过期 | reflog 有时间限制：可达对象（当前在某分支上的提交）保留 90 天；不可达对象（被 reset/rebase 丢弃的提交）保留 30 天。过期后由 git gc 清理，超过窗口就找不回来了 |
| 4 | detached HEAD | "detached HEAD 是 Git 出 bug 了吗？" | 是异常状态，应该尽快 checkout 回分支 | detached HEAD 是**正常状态**——HEAD 直接指向某个 commit（而非分支指针），常见于 checkout 到历史提交/tag。此时可正常浏览代码和提交，但新提交会"游离"，需创建新分支或切回原分支以防丢失 |

## Pinia

> Pinia 的陷阱几乎都在"看起来两边一样但实际不对称"的 API 上——storeToRefs 的范围、Options Store 和 Setup Store 的差异、persist 插件跨版本迁移。这些都是不看文档直接用会出错的地方。下面 5 条每一行都是一个真实踩坑场景。

| 题号 | 话题 | 面试官可能怎么问 | ❌ 错误说法（会扣分） | ✅ 正确说法 |
|:---:|------|------|------|------|
| 1 | storeToRefs | "用 `storeToRefs()` 解构后，可以直接调用 actions 吗？" | 可以，解构后所有东西都是 ref，actions 也变成了 ref | `storeToRefs()` 只将 **state 和 getters** 转为 ref，不解构 actions。actions 仍须通过 `store.xxx()` 调用，不需要也不应该加 `.value` |
| 2 | $patch | "`store.$patch({ count: 1 })` 是把整个 state 替换为 `{ count: 1 }` 吗？" | 对，就是把 state 整个覆盖了 | `$patch` 对象参数是**深度合并**不是覆盖——只更新指定的属性，未指定的属性保持原值。`$patch` 还接受函数参数（回调中直接对 store 操作），函数形式适合批量/条件修改 |
| 3 | Setup Store $reset | "Pinia 的 Setup Store 也能调用 `$reset()` 吗？" | 能，所有 store 都有 $reset 方法 | 只有 **Options Store** 内置 `$reset()`。Setup Store 没有内置的 $reset——调用会抛错，需要自己实现：通过 `$onAction` 钩子或手动编写重置函数 |
| 4 | persist 插件 v4 | "pinia-plugin-persistedstate 升级到 v4 后 API 没变吧？" | 一样，paths 配置还是和 v3 一样用 | v4 API 有变化：`persist.pick` 替代 v3 的 `paths`（指定要持久化的 key），`persist.omit` 替代 v3 `paths` 的反向操作（排除不要持久化的 key）。参数名和语义都改变了 |
| 5 | $subscribe 参数 | "Options Store 和 Setup Store 的 `$subscribe` 完全一样吗？" | 完全一样，回调的第二个参数都是 state | 不一样。Options Store 的 `$subscribe` 回调第二个参数是当前 state；Setup Store 的同参数是 `undefined`（因为 Setup Store 的 state 没有通过单一对象暴露顶层引用） |

## HR

> HR 面的陷阱和前面所有技术模块完全不同——技术问题答错了面试官会追问答到你会，HR 问题踩雷面试官不会说，只是默默把你从名单上划掉。下面 5 条都是"一句话把 offer 说没了"的高危区。

| 题号 | 话题 | 面试官可能怎么问 | ❌ 错误说法（会扣分） | ✅ 正确说法 |
|:---:|------|------|------|------|
| 1 | 离职原因 | "为什么离开上一家公司？" | 前公司加班太狠 / 领导 PUA / 同事甩锅 / 钱太少 | 不吐槽前司、前同事、前领导。话术公式：正面评价前东家（1 句）+ 客观瓶颈（1-2 句）+ 个人诉求（1 句）+ 与面试岗位的匹配（1 句）。详见 [离职原因](./HR/leave-reason.md) |
| 2 | 最大缺点 | "你最大的缺点是什么？" | 我最大的缺点是太追求完美 / 太爱加班 / 太较真 | "太追求完美"是陈词滥调，面试官一周能听十遍。讲**真实可改进的缺点**（如"文档沉淀意识不够""跨团队沟通还可以加强"），重点落在"我已经在怎么改进"的具体行动上 |
| 3 | 空窗期 | "这段空窗期你在做什么？" | 休息了一段时间 / 在家待着找方向 | 如实说明原因，**强调产出**——学了什么技术、做了什么练习项目、关注了什么方向、参与了什么社区。把空窗期包装成"有目的的自我投资"而非"无所事事的等待" |
| 4 | 期望薪资 | "你的期望薪资是多少？" | 我希望 30K / 你们能给多少？ | 先表达对岗位的兴趣和匹配度，让面试官感受到"因为想加入才谈薪资，不是为薪资才来"。薪资是综合考量的一个维度而非唯一因素，给出合理范围而非单个数字 |
| 5 | 职业规划 | "3-5 年的职业规划是什么？" | 3 年做到技术经理 / 5 年做到架构师 | 规划落在**技术深度和方向**而非 title——"3 年内在前端架构/工程化方向形成自己的方法论，能独立负责复杂项目的技术方案设计"。title 是结果不是目标 |

## 按错误模式分类

这 50 条陷阱可以归纳为 5 种常见错误模式。理解模式比记住单条更有复利效应——面试中遇到新问题，你可以先判断它属于哪种模式，再主动规避。

### 模式一：边界模糊 —— "A 能做到 X"变成"A 一定能做到 X"

知道的没错，但**漏了前提条件**。最典型的是 JS-2（rAF 不属于宏任务也不属于微任务）、TS-1（分布式条件类型需要裸类型参数）、CSS-1（overflow: visible 不触发 BFC）。

**防御策略**：描述一个技术能力时，养成立刻加上"什么情况下不行"的条件反射。只说"能做到什么"是 60 分，"什么情况下不能"才是 80 分的答案。

### 模式二：版本差异 —— "B 是这样的"但没说是哪个版本

不同版本行为不同，但面试回答只说了一个版本。Vue3-2（defineModel 需 3.4+）、TLS-3（TLS 1.2 是 2-RTT 不是 1-RTT）、网络-4（301/307 语义完全不同）。

**防御策略**：回答任何涉及 API 或协议的问题时，养成标注版本的习惯。不确定版本就主动说"在最新版本中……在老版本中……"——这本身就展示了你的技术视野。

### 模式三：层级/范围混淆 —— "解决了 X 层的问题"说成"解决了所有层的问题"

典型是网络-1（HTTP/2 只解决应用层队头阻塞，TCP 层仍在）、Pinia-1（storeToRefs 只解构 state/getters 不解构 actions）、浏览器-3（事件委托不覆盖不冒泡的事件）。

**防御策略**：每次做出"X 解决了 Y"的论断时，主动追问自己：解决的是哪个层级的 Y？在 Y 的范围内还有什么情况不被覆盖？给出范围边界比给出结论更体现思考深度。

### 模式四：默认值假设 —— "等价于"但简写背后的默认值弄错了

一直这么写、一直没问题，但从没看过规范里这个简写到底展开成什么。CSS-3（flex: 1 是 1 1 0% 不是 1 1 auto）、浏览器-1（no-cache 不是不缓存）。

**防御策略**：每次看到 CSS 简写属性或 API 快捷参数，心里默问一句"它的完整展开是什么"。如果说不出来，就去查。不需要查很多，但 P0 模块里的常用简写必须精确掌握。

### 模式五：HR 话术失当 —— "说实话"但说的方式把 offer 说没了

技术陷阱是"说法不准"，HR 陷阱是**"话不对"**——HR-1（吐槽前司）、HR-2（太追求完美）、HR-3（空窗期 = 休息）。面试官不会纠正你的 HR 回答，只会默默做减法。

**防御策略**：HR 问题遵循"正向重构"原则——任何可能被理解为负面的信息，都应搭配一个正向的行动或成长视角。不是不诚实，是用面试官能接受的方式讲事实。

## 练习建议

1. **不要一次看完**。50 条一口气扫完，第 30 条开始注意力就涣散了。每天 2 个模块、10-12 条，5 天一轮，考前至少过 2 轮。
2. **三轮递进**。第一轮遮右边两栏，看问题自己答并录音；第二轮遮右边一栏，看错误说法当场说它错在哪；第三轮只看问题，口头复述正确说法。
3. **按模式巩固**。先把 5 种错误模式记在脑子里，每条陷阱做的时候归类到一种模式。当你能看出"哦这条也是边界模糊"时，说明真正内化了。
4. **关联知识文件**。每条陷阱背后都对应知识库中的一篇知识文件——当你说不清"为什么对"时，点进对应文件补机制细节。不要死记硬背，要理解为什么。
5. **上录音回听**。用手机录下你自己的回答，回听时圈出口误和犹豫的地方。说出口和脑子里想的差距比你想象的大——很多"我以为我会了"听完录音变成"原来我讲不清楚"。
6. **找朋友互测**。这比自测效果好一个数量级。朋友不需要懂技术，只要会念"面试官可能怎么问"那一栏就行——你在压力下组织语言的能力会被真实检验。
7. **考前临阵磨枪**。面试当天早晨或前一天晚上，30 分钟快速全扫一遍。重点盯之前标记过的卡壳条目——这些是你在高压下最可能脱口而出的错误。
